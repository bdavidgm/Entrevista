Concepto:
View es la clase base de todos los elementos de UI en el framework clásico de Android (XML/Views): dibuja en pantalla, recibe eventos táctiles y puede medirse y posicionarse. ViewGroup extiende View y contiene otros hijos, organizándolos con LayoutParams (LinearLayout, ConstraintLayout, FrameLayout, etc.).

Explicación detallada:
Medida en dos fases: onMeasure determina tamaño según MODE (EXACTLY, AT_MOST, UNSPECIFIED); onLayout posiciona hijos en ViewGroup. invalidate() redibuja; requestLayout() remide. Jerarquía profunda penaliza rendimiento; preferir ConstraintLayout plano o Compose.

Eventos touch dispatch de padre a hijo. ViewBinding/DataBinding enlazan XML a código type-safe. En migración, Compose convive con AndroidView o ComposeView en Fragment.

En entrevistas: diferencia View vs ViewGroup; wrap_content vs match_parent; overdraw y hierarquía; custom View cuándo. Comparación con Composable: View es imperativo/mutable; Composable es función declarativa.

Cuándo usarlo:
Pantallas legacy XML, widgets del sistema, integración SDK que solo ofrece Views. Nuevas pantallas suelen ser Compose.

Ejemplo:
class BadgeView @JvmOverloads constructor(
    context: Context,
    attrs: AttributeSet? = null
) : View(context, attrs) {
    override fun onDraw(canvas: Canvas) {
        super.onDraw(canvas)
        // dibujar badge
    }
}

// XML ConstraintLayout contiene TextView hijo — View dentro de ViewGroup

Consejo para entrevista:
Ciclo measure/layout. Por qué anidar muchos LinearLayout anidados es malo. Custom View onMeasure básico.
