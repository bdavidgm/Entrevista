Concepto:
Channel es una primitiva de comunicación hot entre corrutinas: transmite elementos de forma similar a una cola bloqueante, con capacidad configurable y soporte para múltiples productores y consumidores.

Explicación detallada:
Tipos de capacidad:
- Channel.RENDEZVOUS: sin buffer, emisor espera receptor.
- BUFFERED, CONFLATED (solo último valor), UNLIMITED.

Operaciones:
- send(element) / receive()
- trySend, close, cancel.

Channel vs Flow:
- Channel: hot, típicamente 1:1 o N:M con API imperativa; puede perder backpressure si mal usado.
- Flow: cold por defecto, declarativo, operadores (map, filter), mejor para streams de UI y datos.

En arquitectura moderna Android se prefiere SharedFlow/StateFlow para eventos y estado; Channel para pipelines internos o integración con APIs callback.

Patrón actor: una corrutina consume channel y serializa mutaciones de estado.

Cuándo usarlo:
Comunicación puntual entre corrutinas, colas de trabajo internas, puente desde callback a corrutinas (channel.trySend). Para UI state expuesto, StateFlow suele ser más idiomático.

Ejemplo:
val channel = Channel<Command>(Channel.BUFFERED)

// Productor
scope.launch {
    channel.send(Command.Refresh)
}

// Consumidor
scope.launch {
    for (cmd in channel) {
        when (cmd) {
            Command.Refresh -> loadData()
        }
    }
}

Consejo para entrevista:
Contrasta Channel (hot, cola) vs Flow (cold, reactivo). Menciona que SharedFlow reemplaza muchos usos de Channel en MVVM. Habla de cerrar channel y cancelación.
