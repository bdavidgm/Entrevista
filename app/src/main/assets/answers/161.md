## Concepto
Repository con caché expone una sola API que lee primero de fuente rápida/local (Room, memoria) y refresca con red cuando toca. Patrón típico “single source of truth”: la UI observa Flow desde Room; el repo escribe en Room tras fetch.

Mejora offline, latencia y consistencia si defines bien invalidación y conflictos.

Ejemplo:
class UserRepository(private val api: UserApi, private val dao: UserDao) {
    fun users(): Flow<List<User>> = dao.observeAll()

    suspend fun refresh() {
        val remote = api.fetchUsers().map { it.toEntity() }
        dao.replaceAll(remote)
    }
}


ExplicaciónKotlin:
`class UserRepository(private val api: UserApi, private val dao: UserDao)` constructor primario con dos dependencias privadas. `fun users(): Flow<List<User>> = dao.observeAll()` expone flujo frío/caliente según implementación DAO sin cuerpo de función explícito — expresión única. `suspend fun refresh()` puede llamar API y persistir. `api.fetchUsers().map { it.toEntity() }` usa `map` de colección sobre lista devuelta por suspend; lambda `it` transforma DTO a entidad con extensión `toEntity()`. `dao.replaceAll(remote)` persiste lista completa en transacción implícita del DAO. El pipeline `fetch -> map -> replace` es secuencial suspendido en una función. Tipos explícitos en firma documentan contrato del repositorio.

Consejo:
Combina con WorkManager para sync periódico y muestra banner “sin conexión” cuando sirves datos viejos. Evita dos fuentes de verdad sin estrategia.
