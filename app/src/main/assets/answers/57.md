## Concepto
Compose y el sistema de Views pueden convivir en la misma app gracias a dos puentes: ComposeView (incrustar Compose en un layout XML) y AndroidView (incrustar una View dentro de un Composable). Esto permite migrar pantalla por pantalla en lugar de reescribir toda la app.

Para integrar bien hay que sincronizar el ciclo de vida (ViewCompositionStrategy), los temas (Material themes vs MaterialTheme) y el estado.

Ejemplo:
<androidx.compose.ui.platform.ComposeView
    android:id="@+id/composeView"
    android:layout_width="match_parent"
    android:layout_height="wrap_content"/>

binding.composeView.setContent {
    MyAppTheme { HomeScreen() }
}

// En Compose
AndroidView(
    factory = { ctx -> MapView(ctx) },
    update = { it.onResume() }
)

ExplicaciónKotlin:
El bloque XML (<androidx.compose.ui.platform.ComposeView .../>) no es Kotlin; en código Kotlin, binding.composeView.setContent { MyAppTheme { HomeScreen() } } usa una lambda trailing sobre setContent, anidando MyAppTheme { HomeScreen() } como composición raíz. MyAppTheme es presumiblemente un @Composable que aplica tema. AndroidView(factory = { ctx -> MapView(ctx) }, update = { it.onResume() }) integra Views clásicas: factory es lambda (Context) -> MapView que recibe ctx tipado implícitamente y construye MapView(ctx). update recibe it referenciando la View creada y llama it.onResume(). Las lambdas factory y update son parámetros nombrados de AndroidView. Este puente muestra interoperabilidad: Kotlin lambdas conectan APIs imperativas (MapView, binding) con el runtime declarativo de Compose.

Consejo:
"ComposeView dentro de XML; AndroidView dentro de Compose". Migra de hojas a raíz (single Activity al final) y comparte ViewModels para no duplicar estado.
