Concepto:
Estructuras de datos y algoritmos (EDA) son la base para escribir código eficiente, elegir colecciones adecuadas, diseñar pantallas escalables y resolver problemas de entrevista técnica sin cuellos de botella en dispositivos móviles.

Explicación detallada:
En Android no implementas quicksort diariamente, pero sí aplicas complejidad y estructuras constantemente:

1) Colecciones Kotlin/Java: List vs Set vs Map; HashMap O(1) para búsqueda por id frente a List O(n); LinkedHashMap para orden de inserción; ArrayDeque para colas. En RecyclerView/DiffUtil, mapas por id aceleran diff.

2) Listas grandes: paginación (Paging 3) en lugar de cargar 10k items en memoria; LazyColumn solo compone visibles pero tu ViewModel no debe retener listas enormes sin necesidad.

3) Algoritmos de UI: DiffUtil (Myers) para actualizaciones mínimas; orden estable; comparadores eficientes en areItemsTheSame/areContentsTheSame.

4) Concurrencia: entender race conditions, inmutabilidad, estructuras thread-safe; por qué no mutar lista compartida desde múltiples corrutinas sin sincronización.

5) Grafos y árboles: navegación (back stack), deep links, jerarquías de composición; árbol de dependencias Gradle.

6) Entrevistas coding: two pointers, sliding window, BFS/DFS en grids (apps mapas, juegos), heap para “top K”. Complejidad temporal y espacial — en móvil importa memoria (OOM en bitmaps).

7) Caché: LRU (LruCache, Coil disk cache), políticas de eviction; trade-off memoria vs red.

8) Procesamiento en background: dividir trabajo O(n) pesado con coroutines y chunks para no bloquear UI.

Cuándo usarlo:
- Optimizar listas con miles de elementos.
- Diseñar capa de datos (índices Room, claves primarias).
- Pasar entrevistas técnicas (LeetCode medio).
- Depurar ANR por trabajo O(n²) en main thread.

Ejemplo:
// Buscar duplicados O(n) con Set vs O(n²) nested loops
fun hasDuplicate(ids: List<String>): Boolean {
    val seen = HashSet<String>(ids.size)
    for (id in ids) {
        if (!seen.add(id)) return true
    }
    return false
}
