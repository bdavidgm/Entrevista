Concepto:
Una Activity es un componente de Android que representa una pantalla con interfaz de usuario y actúa como punto de entrada para la interacción del usuario. Gestiona un ventana en el sistema y coordina el ciclo de vida, permisos, navegación y, en arquitecturas modernas, delega lógica de negocio al ViewModel.

Explicación detallada:
Cada Activity tiene un ciclo de vida gestionado por el sistema (onCreate, onStart, onResume, onPause, onStop, onDestroy) según visibilidad y memoria. El AndroidManifest declara activities con intent-filters para launcher. Una app típica tiene varias activities o una sola con Navigation (Compose o Fragment).

Activity no debe contener lógica pesada ni estado que sobreviva rotación sin ViewModel/SavedState. Patrón actual: Activity delgada + Compose o binding + ViewModel + inyección (Hilt). onCreate infla UI y suscribe observables; onDestroy limpia listeners si no usan lifecycle-aware components.

Comparación con Fragment: Activity es contenedor de ventana; Fragment es sub-unidad reutilizable dentro de Activity. Con Compose, MainActivity suele ser host único. Comparación con Service: Activity es UI foreground visible al usuario (salvo casos especiales).

En entrevistas preguntan launch modes (standard, singleTop, singleTask, singleInstance), task/back stack, y configuration changes. También edge-to-edge, permisos y Activity Result API.

Cuándo usarlo:
Pantalla completa con ventana propia, entry points, hosts de navegación, flows que requieren Activity context (algunos contratos). Evita múltiples activities para cada diálogo pequeño si Compose/Fragment basta.

Ejemplo:
@AndroidEntryPoint
class MainActivity : ComponentActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        enableEdgeToEdge()
        setContent {
            MyAppTheme { AppNavHost() }
        }
    }
}
