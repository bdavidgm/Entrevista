Concepto:
ProGuard y R8 son herramientas de ofuscación, reducción y optimización de bytecode en builds release de Android. R8 es el shrinker por defecto que elimina código no usado, renombra clases/métodos y optimiza dex, integrado en el pipeline de Gradle.

Explicación detallada:
minifyEnabled true activa R8. shrinkResources elimina recursos huérfanos tras análisis de código. Reglas -keep en proguard-rules.pro protegen reflexión (Retrofit, Room, Parcelable, Hilt, Serializable) de rotura.

Ofuscación dificulta ingeniería inversa pero no es seguridad fuerte. mapping.txt subir a Play para deobfuscate crashes. fullMode y reglas específicas por librería.

Debug no minifica para stack traces legibles. Errores ClassNotFound en release suelen ser reglas -keep faltantes.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto de tu experiencia o proyecto demo, y trade-offs frente a alternativas. Mencionar versiones recientes de API o biblioteca (Material3, Paging 3, Hilt 2.x) demuestra que sigues el ecosistema, sin necesidad de memorizar números de versión exactos.

Cuándo usarlo:
Siempre release producción. Probar release build en QA. Mantener reglas documentadas por dependencia.

Ejemplo:
// proguard-rules.pro
-keep class com.example.data.api.** { *; }
-keepclassmembers class * {
    @com.google.gson.annotations.SerializedName <fields>;
}

android {
    buildTypes {
        release {
            isMinifyEnabled = true
            proguardFiles(
                getDefaultProguardFile("proguard-android-optimize.txt"),
                "proguard-rules.pro"
            )
        }
    }
}

Consejo para entrevista:
R8 reemplaza ProGuard; shrink + obfuscate. Menciona mapping.txt. No confundas con Baseline Profile.
