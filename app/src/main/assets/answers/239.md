## Concepto
Además de los “peligrosos” clásicos, existen permisos especiales concedidos solo desde Ajustes: SYSTEM_ALERT_WINDOW (overlay), WRITE_SETTINGS, PACKAGE_USAGE_STATS, REQUEST_INSTALL_PACKAGES, MANAGE_EXTERNAL_STORAGE, ACCESS_NOTIFICATION_POLICY, SCHEDULE_EXACT_ALARM / USE_EXACT_ALARM, QUERY_ALL_PACKAGES (muy restringido en Play), etc.

Suelen abrir Settings.ACTION_* con startActivityForResult.

Ejemplo:
if (!Settings.canDrawOverlays(this)) {
    startActivity(Intent(Settings.ACTION_MANAGE_OVERLAY_PERMISSION, Uri.parse("package:$packageName")))
}

ExplicaciónKotlin:
`if (!Settings.canDrawOverlays(this))` usa una API Java estática; `this` es `Context`/`Activity` y el negado `!` convierte el booleano Java a condición Kotlin. `Intent(Settings.ACTION_MANAGE_OVERLAY_PERMISSION, Uri.parse("package:$packageName"))` construye un intent explícito hacia Ajustes; `"package:$packageName"` aplica interpolación de cadenas Kotlin donde `packageName` es propiedad de `Context`. `Uri.parse` devuelve `Uri?` en algunas APIs, aquí asumida no nula. `startActivity(...)` lanza la pantalla del sistema; permisos especiales no usan `requestPermissions` runtime clásico. En Kotlin moderno, `registerForActivityResult(StartActivityForResult()) { result -> ... }` detecta si el usuario concedió overlay al volver. `SYSTEM_ALERT_WINDOW` es permiso especial auditado por Play. El bloque `if` sin llaves en una sola línea es válido en Kotlin cuando hay una sola sentencia. Muchos permisos especiales siguen el mismo patrón con distintas `Settings.ACTION_*`.

Consejo:
Play Console pide video y justificación para permisos sensibles. Muchos especiales son señal de malware si se abusan: documenta el caso de uso real del producto.
