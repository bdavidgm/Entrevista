## Concepto
LiveData es un contenedor observable consciente del ciclo de vida: solo notifica a observadores que están en STARTED o RESUMED, evitando crashes por actualizar la UI fuera de tiempo. Fue la API recomendada por Jetpack antes de que [StateFlow](pregunta:86) tomara el relevo.

En proyectos modernos suele preferirse [StateFlow](pregunta:86) + [collectAsStateWithLifecycle](pregunta:134), pero LiveData sigue presente en código existente y en APIs como [WorkManager](pregunta:72).

Ejemplo:
class HomeViewModel : ViewModel() {
    private val _users = MutableLiveData<List<User>>(emptyList())
    val users: LiveData<List<User>> = _users
}

viewModel.users.observe(viewLifecycleOwner) { render(it) }

ExplicaciónKotlin:
class HomeViewModel : [ViewModel](pregunta:68)() { private val _users = MutableLiveData<List<User>>(emptyList()); val users: LiveData<List<User>> = _users } usa MutableLiveData privado y LiveData público de solo lectura, patrón equivalente a [MutableStateFlow](pregunta:86)/asStateFlow. emptyList() es función estándar Kotlin que infiere List<User> por el genérico de MutableLiveData. [viewModel](pregunta:68).users.observe(viewLifecycleOwner) { render(it) } está fuera de la clase: observe es método Java/Kotlin con [LifecycleOwner](pregunta:44) y Observer lambda; it es List<User> emitido. viewLifecycleOwner es propiedad de [Fragment](pregunta:32). LiveData no es Kotlin puro pero la lambda { render(it) } y emptyList() sí; el genérico List<User> anidado aparece en ambas declaraciones LiveData.

Consejo:
"LiveData = lifecycle-aware observable". Para nuevos proyectos prefiere [StateFlow](pregunta:86); LiveData sigue siendo útil cuando ya hay infraestructura sobre él.
