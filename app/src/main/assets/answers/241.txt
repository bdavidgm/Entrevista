Concepto:
La programación orientada a objetos organiza el software alrededor de objetos que combinan estado y comportamiento. Sus cuatro pilares son: encapsulación (proteger el estado), abstracción (mostrar solo lo necesario), herencia (especializar un tipo) y polimorfismo (usar distintas implementaciones mediante un contrato común).

Kotlin soporta POO, pero también favorece composición, funciones de orden superior e inmutabilidad.

Ejemplo:
interface Shape {
    fun area(): Double
}

class Circle(private val radius: Double) : Shape {
    override fun area() = Math.PI * radius * radius
}

fun printArea(shape: Shape) = println(shape.area())

ExplicaciónKotlin:
`interface Shape { fun area(): Double }` declara contrato con función sin cuerpo; las interfaces Kotlin pueden tener implementaciones default, aquí no. `class Circle(private val radius: Double) : Shape` implementa la interfaz con herencia sintáctica `: Shape`; `private val radius` declara propiedad inmutable del constructor primario. `override fun area() = Math.PI * radius * radius` usa expresión body de una línea; `Math.PI` es API Java (alternativa Kotlin: `kotlin.math.PI`). El polimorfismo aparece en `fun printArea(shape: Shape)`: acepta cualquier `Shape` y llama `area()` dinámicamente. `println(shape.area())` es función estándar de Kotlin para salida. No se necesita `open` en interfaces para override de métodos abstractos. Este ejemplo muestra POO mínima en Kotlin favoreciendo interfaces sobre clases abstractas cuando no hay estado compartido. Tipos explícitos (`Double`) documentan contrato numérico para áreas.

Consejo:
No fuerces todo a una jerarquía. Usa objetos cuando estado y comportamiento pertenezcan juntos; para reutilizar comportamiento suele ser preferible composición.
