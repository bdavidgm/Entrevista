## Concepto
Un Fragment es un componente de UI reutilizable que vive dentro de una [Activity](pregunta:31) (o de otro Fragment) y tiene su propio ciclo de vida acoplado al del contenedor. Permite dividir la UI en piezas independientes que se pueden combinar en pantallas multi-panel (tablets, navigation drawer).

Su ciclo de vida es más complejo que el de [Activity](pregunta:31) (View lifecycle adicional) y conviene gestionarlo con viewLifecycleOwner para no fugar referencias a la vista.

Ejemplo:
class ProfileFragment : Fragment(R.layout.fragment_profile) {
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        viewLifecycleOwner.lifecycleScope.launch {
            viewModel.state.collect { render(it) }
        }
    }
}

ExplicaciónKotlin:
`class ProfileFragment : Fragment(R.layout.fragment_profile)` usa el constructor de conveniencia de AndroidX que acepta el ID del layout como argumento del constructor primario, reduciendo boilerplate de `onCreateView`. `override fun onViewCreated(view: View, savedInstanceState: Bundle?)` recibe la vista ya inflada; `view` es no nulo aquí. `viewLifecycleOwner.lifecycleScope.launch { ... }` accede a la extension property `lifecycleScope` ligada al ciclo de vida de la vista (no al del Fragment), evitando fugas tras `onDestroyView`. `launch` inicia una [coroutine](pregunta:78) en el [dispatcher Main](pregunta:87). `viewModel.state.collect { render(it) }` es una [función suspend](pregunta:79) que consume un `Flow`; el lambda `it` recibe cada estado emitido. `render(it)` invoca una función de UI dentro del collector. Este patrón conecta programación reactiva con [corrutinas](pregunta:78) de forma idiomática en Kotlin Android.

Consejo:
Distingue Fragment lifecycle del View lifecycle. Hoy el patrón habitual es single-[activity](pregunta:31) + varios Fragments o pantallas Compose con Navigation.
