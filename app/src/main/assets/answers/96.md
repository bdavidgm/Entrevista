## Concepto
ViewHolder en RecyclerView es el objeto que mantiene referencias a las Views de un ítem de lista (findViewById cacheado). RecyclerView recicla ViewHolders al hacer scroll para evitar inflar XML repetidamente y mejorar rendimiento y memoria.

## Explicación detallada
Adapter onCreateViewHolder infla layout una vez por tipo de vista; onBindViewHolder asocia datos al holder existente. Sin ViewHolder (anti-patrón antiguo ListView) cada scroll llamaba findViewById costoso.

ViewTypes múltiples cuando hay layouts distintos (header, item, ad). DiffUtil calcula diferencias y anima cambios eficientemente con ListAdapter.

En Compose, LazyColumn con key estable cumple rol similar de reciclaje interno; conceptualmente sigue siendo pregunta clásica de entrevista Views.

Pool de scrap views limitado; demasiados view types reduce reciclaje.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

## Cuándo usarlo
RecyclerView en código legacy. En Compose explica equivalencia LazyColumn + keys.

Ejemplo:
class ProductVH(private val binding: ItemProductBinding) :
    RecyclerView.ViewHolder(binding.root) {
    fun bind(item: Product) {
        binding.title.text = item.name
    }
}

class ProductAdapter : ListAdapter<Product, ProductVH>(DIFF) {
    override fun onCreateViewHolder(parent: ViewGroup, viewType: Int) =
        ProductVH(ItemProductBinding.inflate(LayoutInflater.from(parent.context), parent, false))
    override fun onBindViewHolder(holder: ProductVH, position: Int) =
        holder.bind(getItem(position))
}


ExplicaciónKotlin:
`class ProductVH(private val binding: ItemProductBinding) : RecyclerView.ViewHolder(binding.root)` define ViewHolder con constructor primario: `private val binding` queda como propiedad inmutable del holder. `binding.root` pasa la raíz del ViewBinding al superconstructor. `fun bind(item: Product)` recibe `Product`, presumiblemente `data class`; `binding.title.text = item.name` accede a propiedad generada del binding y asigna `text` del TextView. `ProductAdapter : ListAdapter<Product, ProductVH>(DIFF)` hereda adapter con DiffUtil; `DIFF` es `DiffUtil.ItemCallback` companion. `onCreateViewHolder` usa expresión única con `=`; `ItemProductBinding.inflate(LayoutInflater.from(parent.context), parent, false)` es API ViewBinding estática. `onBindViewHolder` llama `holder.bind(getItem(position))`; `getItem` proviene de `ListAdapter`. `override` marca implementación de métodos abstractos.
