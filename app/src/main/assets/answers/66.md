## Concepto
Un Use Case (o Interactor) encapsula una operación de negocio única en una clase con un solo método público, normalmente invoke. Vive en la capa de dominio y orquesta uno o varios repositorios sin conocer detalles de UI ni de framework.

Ventajas: nombres expresivos del dominio, fácil de testear, reutilización entre ViewModels y separación clara entre "qué hace el sistema" y "cómo se presenta".

Ejemplo:
class GetActiveUsersUseCase(
    private val users: UserRepository,
    private val sessions: SessionRepository
) {
    suspend operator fun invoke(): List<User> = users.all()
        .filter { sessions.isActive(it.id) }
}

// En el ViewModel
viewModelScope.launch { _state.value = getActiveUsers() }

ExplicaciónKotlin:
class GetActiveUsersUseCase(private val users: UserRepository, private val sessions: SessionRepository) declara dos dependencias inyectables por constructor. suspend operator fun invoke(): List<User> = users.all().filter { sessions.isActive(it.id) } combina operator invoke, expresión body y encadenamiento: users.all() devuelve presumiblemente List<User>, filter recibe lambda { sessions.isActive(it.id) } donde it es cada User e it.id accede a su identificador. El comentario // En el ViewModel muestra viewModelScope.launch { _state.value = getActiveUsers() }: launch inicia coroutine, getActiveUsers() invoca el use case como función por operator invoke sin nombre explícito, y _state.value asigna resultado (MutableStateFlow o similar). El genérico List<User> y las interfaces Repository tipan el dominio.

Consejo:
Crea Use Cases cuando hay lógica de negocio repetida o compleja; para llamadas triviales, llamar al Repository desde el ViewModel también es válido.
