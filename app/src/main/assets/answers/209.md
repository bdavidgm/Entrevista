## Concepto
[Retrofit](pregunta:207) 2.6+ permite declarar endpoints como [suspend](pregunta:79) fun; [Retrofit](pregunta:207) ejecuta la llamada en el dispatcher adecuado y devuelve el resultado o lanza HttpException. Alternativamente, Call.enqueue sigue existiendo para callbacks.

Evita bloquear el hilo principal: [suspend](pregunta:79) ya gestiona la suspensión correctamente.

Ejemplo:
interface GitHubApi {
    @GET("users/{login}")
    suspend fun user(@Path("login") login: String): UserDto
}

viewModelScope.launch {
    try { val u = api.user("octocat") } catch (e: HttpException) { /* 404, 500 */ }
}

ExplicaciónKotlin:
`interface GitHubApi` declara endpoint con `@GET("users/{login}")` y `suspend fun user(@Path("login") login: String): UserDto`; la [función suspend](pregunta:79)ida permite llamar desde [corrutinas](pregunta:78) sin callbacks. `@Path("login")` vincula el parámetro Kotlin `login` al placeholder de la URL. `viewModelScope.launch { ... }` abre una [corrutina](pregunta:78) en el scope del [ViewModel](pregunta:68), cancelada automáticamente al destruir el [ViewModel](pregunta:68). Dentro, `try { val u = api.user("octocat") } catch (e: HttpException) { ... }` usa try/catch estándar de Kotlin: `val u` tiene alcance al bloque try; `HttpException` es tipo concreto de [Retrofit](pregunta:207) para códigos 4xx/5xx. El literal `"octocat"` es argumento `String`. No se necesita `withContext(IO)` porque [Retrofit](pregunta:207) mueve la llamada [suspend](pregunta:79) al dispatcher correcto. El bloque catch comentado indica manejo de errores HTTP sin capturar fallos de red (`IOException`), que requerirían otro catch o `runCatching`.

Consejo:
Para manejar errores de red vs HTTP usa runCatching o capa de Result en el repo. [withContext](pregunta:82)(IO) solo si mezclas APIs no [suspend](pregunta:79).
