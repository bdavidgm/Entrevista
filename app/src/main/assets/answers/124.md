## Concepto
La delegación de clase (class A : I by impl) permite que una clase implemente una interfaz reenviando todas sus llamadas a otra instancia, sin escribir manualmente cada método. Es la forma idiomática de Kotlin para aplicar el principio "[composición sobre herencia](pregunta:251)".

El delegado tiene que implementar la interfaz; los miembros se pueden sobreescribir individualmente si se necesita ajustar algún comportamiento.

Ejemplo:
interface Repository {
    suspend fun load(): List<User>
}

class RemoteRepository : Repository { ... }

class LoggingRepository(
    private val inner: Repository
) : Repository by inner {
    override suspend fun load(): List<User> {
        Log.d(TAG, "load()")
        return inner.load()
    }
}


ExplicaciónKotlin:
`interface Repository { suspend fun load(): List<User> }` — interfaz con [función suspend](pregunta:79)ida. `class RemoteRepository : Repository { ... }` implementación concreta. `class LoggingRepository(private val inner: Repository) : Repository by inner` — delegación de interfaz: compilador genera forward de métodos no sobrescritos a `inner`. `by inner` requiere que `inner` implemente `Repository`. `override suspend fun load(): List<User>` sobrescribe solo este método para logging antes/después. `Log.d(TAG, "load()")` side effect Android. `return inner.load()` llama implementación delegada explícitamente. Patrón decorador sin herencia múltiple; `private val inner` retiene referencia al repo real. Diferente de `val x by lazy { }` que es delegación de propiedad.

Consejo:
Distinguir de la delegación de propiedades (val x [by lazy](pregunta:16)). Útil para decoradores (logging, cache, métrica) sin extender una clase concreta.
