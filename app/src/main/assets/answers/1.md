## Concepto
La null safety es la característica de Kotlin que distingue en el sistema de tipos las referencias anulables (String?) de las no anulables (String). El compilador exige comprobar o convertir antes de usar un valor anulable, convirtiendo muchas NullPointerException de runtime en errores de compilación.

Operadores clave: ?. ([safe call](pregunta:17)), ?: ([Elvis](pregunta:2)), !! (force, peligroso) y requireNotNull. Tras un if (x != null), el [smart cast](pregunta:123) trata x como no anulable dentro del bloque. Los tipos provenientes de Java son platform types (T!), donde el compilador no conoce la nullabilidad.

Ejemplo:
fun showEmail(user: User?) {
    val email = user?.email ?: "Sin correo"
    textView.text = email
}

ExplicaciónKotlin:
La función `showEmail` declara el parámetro `user` como `User?`, un tipo anulable que obliga al compilador a tratar cualquier acceso como potencialmente inexistente. La expresión `user?.email` usa el [safe call](pregunta:17) (`?.`): si `user` es null, toda la cadena se cortocircuita y el resultado es null sin lanzar excepción; si no lo es, accede a la propiedad `email` (que también puede ser anulable según el modelo). El [operador Elvis](pregunta:2) `?: "Sin correo"` evalúa el operando derecho solo cuando el izquierdo es null, asignando un valor por defecto de tipo `String` no anulable a `email`. Kotlin infiere el tipo de `email` como `String` gracias a la rama derecha del [Elvis](pregunta:2). Por último, `textView.text = email` asigna a una propiedad Java/Kotlin de la vista; aquí `email` ya es no anulable, por lo que no hace falta `!!` ni comprobaciones adicionales.

Consejo:
Evita !! salvo invariantes demostrables; prefiere ?., ?: o early return. Menciona los platform types como punto débil al interoperar con Java.
