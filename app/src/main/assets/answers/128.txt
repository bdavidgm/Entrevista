Concepto:
Parcelable es la interfaz de Android para serializar objetos en un Parcel de forma eficiente y pasarlos entre componentes (Intent extras, Bundle, IPC, SavedState) con menor overhead que Serializable de Java.

Explicación detallada:
Parcelable escribe campos en orden con Parcel.write* y los lee en el mismo orden en el constructor o factory CREATOR.

Ventajas vs Serializable:
- Más rápido y menos allocations (importante en intents y transacciones frecuentes).
- Serializable usa reflexión y genera más basura.

En Kotlin moderno:
- @Parcelize en data class (plugin kotlin-parcelize): el compilador genera el boilerplate.
- Requiere plugins { id("kotlin-parcelize") } y @Parcelize annotation.

API 33+ (Tiramisu):
- intent.getParcelableExtra("key", MyClass::class.java) tipado.
- Bundle.getParcelableArrayList con Class parameter.

Limitaciones:
- Solo para datos simples y estables; no persistencia a largo plazo ni red.
- Cambiar estructura del Parcelable rompe compatibilidad si no versionas.
- Objetos grandes en Intent pueden causar TransactionTooLargeException (límite ~1MB binder).

Alternativas:
- Navigation Safe Args para argumentos tipados entre destinos.
- JSON en SavedStateHandle para objetos complejos (con cuidado de tamaño).
- Room/BD para datos grandes.

Cuándo usarlo:
Argumentos entre Activities/Fragments, estado en Bundle, AIDL. Para red usa DTO JSON; para disco usa Room.

Ejemplo:
@Parcelize
data class User(
    val id: String,
    val name: String
) : Parcelable

// Enviar
val intent = Intent(this, DetailActivity::class.java).apply {
    putExtra("user", User("1", "Ana"))
}
startActivity(intent)

// Recibir API 33+
val user = intent.getParcelableExtra("user", User::class.java)

Consejo para entrevista:
Contrasta Parcelable vs Serializable (rendimiento). Menciona @Parcelize y TransactionTooLargeException. Safe Args como alternativa moderna con Navigation.
