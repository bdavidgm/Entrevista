Concepto:
Desde Android 8 (API 26) las apps en background no pueden iniciar servicios en segundo plano libremente; Android 12+ endurece FGS y Android 14 exige tipos declarados; el patrón recomendado es WorkManager, FGS tipado solo cuando el usuario percibe el trabajo.

Explicación detallada:
Android 8 Oreo:
- BackgroundExecutionLimits: al ir a background, ventana breve luego se restringe.
- startService() en background lanza IllegalStateException salvo excepciones (temp whitelist, foreground app).
- Solución: startForegroundService + notificación pronto, o JobScheduler/WorkManager.

Android 9+: restricciones adicionales en acceso Wi‑Fi/scan en background.

Android 12:
- Tiempo limitado para pasar a foreground tras startForegroundService.
- Exact alarms y FGS desde background más estrictos.

Android 14:
- foregroundServiceType obligatorio y permisos por tipo (dataSync, location, etc.).
- Restricción iniciar FGS desde background salvo casos documentados (user-visible action, high-priority FCM, etc.).

Implicaciones prácticas:
- Sync periódico → WorkManager con Constraints.
- Ubicación continua → FGS tipo location + permiso foreground location + justificación Play.
- Descarga archivo → WorkManager o DownloadManager, no Service started largo.
- Boot completed → programar WorkManager, no arrancar red pesada al instante.

Excepciones whitelist corta: app temporalmente en lista tras acción usuario (botón), companion device, etc. — consultar documentación vigente.

Cuándo usarlo:
- Diseñar cualquier feature background post-2017.
- Auditar app para cumplir políticas Play y evitar kills.

Ejemplo:
// Mal (background startService)
// context.startService(Intent(context, SyncService::class.java)) // puede fallar

// Bien
WorkManager.getInstance(context).enqueue(
    OneTimeWorkRequestBuilder<SyncWorker>()
        .setConstraints(
            Constraints.Builder()
                .setRequiredNetworkType(NetworkType.CONNECTED)
                .build()
        )
        .build()
)

// Si usuario pulsa "Navegar"
context.startForegroundService(Intent(context, NavigationService::class.java))

Consejo para entrevista:
Cita Android 8 como punto de inflexión, FGS con notificación, tipos API 34, y WorkManager como default. Menciona que “background service” clásico está muerto para sync. Da ejemplo usuario-visible vs sync silencioso.
