Concepto:
Un Looper es un bucle de mensajes asociado a un hilo, y un Handler es la API para encolar Runnables o Messages en ese bucle. Forman la base del modelo de concurrencia clásico de Android: el hilo principal tiene un Looper que procesa los mensajes de UI y entrada.

Hoy en día se prefieren coroutines + dispatchers, pero Handler/Looper aún aparecen en APIs como postDelayed, en el patrón main-thread executor y en código legado.

Ejemplo:
val handler = Handler(Looper.getMainLooper())
handler.postDelayed({
    textView.text = "Hola"
}, 1_000)


ExplicaciónKotlin:
`Handler(Looper.getMainLooper())` construye handler atado al looper del hilo UI; `getMainLooper()` es estático Java accesible desde Kotlin. `handler.postDelayed({ ... }, 1_000)` encola lambda Runnable con retardo en milisegundos; el literal usa separador `_`. Dentro del lambda, `textView.text = "Hola"` muta propiedad Java `text` — Kotlin permite asignación de propiedad sintética. La lambda es closure que captura `textView` del scope externo. No hay corrutinas: patrón clásico pre-coroutines. El segundo argumento `1_000` es `Long` inferido por sobrecarga de `postDelayed`. Equivalente moderno sería `viewModelScope.launch { delay(1000); ... }` en Main.

Consejo:
"Looper = bucle de mensajes del hilo; Handler = la cola para postear". En código moderno prefiere viewModelScope + Dispatchers.Main.
