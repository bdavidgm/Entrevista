Concepto:
App Startup es una librería de Jetpack que permite inicializar componentes al arranque de la aplicación de forma declarativa y ordenada, sin sobrecargar Application.onCreate con código secuencial pesado. Usa ContentProvider bajo el capó de forma optimizada.

Explicación detallada:
Implementas Initializer<T> con create(context) y dependencies() que declara otros Initializers que deben ejecutarse antes. En manifest o androidx.startup configuras qué inicializadores corren. Startup ejecuta el grafo de dependencias en orden topológico y puede hacer trabajo en background según configuración.

Casos: Timber, Firebase opcional, configuración de Coil, migración ligera. Evita trabajo pesado de red en Initializer; retrasa arranque y StrictMode puede alertar. Prefer WorkManager para sync pesado.

Comparación con Application.onCreate: más modular y testeable por inicializador; varios módulos de librería pueden registrar su Initializer sin tocar Application del host.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Cuándo usarlo:
SDKs que requieren init temprano, varios módulos con setup. Si solo tienes 2 líneas en Application, puede ser overkill.

Ejemplo:
class TimberInitializer : Initializer<Unit> {
    override fun create(context: Context) {
        if (BuildConfig.DEBUG) Timber.plant(Timber.DebugTree())
    }
    override fun dependencies(): List<Class<out Initializer<*>>> = emptyList()
}

// AndroidManifest.xml dentro de <provider> androidx.startup
// tools:node="merge" y meta-data para deshabilitar auto-init si hace falta

Consejo para entrevista:
Menciona orden de dependencias y no bloquear main thread. Contrasta con lazy init en primer uso. Pregunta: ¿Reemplaza Application? No del todo, pero reduce su tamaño.
