## Concepto
inline indica al compilador que copie el cuerpo de la función en el lugar de la llamada en vez de generar una invocación normal. En funciones de orden superior evita crear un objeto Function por cada lambda, lo que ahorra allocations en bucles y APIs frecuentes.

Su uso típico es en utilidades pequeñas con lambdas (forEach, [scope functions](pregunta:21)) y es obligatorio para usar [reified](pregunta:23), que conserva el tipo genérico en runtime. crossinline y noinline modulan el comportamiento de las lambdas que recibe.

Ejemplo:
inline fun <T> measureMs(block: () -> T): Pair<T, Long> {
    val start = System.currentTimeMillis()
    val result = block()
    return result to (System.currentTimeMillis() - start)
}

inline fun <reified T> Bundle.getOrNull(key: String): T? =
    get(key) as? T

ExplicaciónKotlin:
`inline fun <T> measureMs(block: () -> T): Pair<T, Long>` es una función inline genérica: el compilador copia el cuerpo en cada call site, evitando el coste de una lambda como objeto en muchos casos. `block: () -> T` es un parámetro de tipo función sin argumentos que devuelve `T`. Dentro se invoca `block()` y se empaqueta el resultado con el tiempo en `Pair` usando el infix `to`: `result to (System.currentTimeMillis() - start)`. `inline fun <reified T> Bundle.getOrNull(key: String): T?` combina inline con `reified`: el tipo `T` está disponible en runtime en cada call site, permitiendo `get(key) as? T` con cast seguro (`as?` devuelve null si falla). Al ser extension function sobre `Bundle`, el receptor es el bundle sobre el que se llama el método. `reified` solo funciona dentro de funciones `inline` porque el compilador sustituye `T` por el tipo concreto en cada invocación.

Consejo:
No marques inline funciones grandes: aumentan el bytecode. [Reified](pregunta:23) solo funciona en inline. Aprovecha para hablar de crossinline cuando se necesita evitar non-local returns.
