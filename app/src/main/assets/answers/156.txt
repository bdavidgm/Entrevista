Concepto:
Un módulo en Android (Gradle module) es una unidad de compilación separada con su propio build.gradle.kts, código fuente y dependencias, que el proyecto ensambla en una librería (AAR) o aplicación (APK/AAB).

Explicación detallada:
Tipos comunes:
- :app — application module, punto de entrada, applicationId.
- Library modules (com.android.library): features, core, data — generan AAR consumidos por otros.
- Dynamic feature modules (opcional): entrega bajo demanda.

settings.gradle.kts:
include(":app", ":core:ui", ":feature:login")

Dependencias entre módulos:
implementation(project(":core:network"))
- api vs implementation: api expone transitivamente al consumidor.

Ventajas modularización:
- Tiempos de compilación incrementales.
- Límites de arquitectura (feature no depende de otra feature directamente).
- Reutilización y equipos por módulo.
- Test por módulo.

Convenciones:
- :core:model, :core:data, :feature:* 
- Evitar dependencias circulares.

Cuándo usarlo:
Proyectos medianos/grandes, multi-feature, compartir SDK interno. Apps triviales pueden ser monolito single-module.

Ejemplo:
// settings.gradle.kts
include(":app")
include(":core:domain")
include(":core:data")
include(":feature:profile")

// app/build.gradle.kts
dependencies {
    implementation(project(":feature:profile"))
    implementation(project(":core:data"))
}

// feature/profile/build.gradle.kts
plugins { alias(libs.plugins.android.library) }
dependencies {
    implementation(project(":core:domain"))
}

Consejo para entrevista:
Diferencia app vs library module. Menciona implementation vs api y compilación incremental. Relaciona con Clean Architecture por capas físicas.
