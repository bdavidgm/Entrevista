Concepto:
ViewModel sobrevive a los cambios de configuración (rotación, modo oscuro, cambio de idioma) porque está asociado al ViewModelStore del componente, que la Activity/Fragment retiene cuando se destruye y recrea por configuración. Así se evita perder estado y volver a pedir datos a la red.

ViewModel no sobrevive a process death directamente; para eso se combina con SavedStateHandle.

Ejemplo:
@HiltViewModel
class HomeViewModel @Inject constructor(
    private val repo: HomeRepository
) : ViewModel() {
    val state: StateFlow<HomeState> = repo.observe()
        .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), HomeState.Loading)
}

class HomeFragment : Fragment() {
    private val viewModel: HomeViewModel by viewModels()
}

ExplicaciónKotlin:
@HiltViewModel es una anotación de Hilt que marca la clase para que el contenedor DI genere el factory del ViewModel. class HomeViewModel @Inject constructor(...) indica inyección por constructor: Hilt proveerá HomeRepository al crear la instancia. private val repo limita la visibilidad y declara una propiedad inmutable de solo lectura. : ViewModel() hereda de la clase base de Jetpack con el constructor vacío de Kotlin. val state: StateFlow<HomeState> declara un flujo caliente tipado con el genérico HomeState. La cadena repo.observe().stateIn(...) encadena operadores de Flow: observe() devuelve un Flow, y stateIn lo convierte en StateFlow dentro de viewModelScope (scope del ViewModel cancelado en onCleared). SharingStarted.WhileSubscribed(5_000) usa un literal numérico con separador de miles (_) válido en Kotlin. HomeFragment usa by viewModels() que es una propiedad delegada (property delegate) de KTX que resuelve el ViewModel lazy y ligado al Fragment.

Consejo:
"Rotación = ViewModel; muerte de proceso = SavedStateHandle". Esto resuelve la pérdida de datos sin recargar y separa la lógica de la UI.
