## Concepto
AccountManager centraliza cuentas online del usuario (tipo Google, corporativas) y tokens reutilizables entre apps con el consentimiento adecuado. Expone APIs para añadir cuenta, obtener auth token y sincronizar.

Muchas apps consumer ya no lo usan directamente: OAuth en [WebView](pregunta:268)/custom tabs o credenciales por app.

Ejemplo:
val am = AccountManager.get(this)
val accounts = am.getAccountsByType("com.google")

ExplicaciónKotlin:
`AccountManager.get(this)` es una llamada estática Java que recibe un `Context` (`this` de [Activity](pregunta:31)/[Application](pregunta:128)) y devuelve el singleton del gestor de cuentas, asignado a `val am`. `getAccountsByType("com.google")` filtra por tipo de cuenta y devuelve un array Java `Array<Account>` que Kotlin expone como arreglo con índice y `.size`, iterable con `for (account in accounts)`. El literal `"com.google"` es `String` no anulable. No hay lambdas, [corrutinas](pregunta:78) ni [null safety](pregunta:1) explícita en el snippet: asume permisos concedidos y cuentas existentes. En Kotlin moderno este patrón suele envolverse en funciones [suspend](pregunta:79) o repositorios que manejen permisos y cuentas vacías con tipos como `List<Account>` mediante `.toList()` sobre el array devuelto.

Consejo:
GET_ACCOUNTS y flujos legacy cambiaron mucho con permisos y scoped access. Para identidad moderna menciona Credential Manager / Sign in with Google.
