## Concepto
withContext es una [función suspend](pregunta:79) que cambia el [CoroutineContext](pregunta:30) (típicamente el Dispatcher) para el bloque que ejecuta y restaura el contexto anterior al terminar. Permite escribir código secuencial que corre en IO o Main según la fase sin bloquear hilos.

## Explicación detallada
withContext([Dispatchers.IO](pregunta:30)) { heavyWork() } mueve la ejecución al pool IO y devuelve el resultado al caller. Tras el bloque, vuelves al dispatcher del scope padre (por ejemplo Main si launch fue en [viewModelScope](pregunta:131) con Main implícito en actualización UI).

Uso típico en [ViewModel](pregunta:68): launch en [viewModelScope](pregunta:131) (Default o Main) -> withContext(IO) para red/BD -> actualizar [StateFlow](pregunta:86) (thread-safe con update). No mezcles [Dispatchers.IO](pregunta:30) en cada línea si un bloque entero es IO.

withContext es preferible a launch anidado cuando necesitas el resultado de forma secuencial. Para paralelo usa async.

NonCancellable: contexto especial para flush de datos tras cancelación del usuario. test: StandardTestDispatcher con runTest.

## Cuándo usarlo
Cambiar dispatcher para secciones concretas de trabajo [suspend](pregunta:79). Evita withContext(Main) innecesario si ya estás en Main y solo actualizas state.

Ejemplo:
suspend fun fetchAndMap(): List<Item> = withContext(Dispatchers.IO) {
    api.getItems().map { it.toDomain() }
}

viewModelScope.launch {
    val items = fetchAndMap()
    _uiState.update { it.copy(items = items) }
}

ExplicaciónKotlin:
[suspend](pregunta:79) fun fetchAndMap(): List<Item> = withContext([Dispatchers.IO](pregunta:30)) { api.getItems().map { it.toDomain() } } declara [función suspend](pregunta:79) de nivel superior o miembro con retorno List<Item>. withContext([Dispatchers.IO](pregunta:30)) { ... } ejecuta bloque en IO y devuelve último valor del lambda (resultado de map). api.getItems() presumiblemente [suspend](pregunta:79); .map { it.toDomain() } transforma lista con lambda y extensión toDomain(). [viewModelScope](pregunta:131).launch { val items = fetchAndMap(); _uiState.update { it.copy(items = items) } } secuencia: launch, llamada [suspend](pregunta:79) fetchAndMap(), update con copy nombrado items = items. Separar fetchAndMap permite reuso y test unitario de [función suspend](pregunta:79) pura.

Consejo para entrevista:
Contrasta withContext vs launch anidado. Menciona que es [suspend](pregunta:79) y devuelve valor. No confundas con [flowOn](pregunta:145) (para [Flow](pregunta:84)).
