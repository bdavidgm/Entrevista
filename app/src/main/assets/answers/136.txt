Concepto:
recomposeScope (ámbito de recomposición) es el límite del compilador de Compose dentro del cual un cambio de estado provoca recomposición de los Composables hijos afectados, sin invalidar todo el árbol de la app.

Explicación detallada:
Cada función @Composable se compila en una función que recibe Composer y slot table. El compiler inserta grupos (groups) y keys para rastrear qué leer en cada pasada.

Cuando cambia un State observado:
- Compose marca scopes inválidos desde el punto de lectura hacia arriba hasta el root del subtree afectado.
- Solo esos Composables se re-ejecutan (recomposition), no necesariamente padres que no leyeron ese state.

Optimizaciones:
- Skip: si parámetros son estables e iguales, el hijo puede saltarse.
- strong skipping mode (Kotlin 2 + compiler plugin): más agilidad para evitar trabajo.
- Mover lectura de state lo más abajo posible en el árbol (principle of state locality) reduce trabajo.

Herramientas:
- Layout Inspector recomposition counts.
- Log recompositions en debug con Modifier o tooling.

No confundir con CoroutineScope ni ViewModelScope: es concepto de UI declarativa.

Cuándo usarlo:
Al optimizar pantallas con listas, contadores o animaciones que recomponen de más. Entender scopes ayuda a diseñar state hoisting correcto.

Ejemplo:
@Composable
fun Parent() {
    var count by remember { mutableIntStateOf(0) }
  // Solo Counter lee count; si Title no lee count, idealmente no recompondría
  // (depende de estabilidad y estructura)
    Title()           // sin state
    Counter(count)    // lee count -> su scope se invalida
    Button(onClick = { count++ }) { Text("+") }
}

@Composable
fun Counter(value: Int) {
    Text("Count: $value") // lectura de state aquí acota invalidación
}

Consejo para entrevista:
Explica que recomposition no es layout ni draw completos siempre. Menciona estabilidad, keys en LazyColumn y leer state en el nivel más bajo posible.
