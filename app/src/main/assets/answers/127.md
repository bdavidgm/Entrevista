## Concepto
Un Context es un objeto que representa el "entorno" actual de la app: da acceso a recursos, system services, assets, base de datos, [SharedPreferences](pregunta:177) y permite lanzar activities, services o broadcasts. Casi todo en Android necesita un Context para funcionar.

Existen distintas variantes: ApplicationContext (vive toda la app), [Activity](pregunta:31) Context (vive con la [Activity](pregunta:31)), [Service](pregunta:36) Context. Guardar un [Activity](pregunta:31) Context fuera de su ciclo de vida es la causa más común de [memory leak](pregunta:117)s.

Ejemplo:
class Analytics(private val context: Context) {
    fun log(event: String) {
        context.applicationContext // seguro de guardar
    }
}

val service = context.getSystemService(Context.CONNECTIVITY_SERVICE)
        as ConnectivityManager


ExplicaciónKotlin:
`class Analytics(private val context: Context)` — constructor primario con Context Android. `fun log(event: String)` recibe string. `context.applicationContext` accede getter que retorna Context de aplicación (seguro almacenar vs [Activity](pregunta:31)). `val service = context.getSystemService(Context.CONNECTIVITY_SERVICE) as ConnectivityManager` — `getSystemService` retorna `Any?`; cast inseguro `as` a tipo concreto (preferible `as?` o getSystemService tipado en API moderna). `Context.CONNECTIVITY_SERVICE` constante string. Interop Java/Android desde Kotlin: Context es clase Android, no Kotlin stdlib. [Memory leak](pregunta:117) si se guardara `context` de [Activity](pregunta:31) en singleton.

Consejo:
"Usa applicationContext para singletons; [Activity](pregunta:31) context solo para UI/diálogos". [Memory leak](pregunta:117) clásico: guardar this de una [Activity](pregunta:31) en un objeto estático.
