## Concepto
[Flow](pregunta:84) y [LiveData](pregunta:69) son mecanismos observables para UI, pero [Flow](pregunta:84) es un stream frío de Kotlin con operadores ricos y [coroutines](pregunta:78), mientras [LiveData](pregunta:69) es un contenedor lifecycle-aware de Android con API más limitada y sin cold semantics por defecto.

## Explicación detallada
[LiveData](pregunta:69): observe con [LifecycleOwner](pregunta:44), setValue/postValue, integración histórica XML. No tiene debounce nativo, combine complejo, ni backpressure explícito. Obsoleto para código nuevo según guías actuales, pero perviviente.

[Flow](pregunta:84): map, combine, flatMapLatest, catch, retry, integración [Room](pregunta:70)/[Retrofit](pregunta:207) [suspend](pregunta:79). Requiere collector con lifecycle (repeatOnLifecycle) o Compose [collectAsStateWithLifecycle](pregunta:134). stateIn convierte a hot [StateFlow](pregunta:86) para UI.

Migración: [LiveData](pregunta:69) en [Repository](pregunta:63) -> [Flow](pregunta:84); [ViewModel](pregunta:68) stateIn o expone [Flow](pregunta:84); [Fragment](pregunta:32) collect en STARTED.

[Room](pregunta:70) devuelve ambos; preferir [Flow](pregunta:84). [Paging 3](pregunta:76) usa [Flow](pregunta:84)<PagingData>.

Threading: [LiveData](pregunta:69) postValue; [Flow](pregunta:84) [flowOn](pregunta:145) y [withContext](pregunta:82) en collector.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, [ANR](pregunta:95), Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto de tu experiencia o proyecto demo, y trade-offs frente a alternativas. ## Cuándo usarlo
[Flow](pregunta:84) en proyectos Kotlin nuevos; [LiveData](pregunta:69) solo legacy o APIs que solo exponen [LiveData](pregunta:69).

Ejemplo:
// Antes LiveData
val user: LiveData<User> = repo.observeUser()

// Moderno
val user: StateFlow<User> = repo.observeUser()
    .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), User.Empty)

ExplicaciónKotlin:
Comentarios // Antes [LiveData](pregunta:69) y // Moderno separan estilos. val user: [LiveData](pregunta:69)<User> = repo.observeUser() tipado explícito [LiveData](pregunta:69) Java arquitectura. val user: [StateFlow](pregunta:86)<User> = repo.observeUser().stateIn([viewModelScope](pregunta:131), SharingStarted.WhileSubscribed(5_000), User.Empty) convierte [Flow](pregunta:84) a [StateFlow](pregunta:86) hot con valor inicial User.Empty ([companion object](pregunta:7) o constante). stateIn requiere scope, SharingStarted y initialValue. Mismo repo.observeUser() supone [Flow](pregunta:84) subyacente en versión moderna. WhileSubscribed(5_000) detiene upstream 5 segundos sin suscriptores. Contraste muestra migración API manteniendo tipado genérico User en [LiveData](pregunta:69) vs [StateFlow](pregunta:86).

Consejo para entrevista:
Tabla mental: lifecycle (ambos con wrapper), operadores ([Flow](pregunta:84) gana), testing ([Turbine](pregunta:93) vs InstantTaskExecutorRule). No digas [LiveData](pregunta:69) está prohibido; di preferencia [Flow](pregunta:84).
