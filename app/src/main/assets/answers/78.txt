Concepto:
suspend es un modificador de Kotlin que marca una función que puede pausar y reanudar de forma segura dentro de una coroutine, sin bloquear el hilo subyacente. Solo puede llamarse desde otra función suspend o desde un coroutine builder como launch.

Explicación detallada:
El compilador transforma suspend en máquina de estados (continuations). Cuando la función espera IO, libera el hilo para otras coroutines. Retrofit con suspend fun, Room con suspend DAO, y delay() son ejemplos idiomáticos.

Llamar suspend desde código no suspend requiere launch o runBlocking (tests). Encadenar varias suspend es código lineal legible frente a callbacks hell.

suspendCancellableCoroutine convierte callbacks legacy a suspend. withTimeout y withContext son suspend estándar.

Cancellation: las suspend deben ser cooperativas; revisar isActive o usar funciones cancelables de kotlinx.coroutines.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Cuándo usarlo:
APIs que esperan resultado único async. Para streams usar Flow, no suspend repetido en loop manual.

Ejemplo:
interface ApiService {
    @GET("users/{id}")
    suspend fun getUser(@Path("id") id: Long): UserDto
}

suspend fun loadUser(id: Long): User = withContext(Dispatchers.IO) {
    api.getUser(id).toDomain()
}

Consejo para entrevista:
Define suspend como pausable, no como async automático en otro hilo (el dispatcher decide). Pregunta: ¿puedo llamar suspend desde onClick directo? No, necesitas scope.launch.
