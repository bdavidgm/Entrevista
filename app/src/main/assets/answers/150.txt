Concepto:
El patrón Mapper transforma datos entre capas (DTO de red, entidad Room, modelo de dominio, UiModel) manteniendo cada capa independiente y evitando que la UI conozca detalles de serialización o BD.

Explicación detallada:
Motivación:
- API devuelve JSON con nombres y tipos distintos al dominio.
- Room Entity tiene anotaciones y columnas que no deben filtrarse a Compose.
- Separación Clean Architecture: capa de datos != dominio != presentación.

Implementación:
- Funciones puras: fun UserDto.toDomain(): User
- Clases Mapper @Inject si hay dependencias (formateo, locale).
- MapStruct en Java; en Kotlin extension functions o clases dedicadas.

Direcciones:
- Dto -> Domain en Repository tras fetch.
- Entity <-> Domain en capa data.
- Domain -> UiModel en ViewModel o mapper de presentación.

Evitar:
- Mapear en Composable (difícil de testear).
- Un solo modelo para todo (acopla API a cambios de UI).

Cuándo usarlo:
Siempre que haya más de una representación del mismo concepto. Especialmente con múltiples fuentes (cache + network).

Ejemplo:
data class UserDto(val user_id: String, val full_name: String)
data class User(val id: String, val name: String)
data class UserUiModel(val id: String, val displayName: String)

fun UserDto.toDomain() = User(id = user_id, name = full_name)
fun User.toUiModel() = UserUiModel(id = id, displayName = name.uppercase())

class UserRepository @Inject constructor(private val api: UserApi, private val dao: UserDao) {
    suspend fun refresh(id: String): User {
        val dto = api.getUser(id)
        val domain = dto.toDomain()
        dao.insert(domain.toEntity())
        return domain
    }
}

Consejo para entrevista:
Explica por capas y testabilidad de funciones puras. Menciona extension toDomain() idiomática en Kotlin vs clase Mapper cuando la lógica crece.
