Concepto:
InputMethodManager (IMM) controla el teclado virtual (IME): mostrarlo, ocultarlo y consultar métodos de entrada disponibles, asociado al Context de una Activity con ventana enfocada.

Explicación detallada:
El flujo de entrada de texto pasa por EditText/Compose TextField → InputConnection → IME del usuario (Gboard, SwiftKey). IMM métodos habituales:
- showSoftInput(view, flags): muestra teclado para una View con foco.
- hideSoftInputFromWindow(windowToken, flags): oculta usando token de la ventana.
- toggleSoftInput(show, hideSoftInputOnly)
- isActive(view): si hay IME activo para esa vista.

En Compose, el enfoque moderno es LocalSoftwareKeyboardController (show/hide) en lugar de IMM directo, pero internamente sigue el mismo stack.

Flags: SHOW_IMPLICIT, HIDE_NOT_ALWAYS. Para ScrollView con campos, ajusta windowSoftInputMode en manifest (adjustResize, adjustPan) o WindowInsets en Compose (imePadding()).

Problemas comunes: llamar showSoftInput antes de que la View tenga foco (post { requestFocus(); show }); fugas si guardas Activity en listener; teclado que tapa botones (solución: insets, no hacks de margin fijos).

Cuándo usarlo:
- Botón “Buscar” que abre teclado explícitamente.
- Cerrar teclado al enviar formulario o navegar.
- Pantallas login con auto-focus en primer campo (con retraso post-layout).

Ejemplo:
// Views tradicional
val imm = getSystemService(InputMethodManager::class.java)
editText.requestFocus()
imm.showSoftInput(editText, InputMethodManager.SHOW_IMPLICIT)

// Ocultar
imm.hideSoftInputFromWindow(editText.windowToken, 0)

// Compose
val keyboard = LocalSoftwareKeyboardController.current
keyboard?.hide()

Consejo para entrevista:
Prefiere WindowInsets/imePadding en Compose y windowSoftInputMode bien elegido. Si mencionas IMM, demuestra conocer windowToken y foco. Contrasta con enableEdgeToEdge y Behavior del IME en Material3.
