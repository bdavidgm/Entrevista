Concepto:
El Repository es el componente que abstrae las fuentes de datos (red, base de datos, cache, sensores) detrás de una API orientada al dominio. La capa de presentación nunca habla con Retrofit ni Room directamente: pide datos al Repository, que decide de dónde sacarlos y cómo combinarlos.

Permite cambiar implementación sin tocar la UI, facilita los tests con fakes y centraliza políticas como cache-first, offline-first o single source of truth.

Ejemplo:
class UserRepository(
    private val api: UserApi,
    private val dao: UserDao
) {
    fun users(): Flow<List<User>> = dao.getAll()

    suspend fun refresh() {
        val remote = api.fetchUsers()
        dao.insertAll(remote.map { it.toEntity() })
    }
}

ExplicaciónKotlin:
UserRepository es una clase Kotlin con constructor primario que declara private val api: UserApi y private val dao: UserDao como dependencias inyectables manualmente. fun users(): Flow<List<User>> = dao.getAll() usa expresión de cuerpo único (=) devolviendo Flow genérico anidado List<User>. suspend fun refresh() marca función suspendible solo invocable desde coroutine o otra suspend. val remote = api.fetchUsers() asigna resultado de llamada suspend. dao.insertAll(remote.map { it.toEntity() }) aplica map sobre la lista: lambda { it.toEntity() } transforma cada elemento con extensión o método toEntity(). insertAll es suspend del DAO. El patrón combina programación funcional (map) con corrutinas (suspend) sin bloquear hilos.

Consejo:
"Single source of truth": para offline-first el DAO emite y la red solo refresca. Devuelve tipos del dominio, no DTOs.
