Concepto:
Clean Architecture (Uncle Bob adaptada a Android) organiza el código en capas concéntricas: entidades y reglas de negocio en el centro, casos de uso (use cases), interfaces de repositorio, y capas externas de UI y frameworks. La regla de dependencia: las capas internas no conocen Android, Retrofit ni Room.

Explicación detallada:
Capa de dominio (:domain): entidades puras Kotlin, interfaces Repository, use cases (GetUserUseCase) con una responsabilidad. Sin imports de android.*. Capa de datos (:data): implementaciones de repositorio, DTOs, Retrofit, Room, mappers. Capa de presentación (:app o :feature): ViewModel, Compose, Navigation.

Los use cases encapsulan una acción de negocio ("comprar", "validar cupón") y coordinan repositorios. El ViewModel orquesta use cases, no reglas de negocio complejas inline. Esto escala en equipos grandes y módulos múltiples.

Dependency Inversion: domain define UserRepository interface; data implementa UserRepositoryImpl. Hilt enlaza en módulos @Module @InstallIn. Beneficios: tests de dominio sin emulador, sustituir Firebase por backend propio cambiando :data.

Costes: más boilerplate, curva de aprendizaje, riesgo de sobre-arquitectura en apps triviales. Para entrevista junior-mid, explica el principio aunque tu proyecto use MVVM simplificado con repository único.

Modularización: feature modules dependen de :domain, no entre sí lateralmente sin interfaces. shared :core para utilidades.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto de tu experiencia o proyecto demo, y trade-offs frente a alternativas. Mencionar versiones recientes de API o biblioteca (Material3, Paging 3, Hilt 2.x) demuestra que sigues el ecosistema, sin necesidad de memorizar números de versión exactos.

Para profundizar tras la entrevista, revisa documentación oficial, codelabs y el código fuente de tu propio proyecto Entrevista aplicando el concepto en un flujo mínimo reproducible. La práctica deliberada con tests unitarios refuerza la confianza al explicar el tema en viva voz.

Preguntas de seguimiento frecuentes en entrevistas para este tema incluyen comparación con alternativas del ecosistema, implicaciones en testing y en el ciclo de vida de Android, y cómo lo aplicarías en una app modular con varios flavors. Si el entrevistador pide profundidad, menciona herramientas de diagnóstico (Profiler, Lint, StrictMode, Layout Inspector) y cómo validarías en CI que no hay regresiones.

En proyectos reales, documenta decisiones en ADR breves o en README del módulo para que el equipo no revierta el patrón por desconocimiento. La mejor respuesta de entrevista combina definición precisa, ejemplo Kotlin o Compose concreto y honestidad sobre límites: ninguna solución es gratis en complejidad o tamaño de APK.

Relaciona este concepto con los demás bloques del manifiesto Entrevista (arquitectura, corrutinas, seguridad, build) para demostrar visión de sistema completo, no conocimiento aislado de una API.

Cuándo usarlo:
Apps con vida larga, varios desarrolladores, lógica de negocio rica o múltiples clients (móvil, wear). Para prototipos o apps muy pequeñas, MVVM con repository puede bastar; menciona pragmatismo.

Ejemplo:
// domain
class GetProductsUseCase @Inject constructor(
    private val repo: ProductRepository
) {
    suspend operator fun invoke(): Result<List<Product>> = repo.getProducts()
}

// presentation
class CatalogViewModel @Inject constructor(
    private val getProducts: GetProductsUseCase
) : ViewModel() {
    fun load() = viewModelScope.launch { getProducts() }
}

Consejo para entrevista:
Dibuja círculos y flechas hacia dentro. Menciona Dependency Rule y use cases. Si preguntan "¿siempre Clean?": no, trade-off tamaño del equipo. Evita mezclar DTO de Retrofit en ViewModel.
