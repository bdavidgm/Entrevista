## Concepto
En Android existen varias pirámides de pruebas: unit tests (JVM, lógica pura), instrumented tests (dispositivo/emulador, framework Android), y UI tests ([Espresso](pregunta:91), [Compose UI Test](pregunta:267)). Se organizan por velocidad, coste y alcance.

## Explicación detallada
Unit tests en test/: [ViewModel](pregunta:68), [use case](pregunta:66)s, mappers, reducers [MVI](pregunta:65) sin [Robolectric](pregunta:266). Rápidos, CI barato. Instrumented en androidTest/: requieren dispositivo; DAO [Room](pregunta:70) real, integración con [Context](pregunta:127). UI tests: [Espresso](pregunta:91) para Views, createComposeRule para Compose; lentos y frágiles, usar para flujos críticos.

[Robolectric](pregunta:266) simula Android en JVM para tests intermedios (menos usado hoy con mejor testabilidad por capas). Snapshot/screenshot tests opcionales.

Proporción ideal: muchos unit, algunos integration, pocos UI end-to-end. [Hilt](pregunta:74) test modules, fakes, TestDispatcher para [coroutines](pregunta:78).

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, [ANR](pregunta:95), Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto de tu experiencia o proyecto demo, y trade-offs frente a alternativas. Mencionar versiones recientes de API o biblioteca (Material3, [Paging 3](pregunta:76), [Hilt](pregunta:74) 2.x) demuestra que sigues el ecosistema, sin necesidad de memorizar números de versión exactos.

Para profundizar tras la entrevista, revisa documentación oficial, codelabs y el código fuente de tu propio proyecto Entrevista aplicando el concepto en un flujo mínimo reproducible. La práctica deliberada con tests unitarios refuerza la confianza al explicar el tema en viva voz.

## Cuándo usarlo
Unit para regresión diaria; androidTest para migraciones [Room](pregunta:70) y permisos; UI para checkout o login crítico.

Ejemplo:
// test/ — unit
@Test
fun `when refresh then loading then data`() = runTest { /* ViewModel */ }

// androidTest/ — instrumented
@Test
fun migrateDatabase() { /* MigrationTestHelper */ }


ExplicaciónKotlin:
Los comentarios `// test/` y `// androidTest/` marcan source sets distintos en [Gradle](pregunta:109), no sintaxis Kotlin, pero el código de ejemplo usa anotaciones [JUnit](pregunta:89). `@Test` marca funciones de prueba ejecutables por el runner de [JUnit](pregunta:89); en Kotlin el nombre puede ir entre backticks con espacios (`when refresh then loading then data`), lo que mejora la legibilidad como especificación. `runTest { }` proviene de `kotlinx-coroutines-test` y crea un `TestScope` con un `TestDispatcher` controlado: las [corrutinas](pregunta:78) lanzadas dentro avanzan de forma determinista sin hilos reales, ideal para probar ViewModels. El bloque vacío `/* ViewModel */` indica dónde irían aserciones sobre `StateFlow` o llamadas a métodos del [ViewModel](pregunta:68). El segundo `@Test` en `androidTest/` compila contra el SDK Android y se ejecuta en dispositivo; `MigrationTestHelper` es API de [Room](pregunta:70) para validar migraciones de base de datos en entorno instrumentado.

Consejo para entrevista:
Dibuja pirámide. Menciona qué va en cada capa. No digas solo [Espresso](pregunta:91); unit tests son la base.
