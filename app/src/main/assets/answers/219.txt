Concepto:
Signing firma el binario para que Play y los dispositivos verifiquen autoría e integridad. Usas un keystore (upload key) que firma el AAB; Play puede gestionar la app signing key de distribución.

V1/JAR + V2/APK + V3/v4 esquemas de firma; release debe usar configuración segura.

Ejemplo:
android {
    signingConfigs {
        create("release") {
            storeFile = file("release.keystore")
            storePassword = System.getenv("STORE_PW")
            keyAlias = "upload"
            keyPassword = System.getenv("KEY_PW")
        }
    }
}

ExplicaciónKotlin:
El bloque pertenece a un `build.gradle.kts`, archivo ejecutado como script Kotlin con acceso al API del Android Gradle Plugin. `android { }` es una función de extensión que configura el módulo; dentro, `signingConfigs { }` agrupa configuraciones de firma. `create("release") { ... }` crea una entrada nombrada y ejecuta una lambda con receptor (`SigningConfig`), patrón habitual en Gradle Kotlin DSL. `storeFile = file("release.keystore")` usa la función `file()` del Project para resolver rutas relativas al módulo. `storePassword` y `keyPassword` se asignan desde `System.getenv(...)`, que devuelve `String?`; si la variable no existe, el valor será null y el build fallará en release, lo cual es deseable para no firmar con contraseñas vacías. `keyAlias = "upload"` es un literal de cadena que identifica la clave dentro del keystore. Las asignaciones con `=` configuran propiedades del objeto Gradle, no variables Kotlin locales. Este enfoque mantiene secretos fuera del código fuente y permite inyectarlos en CI.

Consejo:
Nunca subas keystore al repo en claro. Play App Signing permite rotación de upload key si pierdes acceso controlado.
