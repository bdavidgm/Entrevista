Concepto:
Un Service iniciado (started) ejecuta tareas en background tras startService/startForegroundService hasta que se detiene con stopSelf/stopService; un Service enlazado (bound) ofrece API a clientes mediante bindService y vive mientras haya conexiones activas.

Explicación detallada:
Started service:
- onCreate → onStartCommand(intent, flags, startId) por cada start.
- Retorna START_STICKY (reinicia si sistema mata), START_NOT_STICKY, START_REDELIVER_INTENT.
- No expone binder; cliente no llama métodos directamente.
- Debe migrar a foreground con notificación si trabajo visible al usuario o restricciones background (Android 8+).
- stopSelf(startId) cuando termina trabajo.

Bound service:
- onBind devuelve IBinder; cliente obtiene interfaz local y llama métodos.
- Muere cuando último cliente unbind (a menos que también esté started).
- Patrón: música, IPC en misma app, servicios en proceso local.

Started + bound combinados posibles.

Alternativas modernas: WorkManager para trabajo deferible; Foreground Service tipado para GPS/media; coroutines en proceso con ProcessLifecycleOwner para tareas cortas.

Cuándo usarlo:
- Started: descarga legacy, sincronización antes de WorkManager (evitar hoy).
- Bound: API en segundo plano mientras UI conectada (servicio de descarga con progreso callback).
- Foreground started: reproducción audio, tracking navegación.

Ejemplo:
// Started
class SyncService : Service() {
    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        // hacer trabajo; stopSelf(startId)
        return START_NOT_STICKY
    }
    override fun onBind(intent: Intent?) = null
}

// Bound
class LocalBinder : Binder() {
    fun getService(): MusicService = this@MusicService
}
override fun onBind(intent: Intent) = LocalBinder()
