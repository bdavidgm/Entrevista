Concepto:
Los permisos peligrosos (dangerous permissions) en Android protegen datos sensibles del usuario (cámara, ubicación, contactos, micrófono, SMS, etc.) y requieren declaración en el manifest más solicitud en tiempo de ejecución (runtime) con posible denegación y rationale.

Explicación detallada:
Normal permissions (INTERNET) se conceden en install. Dangerous: el usuario aprueba por sesión o permanentemente. registerForActivityResult con RequestPermission() o múltiple RequestMultiplePermissions.

shouldShowRequestPermissionRationale indica si mostrar explicación antes de volver a pedir. Denegación permanente: abrir ajustes con Intent ACTION_APPLICATION_DETAILS_SETTINGS.

Android 13+: permisos granulares (POST_NOTIFICATIONS, READ_MEDIA_IMAGES vs READ_EXTERNAL_STORAGE legacy). Background location requiere flujo en dos pasos y revisión Play.

Principio mínimo: pide solo cuando la feature lo necesita (just-in-time), no al arranque.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto de tu experiencia o proyecto demo, y trade-offs frente a alternativas. Mencionar versiones recientes de API o biblioteca (Material3, Paging 3, Hilt 2.x) demuestra que sigues el ecosistema, sin necesidad de memorizar números de versión exactos.

Cuándo usarlo:
Cualquier API que acceda datos privados o hardware sensible. Declara en manifest y maneja denegación con UX degradada.

Ejemplo:
val launcher = rememberLauncherForActivityResult(
    ActivityResultContracts.RequestPermission()
) { granted ->
    if (granted) startCamera() else showPermissionDenied()
}

fun requestCamera() {
    when {
        ContextCompat.checkSelfPermission(ctx, Manifest.permission.CAMERA) ==
            PackageManager.PERMISSION_GRANTED -> startCamera()
        else -> launcher.launch(Manifest.permission.CAMERA)
    }
}


ExplicaciónKotlin:
`rememberLauncherForActivityResult(ActivityResultContracts.RequestPermission()) { granted -> ... }` es API Compose que registra contrato de Activity Result; la lambda `{ granted -> }` recibe `Boolean` con resultado. `rememberLauncher` persiste el launcher entre recomposiciones. `fun requestCamera()` usa `when` sin argumento como sustituto de if-else encadenado (estilo Kotlin idiomático). `ContextCompat.checkSelfPermission(ctx, Manifest.permission.CAMERA) == PackageManager.PERMISSION_GRANTED` compara enteros de estado. `Manifest.permission.CAMERA` es constante string del manifest mergeado. `launcher.launch(Manifest.permission.CAMERA)` dispara diálogo del sistema. `startCamera()` y `showPermissionDenied()` serían funciones locales o métodos del composable scope. El patrón combina nullable safety, expresión `when`, y lambdas trailing en Compose.
