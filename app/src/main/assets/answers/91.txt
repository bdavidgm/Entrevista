Concepto:
Espresso es el framework de Google para pruebas de UI instrumentadas en Android Views: simula toques, escribe texto y verifica visibilidad y estado de vistas en el dispositivo o emulador de forma sincrónica e idle-aware.

Explicación detallada:
onView(withId(R.id.button)).perform(click()); onView(withText("Hola")).check(matches(isDisplayed())). IdlingResource espera async (Retrofit legacy, animaciones) antes de continuar.

ActivityScenario.launch para aislar Activity. No mezcles sleeps largos; usa idling o arquitectura testeable con contadores.

Compose: androidx.compose.ui.test (createComposeRule) es el equivalente moderno; Espresso sigue para apps XML. Semantics en Compose: onNodeWithText, performClick.

Tests lentos y flaky si dependen de red real: usa Hilt test o mocks. Ejecutar en CI con emulador o Firebase Test Lab.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto de tu experiencia o proyecto demo, y trade-offs frente a alternativas. Mencionar versiones recientes de API o biblioteca (Material3, Paging 3, Hilt 2.x) demuestra que sigues el ecosistema, sin necesidad de memorizar números de versión exactos.

Para profundizar tras la entrevista, revisa documentación oficial, codelabs y el código fuente de tu propio proyecto Entrevista aplicando el concepto en un flujo mínimo reproducible. La práctica deliberada con tests unitarios refuerza la confianza al explicar el tema en viva voz.

Cuándo usarlo:
Flujos críticos en XML, regresión UI. Compose nuevo: Compose Test. No para lógica de negocio (unit test).

Ejemplo:
@Test
fun clickShowsMessage() {
    ActivityScenario.launch(MainActivity::class.java)
    onView(withId(R.id.submit)).perform(click())
    onView(withId(R.id.message)).check(matches(withText("Enviado")))
}


ExplicaciónKotlin:
`@Test fun clickShowsMessage()` es un test instrumentado JUnit que corre en dispositivo. `ActivityScenario.launch(MainActivity::class.java)` usa la referencia de clase Kotlin (`KClass` vía `::class.java`) para arrancar la Activity en aislamiento. `onView(withId(R.id.submit))` selecciona una vista del árbol UI; `R.id.submit` es constante generada por el compilador de recursos Android. `.perform(click())` simula interacción; `.check(matches(withText("Enviado")))` encadena aserción sobre el texto visible. Espresso usa matchers y ViewActions importados estáticamente; en Kotlin se escriben igual que en Java. `withText("Enviado")` compara `CharSequence`; la cadena es literal Kotlin. Este test no usa corrutinas ni mocks: valida integración Activity + layout XML + listeners.

Consejo para entrevista:
Menciona idling y flakiness. Contrasta Espresso vs Compose UI test. Pirámide: pocos tests UI.
