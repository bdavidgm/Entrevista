## Concepto
onSaveInstanceState es el callback en el que la Activity (o Fragment) guarda un Bundle ligero con el estado mínimo que necesita restaurar si el sistema mata el proceso por falta de memoria o cambia la configuración. Se restaura en onCreate o en onRestoreInstanceState.

No sustituye a un almacenamiento persistente: es para datos UI transitorios (campo de búsqueda escrito, scroll). Para sobrevivir a process death se complementa con SavedStateHandle en ViewModel.

Ejemplo:
class SearchActivity : AppCompatActivity() {
    override fun onSaveInstanceState(outState: Bundle) {
        super.onSaveInstanceState(outState)
        outState.putString(KEY_QUERY, editText.text.toString())
    }

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        editText.setText(savedInstanceState?.getString(KEY_QUERY).orEmpty())
    }
}

ExplicaciónKotlin:
`class SearchActivity : AppCompatActivity()` define la Activity. `override fun onSaveInstanceState(outState: Bundle)` recibe el `Bundle` mutable donde se guarda estado; `super.onSaveInstanceState(outState)` delega primero al padre. `outState.putString(KEY_QUERY, editText.text.toString())` usa la API Android: `KEY_QUERY` es probablemente constante companion; `editText.text` es propiedad CharSequence convertida con `toString()`. En `onCreate`, `savedInstanceState: Bundle?` es anulable en el primer arranque. `savedInstanceState?.getString(KEY_QUERY).orEmpty()` encadena safe call, obtiene `String?` y aplica la extension `orEmpty()` de Kotlin (devuelve `""` si null). `editText.setText(...)` restaura el valor. El patrón combina nullability kotliniana con APIs clásicas de Activity para sobrevivir a process death ligero.

Consejo:
Bundle es ligero (<= ~500 KB). Para datos grandes usa Room/DataStore. En ViewModel, usa SavedStateHandle para combinar process death + rotación.
