Concepto:
La recomposición es el proceso por el cual el runtime de Compose vuelve a ejecutar funciones Composable afectadas cuando cambia el estado que leen, actualizando la UI de forma incremental y eficiente. Solo se recompone el subárbol necesario, no toda la pantalla completa si puede evitarse.

Explicación detallada:
Cuando mutableStateOf cambia, Compose marca lecturas asociadas y programa recomposition. El compilador genera grupos y keys para reconciliar el árbol (slot table). Skipping: si parámetros estables no cambiaron, puede saltar ejecutar cuerpo.

Evita crear lambdas o objetos nuevos innecesarios en cada frame si pasan a hijos no estables y fuerzan recomposición amplia. Usa remember, derivedStateOf, keys en LazyColumn.

En entrevistas: diferencia recomposition vs layout vs draw phases; idempotencia del Composable; "recompose often, layout/draw less". Anti-pattern: leer estado global sin snapshot.

Cuándo usarlo:
Entender recomposición guía dónde poner estado (hoisting), remember, y optimización. No optimices prematuramente sin medir Layout Inspector.

Ejemplo:
@Composable
fun Counter() {
    var count by remember { mutableStateOf(0) }
    // Solo este Text y hermanos dependientes se recompiten al pulsar
    Button(onClick = { count++ }) { Text("Incrementar") }
    Text("Clicks: $count")
}

Consejo para entrevista:
Tres fases composition/layout/draw. Stable vs unstable. Preguntan por qué count++ dispara recomposition. remember evita reset en recomposition.
