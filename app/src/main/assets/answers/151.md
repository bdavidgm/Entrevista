## Concepto
El patrón Mapper convierte modelos entre capas (DTO ↔ Entity ↔ Domain ↔ UI) en funciones puras y testeables. Cada capa expone su propio tipo y el mapper aísla los detalles, lo que permite cambiar el modelo de una capa sin afectar al resto.

Suele implementarse como [funciones de extensión](pregunta:9) (toDomain, toUi, toEntity) o como interfaces dedicadas en proyectos grandes.

Ejemplo:
data class UserDto(val id: String, val first: String, val last: String)
data class User(val id: String, val fullName: String)

fun UserDto.toDomain() = User(id, "$first $last")

class UserRepository(private val api: UserApi) {
    suspend fun get(id: String): User = api.fetch(id).toDomain()
}


ExplicaciónKotlin:
`data class UserDto(...)` y `data class User(...)` generan automáticamente `equals`, `hashCode`, `copy` y `componentN`. `fun UserDto.toDomain() = User(id, "$first $last")` es [función de extensión](pregunta:9) sobre `UserDto` que retorna tipo de dominio; el string template concatena propiedades con espacio. `class UserRepository(private val api: UserApi)` declara dependencia privada inyectada manualmente. `suspend fun get(id: String): User = api.fetch(id).toDomain()` encadena llamada [suspend](pregunta:79) con mapper en expresión única: primero DTO, luego extensión `.toDomain()`. El tipo de retorno explícito `User` documenta la capa expuesta. Las extensiones no modifican la clase original; viven en el mismo paquete o archivo de mappers.

Consejo:
"Una capa, un modelo, un mapper". Evita exponer DTOs en el [ViewModel](pregunta:68); eso filtra detalles de la red por toda la app.
