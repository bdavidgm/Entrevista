## Concepto
StrictMode es una herramienta de desarrollo de Android que detecta en tiempo de ejecución (debug) políticas violadas: operaciones de disco o red en el hilo principal, leaks de Activity, uso incorrecto de intents, etc., mediante logs o diálogo según configuración.

## Explicación detallada
ThreadPolicy penaliza diskReads, network en main con penaltyLog o penaltyDeath en tests agresivos. VmPolicy detecta leaks con detectActivityLeaks(). Habilitar solo en Application.onCreate si BuildConfig.DEBUG.

Útil al migrar código legacy o auditar arranque. No va en release: impacto rendimiento y falsos positivos en librerías.

Complementa Lint (estático) con runtime. Combina con Profiler para ANR.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto de tu experiencia o proyecto demo, y trade-offs frente a alternativas. Mencionar versiones recientes de API o biblioteca (Material3, Paging 3, Hilt 2.x) demuestra que sigues el ecosistema, sin necesidad de memorizar números de versión exactos.

## Cuándo usarlo
Debug builds durante desarrollo y refactor. Desactivar en release. Investigar stack trace StrictMode al ver warning.

Ejemplo:
if (BuildConfig.DEBUG) {
    StrictMode.setThreadPolicy(
        StrictMode.ThreadPolicy.Builder()
            .detectDiskReads()
            .detectNetwork()
            .penaltyLog()
            .build()
    )
}


ExplicaciónKotlin:
`if (BuildConfig.DEBUG)` es condición compilada: `BuildConfig` es clase generada por Gradle con constantes del build type. Solo en debug se configura StrictMode. `StrictMode.setThreadPolicy(...)` recibe política construida con builder encadenado estilo Kotlin/Java interop. `StrictMode.ThreadPolicy.Builder()` crea builder; `.detectDiskReads()`, `.detectNetwork()`, `.penaltyLog()` retornan el mismo builder para fluency API. `.build()` materializa el objeto policy. No hay corrutinas ni delegates: es configuración imperativa en Application o Activity. El bloque demuestra uso de propiedades booleanas generadas y APIs estáticas de Android desde Kotlin sin necesidad de punto y coma.

Consejo para entrevista:
Solo debug, detecta disk/network en main. No confundas con ProGuard ni Lint.
