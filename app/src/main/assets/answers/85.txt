Concepto:
Flow y LiveData son mecanismos observables para UI, pero Flow es un stream frío de Kotlin con operadores ricos y coroutines, mientras LiveData es un contenedor lifecycle-aware de Android con API más limitada y sin cold semantics por defecto.

Explicación detallada:
LiveData: observe con LifecycleOwner, setValue/postValue, integración histórica XML. No tiene debounce nativo, combine complejo, ni backpressure explícito. Obsoleto para código nuevo según guías actuales, pero perviviente.

Flow: map, combine, flatMapLatest, catch, retry, integración Room/Retrofit suspend. Requiere collector con lifecycle (repeatOnLifecycle) o Compose collectAsStateWithLifecycle. stateIn convierte a hot StateFlow para UI.

Migración: LiveData en Repository -> Flow; ViewModel stateIn o expone Flow; Fragment collect en STARTED.

Room devuelve ambos; preferir Flow. Paging 3 usa Flow<PagingData>.

Threading: LiveData postValue; Flow flowOn y withContext en collector.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto de tu experiencia o proyecto demo, y trade-offs frente a alternativas. Mencionar versiones recientes de API o biblioteca (Material3, Paging 3, Hilt 2.x) demuestra que sigues el ecosistema, sin necesidad de memorizar números de versión exactos.

Para profundizar tras la entrevista, revisa documentación oficial, codelabs y el código fuente de tu propio proyecto Entrevista aplicando el concepto en un flujo mínimo reproducible. La práctica deliberada con tests unitarios refuerza la confianza al explicar el tema en viva voz.

Cuándo usarlo:
Flow en proyectos Kotlin nuevos; LiveData solo legacy o APIs que solo exponen LiveData.

Ejemplo:
// Antes LiveData
val user: LiveData<User> = repo.observeUser()

// Moderno
val user: StateFlow<User> = repo.observeUser()
    .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), User.Empty)

ExplicaciónKotlin:
Comentarios // Antes LiveData y // Moderno separan estilos. val user: LiveData<User> = repo.observeUser() tipado explícito LiveData Java arquitectura. val user: StateFlow<User> = repo.observeUser().stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), User.Empty) convierte Flow a StateFlow hot con valor inicial User.Empty (companion object o constante). stateIn requiere scope, SharingStarted y initialValue. Mismo repo.observeUser() supone Flow subyacente en versión moderna. WhileSubscribed(5_000) detiene upstream 5 segundos sin suscriptores. Contraste muestra migración API manteniendo tipado genérico User en LiveData vs StateFlow.

Consejo para entrevista:
Tabla mental: lifecycle (ambos con wrapper), operadores (Flow gana), testing (Turbine vs InstantTaskExecutorRule). No digas LiveData está prohibido; di preferencia Flow.
