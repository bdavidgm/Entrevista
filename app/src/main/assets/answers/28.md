## Concepto
internal es un modificador de visibilidad que limita el acceso al mismo módulo de [Gradle](pregunta:109)/IDE. A diferencia de public (todos lados) o private (mismo archivo/clase), internal permite compartir código entre archivos de un módulo sin exponerlo al exterior.

Es especialmente útil en proyectos modulares y librerías para esconder detalles de implementación, sin sacrificar la posibilidad de testearlos desde dentro.

Ejemplo:
// módulo :feature:users
internal class UserMapperImpl : UserMapper { ... }
internal fun parseUserDto(dto: Dto): User = ...

// otro módulo: no compila
val mapper = UserMapperImpl() // error

ExplicaciónKotlin:
`internal class UserMapperImpl : UserMapper` declara una clase con visibilidad `internal`: accesible dentro del mismo módulo [Gradle](pregunta:109), pero no desde otros módulos, aunque el paquete sea distinto. `: UserMapper` indica implementación de interfaz con sintaxis kotliniana. `internal fun parseUserDto(dto: Dto): User` es una función de nivel superior igualmente internal. Los comentarios documentan que `val mapper = UserMapperImpl()` en otro módulo no compila: Kotlin enforcea la frontera del módulo en compile-time. `internal` es el nivel por defecto en muchos estilos de proyecto multi-módulo para ocultar detalles de implementación sin marcar todo como `private`. Para Java interop, `internal` se expone como `public` con nombre mangled a menos que uses `@PublishedApi` o diseño consciente.

Consejo:
"internal = mismo módulo". Combinado con módulos por feature ayuda a mantener APIs limpias. Recuerda que la unidad es el módulo [Gradle](pregunta:109), no el paquete.
