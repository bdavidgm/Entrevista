## Concepto
MVP separa View (pasiva, interfaz) de Presenter, que contiene la lógica de presentación y habla con el Model. La View implementa contratos (showLoading) y reenvía eventos al Presenter; el Presenter no conoce detalles de Android salvo lo mínimo.

Fue popular antes de [ViewModel](pregunta:68); hoy [MVVM](pregunta:62)/[MVI](pregunta:65) + [ViewModel](pregunta:68) es más idiomático.

Ejemplo:
interface LoginView { fun showError(msg: String) }
class LoginPresenter(private val view: LoginView, private val repo: AuthRepository) {
    fun login(email: String) { /* repo + view.showError */ }
}

ExplicaciónKotlin:
`interface LoginView { fun showError(msg: String) }` define un contrato con un método que recibe `String` no anulable; las interfaces Kotlin no llevan cuerpo salvo default methods. `class LoginPresenter(private val view: LoginView, private val repo: AuthRepository)` usa constructor primario con propiedades declaradas en parámetros: `private val` genera campos inmutables accesibles solo dentro de la clase. `fun login(email: String)` es método público con parámetro tipado; el cuerpo comentado `/* repo + view.showError */` indica interacción típica: llamar al [repositorio](pregunta:63) y notificar la vista. No hay [corrutinas](pregunta:78) ni nullables en la firma; MVP en Kotlin suele evolucionar a [suspend](pregunta:79) functions en el repo y `viewModelScope` en lugar de Presenter manual. La separación interface/clase facilita tests con dobles de `LoginView` implementados como `object` o clases fake en Kotlin.

Consejo:
Contrasta con [MVVM](pregunta:62): Presenter no sobrevive a rotación salvo que lo gestiones tú; [ViewModel](pregunta:68) lo resuelve de serie.
