Concepto:
El CoroutineContext es la combinación de elementos (Job, CoroutineDispatcher, CoroutineName, CoroutineExceptionHandler) que define en qué condiciones se ejecuta una coroutine. El Dispatcher determina el pool de hilos: Main (UI), Default (CPU), IO (red/disco), Unconfined (sin cambiar de hilo).

Los contextos se combinan con + y se heredan al lanzar coroutines hijas. withContext cambia el dispatcher temporalmente para un bloque, ideal para operaciones bloqueantes.

Ejemplo:
viewModelScope.launch {
    val data = withContext(Dispatchers.IO) {
        repository.fetch()
    }
    _state.value = data
}

val scope = CoroutineScope(SupervisorJob() + Dispatchers.Default + CoroutineName("sync"))

ExplicaciónKotlin:
`viewModelScope.launch { ... }` usa la extension property `viewModelScope` de AndroidX y `launch` como builder de coroutine: el bloque es un `suspend` lambda que se ejecuta en el dispatcher del scope (Main por defecto en ViewModel). `withContext(Dispatchers.IO) { repository.fetch() }` cambia temporalmente de contexto a un hilo de IO para trabajo bloqueante; es una función suspend que devuelve el resultado del bloque y restaura el dispatcher anterior al terminar. `_state.value = data` asigna a una propiedad mutable (probablemente `MutableStateFlow`). `CoroutineScope(SupervisorJob() + Dispatchers.Default + CoroutineName("sync"))` compone contexto con el operador `+`: combina Job supervisor (fallos hijos no cancelan hermanos), dispatcher de CPU y nombre para debugging. `SupervisorJob()`, `Dispatchers.Default` y `CoroutineName` son elementos del `CoroutineContext`.

Consejo:
"Main para UI, IO para red/disco, Default para CPU". No abuses de withContext(Main): la coroutine ya está en Main si lanzas desde viewModelScope. Recuerda que el contexto se compone con +.
