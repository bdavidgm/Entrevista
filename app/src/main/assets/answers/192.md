## Concepto
Vibrator (o VibratorManager en API 31+) reproduce patrones de vibración con VibrationEffect: createOneShot, createWaveform o predefined constants (EFFECT_CLICK). Requiere permiso VIBRATE (normal).

En API 26+ usa siempre VibrationEffect; APIs antiguas vibrate(long) están deprecadas.

Ejemplo:
val vibrator = getSystemService(Vibrator::class.java)
if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.O) {
    vibrator.vibrate(VibrationEffect.createOneShot(40, VibrationEffect.DEFAULT_AMPLITUDE))
} else {
    @Suppress("DEPRECATION")
    vibrator.vibrate(40)
}

ExplicaciónKotlin:
`getSystemService(Vibrator::class.java)` obtiene el servicio tipado con referencia de clase Kotlin. `if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.O)` es una comprobación de API en tiempo de ejecución: Kotlin compara enteros y no aplica [smart cast](pregunta:123) aquí porque `SDK_INT` no es una propiedad estable para el compilador. En la rama moderna, `VibrationEffect.createOneShot(40, VibrationEffect.DEFAULT_AMPLITUDE)` crea el efecto y `vibrator.vibrate(...)` lo reproduce. La rama `else` lleva `@Suppress("DEPRECATION")` para silenciar el aviso del compilador al usar `vibrate(long)` obsoleto; `40` es literal `Int` promocionado a `long`. Este patrón de bifurcación por versión es habitual en Kotlin Android cuando APIs nuevas sustituyen métodos deprecados. `val vibrator` declara referencia inmutable al servicio.

Consejo:
Respeta “Desactivar vibración” y canales de [accesibilidad](pregunta:106); no abuses de patrones largos. Para haptics finos en UI, considera performHapticFeedback en vistas.
