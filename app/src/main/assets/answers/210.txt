Concepto:
RxJava modela flujos asíncronos con Observable/Single/Flowable y operadores encadenables. En Android fue popular con Retrofit adapters y debounce en búsquedas, pero hoy muchos equipos migran a Kotlin Flow + corrutinas por integración con el lenguaje y menos dependencias.

Mantenimiento legacy o librerías solo Rx siguen siendo motivos válidos.

Ejemplo:
api.user("1")
    .subscribeOn(Schedulers.io())
    .observeOn(AndroidSchedulers.mainThread())
    .subscribe({ u -> show(u) }, { e -> showError(e) })

ExplicaciónKotlin:
`api.user("1")` devuelve un `Single`/`Observable` RxJava (según firma Retrofit adapter), encadenado con operadores fluentes. `.subscribeOn(Schedulers.io())` programa la suscripción en hilo de IO; `.observeOn(AndroidSchedulers.mainThread())` redirige emisiones al main thread para UI. `.subscribe({ u -> show(u) }, { e -> showError(e) })` pasa dos lambdas Kotlin como consumidor de éxito y error: `{ u -> show(u) }` recibe el usuario tipado inferido; `{ e -> showError(e) }` recibe la excepción. RxJava espera interfaces funcionales Java; Kotlin las adapta automáticamente. No hay `Disposable` asignado a variable — en producción conviene guardarlo y disponer en `onDestroy`. El estilo contrasta con corrutinas Kotlin (`launch { api.user(...) }`), pero la sintaxis lambda es nativa de Kotlin. `"1"` es literal de login/id.

Consejo:
Si el proyecto ya es Flow-first, no mezcles Rx salvo en bordes; usa adaptadores o reescribe gradualmente. Rx exige dominar dispose/subscriptions.
