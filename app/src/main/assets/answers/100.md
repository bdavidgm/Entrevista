## Concepto
La [recomposición](pregunta:47) excesiva en Compose ocurre cuando demasiados Composables se invalidan y vuelven a ejecutar sin necesidad, desperdiciando CPU en el hilo principal y causando jank. Suele deberse a estado mal ubicado, inestabilidad de parámetros o lecturas amplias de State.

## Explicación detallada
Compose recompone scopes afectados por State leído durante composición. Si el State está alto en el árbol, un cambio pequeño (contador) recomponer pantalla entera. Soluciones: [state hoisting](pregunta:50) granular, [remember](pregunta:48), [derivedStateOf](pregunta:56) para derivar sin propagar lecturas innecesarias, estabilidad de tipos (@Stable, inmutables).

Parámetros inestables: crear lambda nueva cada vez sin [remember](pregunta:48) -> hijos skippables no saltan [recomposición](pregunta:47). Usa key en [LazyColumn](pregunta:52) para items. Evita leer state.global en raíz si solo afecta hijo.

Herramientas: Layout Inspector [recomposition](pregunta:47) counts, Compose compiler reports (strong skipping mode en versiones recientes).

## Cuándo usarlo
Cuando Profiler o Layout Inspector muestran [recomposiciones](pregunta:47) excesivas o frames por encima de 16 ms. No optimizar prematuramente sin medición previa.

Ejemplo:
// Mal: todo el árbol lee counter
@Composable
fun Screen() {
    var counter by remember { mutableIntStateOf(0) }
    ExpensiveList() // recomponer siempre
    Text("$counter")
}

// Mejor: aislar estado
@Composable
fun Screen() {
    Column {
        ExpensiveList()
        CounterChip() // solo este subárbol
    }
}


ExplicaciónKotlin:
`@Composable fun Screen()` declara funciones composables: el compilador de Compose las transforma para soportar [recomposición](pregunta:47). `var counter by remember { mutableIntStateOf(0) }` combina `remember` (estado sobrevive [recomposición](pregunta:47)), delegación `by` sobre `MutableState<Int>` especializado, y `mutableIntStateOf(0)` para primitivos sin boxing. `ExpensiveList()` invoca [composable](pregunta:46) hijo que se recompone cuando `counter` cambia porque comparten scope si está en la misma función. La versión mejorada extrae estado a `CounterChip()` [composable](pregunta:46) separado: al subir el estado al hijo, `ExpensiveList()` queda fuera del scope que lee `counter`, permitiendo skipping de [recomposición](pregunta:47). `Column { }` es [composable](pregunta:46) contenedor con trailing lambda `@Composable () -> Unit`. La estructura ilustra hoisting de estado en Compose con tipos Kotlin estándar y delegates.

Consejo para entrevista:
Menciona estabilidad, [derivedStateOf](pregunta:56), keys, y medir antes. No digas "Compose es lento"; di "mal usado".
