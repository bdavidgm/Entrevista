Concepto:
DataStore es la solución moderna de Jetpack para persistir datos clave-valor y preferencias, reemplazando SharedPreferences. Ofrece API asíncrona con coroutines y Flow, transacciones seguras y manejo de errores de corrupción superior al XML de preferences.

Explicación detallada:
Dos variantes: Preferences DataStore (como SharedPreferences con keys tipadas) y Proto DataStore (objetos tipados con protobuf). Lectura: dataStore.data es Flow<Preferences>; escritura: edit { prefs -> prefs[KEY] = value } suspend.

Ventajas sobre SharedPreferences: no hay apply/commit en main thread problemático, consistencia con Flow reactivo, y migración oficial desde SharedPreferences con SharedPreferencesMigration.

Inyección: provide DataStore<Preferences> singleton en Hilt. En ViewModel collectLatest o first() según necesidad. No guardes objetos enormes ni bases relacionales.

Encrypted DataStore o EncryptedSharedPreferences para tokens: combinar con seguridad de Keystore en datos sensibles.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Cuándo usarlo:
Flags de usuario, tema, tokens de sesión pequeños, última pantalla visitada. Room para datos relacionales; archivos para blobs grandes.

Ejemplo:
private val Context.dataStore by preferencesDataStore("settings")

object Keys {
    val DARK_MODE = booleanPreferencesKey("dark_mode")
}

suspend fun Context.setDarkMode(enabled: Boolean) {
    dataStore.edit { it[Keys.DARK_MODE] = enabled }
}

fun Context.darkModeFlow(): Flow<Boolean> =
    dataStore.data.map { it[Keys.DARK_MODE] ?: false }

Consejo para entrevista:
Di "reemplazo de SharedPreferences". Menciona Flow y corrupción. No uses DataStore como base de datos de listas de productos.
