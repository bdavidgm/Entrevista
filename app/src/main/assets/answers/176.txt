Concepto:
WorkManager ejecuta trabajo diferible respetando Doze y restricciones de batería; garantiza intentos y persistencia. AlarmManager dispara a una hora concreta o repetición con más control temporal, incluso despertando el dispositivo, pero no es cola de trabajo con reintentos automáticos.

Regla práctica: datos/sync/reporting → WorkManager; despertador/calendario/alarma exacta → AlarmManager (+ permisos recientes).

Ejemplo:
// Diferible
WorkManager.getInstance(ctx).enqueue(OneTimeWorkRequest.from(SyncWorker::class.java))

// Exacta (con permisos y buena UX)
alarmMgr.setExactAndAllowWhileIdle(AlarmManager.RTC_WAKEUP, atMillis, pi)

ExplicaciónKotlin:
El comentario `// Diferible` precede a una llamada encadenada sobre la API Java de WorkManager: `WorkManager.getInstance(ctx)` recibe un `Context` y devuelve el singleton, seguido de `.enqueue(...)`. `OneTimeWorkRequest.from(SyncWorker::class.java)` usa la referencia de clase de Kotlin (`SyncWorker::class.java`) para que el builder genere un `WorkRequest` tipado al `Worker` concreto; es la forma idiomática de pasar metadatos de clase a APIs Java. En la rama de alarma exacta, `alarmMgr` y `pi` se reutilizan como variables ya inicializadas; `atMillis` es un `Long` con el instante de disparo. Las constantes `AlarmManager.RTC_WAKEUP` son estáticas Java importadas implícitamente. El ejemplo contrasta dos estilos de programación en Kotlin sobre Android: cola de trabajo Jetpack frente a API imperativa del sistema, ambos invocados desde el mismo lenguaje sin cambiar sintaxis.

Consejo:
Menciona SCHEDULE_EXACT_ALARM y que FGS + ubicación tienen reglas aparte. No uses alarmas exactas para polling de red.
