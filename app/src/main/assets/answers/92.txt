Concepto:
Un test de ViewModel verifica la lógica de presentación de forma aislada: dado un evento o acción, el estado expuesto (StateFlow/LiveData) o efectos esperados cambian correctamente, usando repositorios fake o mocks y TestDispatcher sin UI ni emulador.

Explicación detallada:
Configura MainDispatcherRule o StandardTestDispatcher para controlar coroutines. Instancia ViewModel con FakeRepository. Llama onAction() o métodos públicos. Assert en uiState.value o collect con Turbine.

Casos: loading -> success, loading -> error, cancelación no actualiza estado tras destroy (opcional). No uses Robolectric si el ViewModel no toca Android.

SavedStateHandle: pasar SavedStateHandle(mapOf("id" to 1)) en constructor si aplica. Hilt: sin @HiltAndroidTest si inyectas manualmente en unit test.

Turbine: vm.uiState.test { awaitItem(); ... }. advanceUntilIdle() con test scheduler.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto de tu experiencia o proyecto demo, y trade-offs frente a alternativas. Mencionar versiones recientes de API o biblioteca (Material3, Paging 3, Hilt 2.x) demuestra que sigues el ecosistema, sin necesidad de memorizar números de versión exactos.

Cuándo usarlo:
Cada ViewModel con lógica no trivial. Complementa tests de use case y repository.

Ejemplo:
@OptIn(ExperimentalCoroutinesApi::class)
class CartViewModelTest {
    @get:Rule val main = MainDispatcherRule()

    @Test
    fun addItemUpdatesTotal() = runTest {
        val vm = CartViewModel(FakeCartRepo())
        vm.addItem(Item("1", 10.0))
        assertEquals(10.0, vm.uiState.value.total, 0.0)
    }
}


ExplicaciónKotlin:
`@OptIn(ExperimentalCoroutinesApi::class)` opta explícitamente en APIs experimentales del módulo de test de corrutinas (requerido en algunas versiones para reglas o APIs inestables). `@get:Rule val main = MainDispatcherRule()` expone una regla JUnit: `@get:Rule` aplica el annotation al getter de la propiedad `val`, patrón idiomático Kotlin para reglas. `MainDispatcherRule` sustituye `Dispatchers.Main` por un `TestDispatcher` en tests. `runTest { }` proporciona scope con reloj virtual. `CartViewModel(FakeCartRepo())` construye el ViewModel con un fake (implementación real simplificada) en lugar de mock. `vm.addItem(Item("1", 10.0))` invoca lógica síncrona o suspend según implementación; `Item("1", 10.0)` usa constructor de data class. `vm.uiState.value.total` lee propiedad del estado; `assertEquals(10.0, ..., 0.0)` compara `Double` con delta cero.

Consejo para entrevista:
Menciona runTest, fake repo, Turbine. No pruebes framework Android en unit test de ViewModel.
