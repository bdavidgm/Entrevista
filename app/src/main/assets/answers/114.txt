Concepto:
El patrón Observer define una relación uno-a-muchos: cuando el sujeto (observable) cambia de estado, todos los observadores registrados son notificados automáticamente.

Explicación detallada:
Componentes clásicos:
- Subject/Observable: mantiene estado y lista de observadores.
- Observer: interfaz con update() o callback equivalente.
- Implementaciones concretas en cada capa.

En Android el Observer está omnipresente:
- LiveData.observe(lifecycleOwner) { } notifica cambios de valor respetando el ciclo de vida.
- StateFlow y SharedFlow: collectors reaccionan a cada emisión.
- RxJava Observable con subscribe (legacy en muchos codebases).
- ViewModel como productor de estado; Activity, Fragment o Compose como consumidores.
- Listeners de sistema y vistas: OnClickListener, TextWatcher, addOnPropertyChangedCallback.

Ventajas: desacoplamiento entre quien produce datos y quien los muestra; varias pantallas o widgets pueden reaccionar al mismo estado (sincronización de tabs, widgets de escritorio).

Riesgos frecuentes en entrevistas:
- Memory leaks si el subject retiene la Activity o View destruida (registrar listener sin desregistrar, LiveData sin LifecycleOwner).
- Actualizar UI desde un hilo de fondo (debe ser Main).
- Cascadas de callbacks difíciles de seguir sin arquitectura clara.

Variantes modernas: MVVM + StateFlow; Compose recompone cuando cambia el state observado (collectAsState). La idea Observer sigue siendo la base del reactive UI.

Cuándo usarlo:
Siempre que la interfaz deba reaccionar a datos asíncronos o que cambian con el tiempo: red, base de datos, ubicación, otro componente, usuario en otra pantalla.

Ejemplo:
class UserRepository {
    private val _user = MutableStateFlow<User?>(null)
    val user: StateFlow<User?> = _user.asStateFlow()

    suspend fun refresh() {
        _user.value = api.fetchUser()
    }
}

@Composable
fun ProfileScreen(viewModel: ProfileViewModel = hiltViewModel()) {
    val user by viewModel.user.collectAsState()
    Text(text = user?.name ?: "Cargando...")
}


ExplicaciónKotlin:
`class UserRepository` encapsula fuente de datos. `private val _user = MutableStateFlow<User?>(null)` crea StateFlow mutable privado con valor inicial nullable `null`. `val user: StateFlow<User?> = _user.asStateFlow()` expone lectura readonly mediante función de extensión que envuelve sin copiar. `suspend fun refresh()` es función suspendida callable desde corrutina. `_user.value = api.fetchUser()` asigna directamente al `MutableStateFlow` (set síncrono thread-safe); dispara collectors. `@Composable fun ProfileScreen(viewModel: ProfileViewModel = hiltViewModel())` usa parámetro default con composable de Hilt Navigation. `val user by viewModel.user.collectAsState()` delega estado Compose: `by` sobre `State<T>` devuelto por `collectAsState()`. `user?.name ?: "Cargando..."` encadena safe call y Elvis para placeholder.

Consejo para entrevista:
Empieza con la definición genérica Subject/Observer y luego mapea a LiveData, Flow y Compose. Menciona observación lifecycle-aware y cancelación de suscripciones para evitar fugas.
