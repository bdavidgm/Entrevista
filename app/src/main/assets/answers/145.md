## Concepto
flowOn cambia el [CoroutineContext](pregunta:30) en el que se ejecuta el upstream de un [Flow](pregunta:84) (lo que está antes del operador), sin afectar al downstream salvo que vuelvas a usar flowOn. Sirve para mover trabajo pesado a [Dispatchers.Default](pregunta:30) o IO y dejar la recolección en Main.

El orden importa: solo lo que está encima de flowOn se mueve de dispatcher.

Ejemplo:
fun users(): Flow<List<User>> = flow {
    emit(database.loadUsers()) // IO
}.flowOn(Dispatchers.IO)


ExplicaciónKotlin:
`fun users(): Flow<List<User>> = flow { emit(database.loadUsers()) }` declara función que retorna `Flow` mediante builder `flow { }`: el lambda es [suspend](pregunta:79) y define el upstream. `emit(...)` envía valor downstream; debe llamarse en contexto de collector activo. `.flowOn(Dispatchers.IO)` es operador de extensión que cambia contexto de ejecución del segmento upstream (incluido el bloque `flow` y `emit`), no del collector. `Dispatchers.IO` es singleton del pool para I/O. El tipo de retorno explícito `Flow<List<User>>` documenta emisiones de listas inmutables. La cadena es expresión de una sola asignación con `=`. El orden de operadores importa: solo lo anterior a `flowOn` corre en IO.

Consejo:
No confundas con subscribeOn de [RxJava](pregunta:210): en [Flow](pregunta:84), cada operador respeta el contexto del segmento. Para el collector, usa [withContext](pregunta:82) o collect en el scope adecuado.
