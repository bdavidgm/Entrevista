## Concepto
MainDispatcherRule (JUnit4) o MainDispatcherExtension (JUnit5) sustituye [Dispatchers.Main](pregunta:87) en tests por un dispatcher controlado (StandardTestDispatcher o UnconfinedTestDispatcher). Así las [corrutinas](pregunta:78) que usan Main se ejecutan de forma determinista sin el hilo real de Android.

Imprescindible para testear ViewModels que usan [viewModelScope](pregunta:131) o [withContext](pregunta:82)(Main).

Ejemplo:
@get:Rule
val mainDispatcherRule = MainDispatcherRule()

@Test
fun `carga estado`() = runTest {
    val vm = HomeViewModel(repo)
    vm.load()
    advanceUntilIdle()
    assertIs<HomeState.Success>(vm.state.value)
}


ExplicaciónKotlin:
`@get:Rule val mainDispatcherRule = MainDispatcherRule()` usa anotación de uso `@get:Rule` sobre propiedad para JUnit4: expone getter anotado. `MainDispatcherRule` sustituye `Dispatchers.Main` en tests. La función de test llamada carga estado (nombre entre backticks en Kotlin) ejecuta en entorno de [corrutinas](pregunta:78) controlado con `runTest`. `HomeViewModel(repo)` instancia SUT con fake o mock de repo. `vm.load()` lanza trabajo en `viewModelScope` que ahora usa test dispatcher. `advanceUntilIdle()` avanza virtualmente el scheduler hasta vaciar cola — API de [coroutines](pregunta:78)-test. `assertIs<HomeState.Success>(vm.state.value)` es aserción de kotlinx-[coroutines](pregunta:78)-test o similar que verifica tipo en runtime y smart-castea; `vm.state.value` lee último valor del `StateFlow`.

Consejo:
Con StandardTestDispatcher avanza el tiempo con advanceUntilIdle; con UnconfinedTestDispatcher todo corre de inmediato (menos control, más simple).
