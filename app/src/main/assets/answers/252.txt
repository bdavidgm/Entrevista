Concepto:
Una clase nested declarada dentro de otra no conserva referencia a la instancia exterior; equivale a una clase static anidada de Java. Una clase inner sí mantiene esa referencia y puede acceder a los miembros del objeto exterior.

Ejemplo:
class Screen(private val title: String) {
    class Arguments(val id: String)

    inner class Renderer {
        fun render() = println(title)
    }
}

val args = Screen.Arguments("42")
val renderer = Screen("Perfil").Renderer()

ExplicaciónKotlin:
`class Screen(private val title: String)` guarda título en propiedad privada del exterior. `class Arguments(val id: String)` es nested class: no lleva `inner`, equivalente a static nested en Java; no ve instancia de `Screen`. `inner class Renderer { fun render() = println(title) }` mantiene referencia implícita al `Screen` exterior y puede leer `title` privado gracias a visibilidad de outer class. `Screen.Arguments("42")` instancia nested sin crear `Screen`. `Screen("Perfil").Renderer()` requiere instancia exterior antes de crear `Renderer`; sintaxis `Outer().Inner()` típica de inner classes. Inner classes llevan referencia al outer y pueden causar memory leaks si el inner sobrevive al outer (Handlers, callbacks). Nested `Arguments` es ideal para `Bundle` keys o nav args sin capturar Activity. Modificador `inner` es palabra clave Kotlin específica frente a Java.

Consejo:
Prefiere nested. Usa inner solo si necesita realmente el objeto exterior, porque esa referencia también puede prolongar su vida y causar leaks.
