## Concepto
SOLID reúne cinco principios para diseñar objetos mantenibles:
- S: una clase tiene una razón principal para cambiar.
- O: se amplía mediante nuevos tipos sin modificar lógica estable.
- L: un subtipo puede sustituir al tipo base sin romper su contrato.
- I: interfaces pequeñas y específicas.
- D: las capas de alto nivel dependen de abstracciones, no de implementaciones.

Ejemplo:
interface UserRepository {
    suspend fun user(id: String): User
}

class GetUser(private val repository: UserRepository) {
    suspend operator fun invoke(id: String) = repository.user(id)
}

class RoomUserRepository(private val dao: UserDao) : UserRepository {
    override suspend fun user(id: String) = dao.user(id).toDomain()
}

ExplicaciónKotlin:
`interface UserRepository { suspend fun user(id: String): User }` aplica Interface Segregation y Dependency Inversion: capas altas dependen de abstracción, no de [Room](pregunta:70) concreto. `class GetUser(private val repository: UserRepository)` recibe colaborador por constructor (`private val`), facilitando tests con fake [repository](pregunta:63) (Single Responsibility del [caso de uso](pregunta:66)). `suspend operator fun invoke(id: String) = repository.user(id)` hace la clase invocable como función (`getUser("1")`), patrón idiomático Kotlin para [use case](pregunta:66)s. `RoomUserRepository(private val dao: UserDao) : UserRepository` es implementación intercambiable (Open/Closed: extiendes con nuevo repo sin tocar `GetUser`). `dao.user(id).toDomain()` encadena acceso [Room](pregunta:70) y mapper extensión o función `toDomain()`. `override suspend` mantiene contrato asíncrono. SOLID aquí no añade capas gratuitas: separa obtención de datos de orquestación. Inyección con [Hilt](pregunta:74) usaría `@Inject constructor` en implementaciones.

Consejo:
SOLID son criterios, no reglas para crear más capas. Aplícalos cuando reduzcan acoplamiento, faciliten tests o separen responsabilidades reales.
