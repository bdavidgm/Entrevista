Concepto:
WorkManager es la API recomendada de Jetpack para ejecutar trabajo en segundo plano garantizado, flexible y consciente de restricciones del sistema (red, batería, almacenamiento). Sustituye patrones frágiles como IntentService o AlarmManager para tareas diferibles y persistentes.

Explicación detallada:
Defines Worker o CoroutineWorker que implementa doWork() y devuelve Result.success(), failure() o retry(). Encolas con OneTimeWorkRequest o PeriodicWorkRequest (mínimo 15 min en período). Constraints: setRequiredNetworkType(UNMETERED), requiresCharging(), etc.

WorkManager persiste la cola: si el proceso muere, el trabajo se reanuda. Encadena con WorkContinuation (A luego B). Unique work evita duplicados (KEEP, REPLACE). Observa estado con getWorkInfoByIdLiveData o Flow.

CoroutineWorker usa suspend doWork() en dispatcher apropiado. Hilt integra HiltWorkerFactory para inyectar dependencias en Workers.

Diferencia con Foreground Service: trabajo inmediato visible al usuario (música, GPS activo) vs trabajo deferible (sync, upload, limpieza). Diferencia con coroutine en ViewModel: ViewModel se cancela al salir; WorkManager sobrevive.

Restricciones Android 12+: Doze y battery saver retrasan jobs; WorkManager respeta políticas mejor que AlarmManager para la mayoría de casos no exactos.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto de tu experiencia o proyecto demo, y trade-offs frente a alternativas. Mencionar versiones recientes de API o biblioteca (Material3, Paging 3, Hilt 2.x) demuestra que sigues el ecosistema, sin necesidad de memorizar números de versión exactos.

Para profundizar tras la entrevista, revisa documentación oficial, codelabs y el código fuente de tu propio proyecto Entrevista aplicando el concepto en un flujo mínimo reproducible. La práctica deliberada con tests unitarios refuerza la confianza al explicar el tema en viva voz.

Cuándo usarlo:
Sincronización periódica, subida de logs, backup, indexación. No para UI en tiempo real ni latencia sub-segundo (usa FGS o alarm exacto con justificación). No reemplaza push de servidor para notificaciones instantáneas.

Ejemplo:
class SyncWorker @AssistedInject constructor(
    @Assisted context: Context,
    @Assisted params: WorkerParameters,
    private val repo: SyncRepository
) : CoroutineWorker(context, params) {
    override suspend fun doWork(): Result = runCatching {
        repo.syncAll()
        Result.success()
    }.getOrElse { Result.retry() }
}

// En Application o ViewModel
val request = OneTimeWorkRequestBuilder<SyncWorker>()
    .setConstraints(Constraints.Builder().setRequiredNetworkType(NetworkType.CONNECTED).build())
    .build()
WorkManager.getInstance(context).enqueue(request)

Consejo para entrevista:
Contrasta con AlarmManager, JobScheduler legacy y Foreground Service. Menciona Result.retry() y constraints. Pregunta: ¿Garantiza ejecución exacta a las 8:00? No para periódico estricto; AlarmManager o exact alarms con permiso.
