Concepto:
Un DAO (Data Access Object) en Room es una interfaz que declara métodos de acceso a datos SQL para una o más entidades: consultas, inserciones, actualizaciones y borrados. Room genera la implementación en tiempo de compilación a partir de anotaciones.

Explicación detallada:
@Dao en interface ProductDao. @Query("SELECT * FROM products WHERE category = :cat") suspend fun byCategory(cat: String): List<ProductEntity>. Parámetros enlazados evitan SQL injection. @Insert puede devolver Long o List<Long> de row ids; onConflict define REPLACE, IGNORE, ABORT.

Updates y deletes por entidad o por query @Query("DELETE FROM products WHERE id = :id"). @Transaction agrupa operaciones atómicas cuando un método llama a varios DAO o múltiples escrituras.

Flow y LiveData en @Query hacen que Room re-emita al cambiar la tabla observada. PagingSource en @Query integra RecyclerView o LazyColumn con Paging 3.

Raw queries: @RawQuery para SQL dinámico con SupportSQLiteQuery. Views: @DatabaseView para consultas precalculadas.

Buenas prácticas: un DAO por agregado o tabla según tamaño del proyecto; no mezclar lógica de negocio en DAO (solo acceso datos). Mappers convierten Entity a dominio fuera del DAO.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto de tu experiencia o proyecto demo, y trade-offs frente a alternativas. Mencionar versiones recientes de API o biblioteca (Material3, Paging 3, Hilt 2.x) demuestra que sigues el ecosistema, sin necesidad de memorizar números de versión exactos.

Para profundizar tras la entrevista, revisa documentación oficial, codelabs y el código fuente de tu propio proyecto Entrevista aplicando el concepto en un flujo mínimo reproducible. La práctica deliberada con tests unitarios refuerza la confianza al explicar el tema en viva voz.

Cuándo usarlo:
Siempre con Room; es la única forma de acceder a tablas definidas. La lógica "si vacío fetch red" va en Repository, no en DAO.

Ejemplo:
@Dao
interface UserDao {
    @Query("SELECT * FROM users WHERE id = :userId")
    suspend fun getById(userId: Long): UserEntity?

    @Query("SELECT * FROM users ORDER BY name ASC")
    fun observeAll(): Flow<List<UserEntity>>

    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun upsert(user: UserEntity)

    @Query("DELETE FROM users WHERE id = :userId")
    suspend fun deleteById(userId: Long)
}

Consejo para entrevista:
Explica anotaciones principales y @Transaction. No confundas DAO con Repository. Seguimiento: ¿Puedo tener JOIN? Sí en @Query con data class de relación o @Relation.
