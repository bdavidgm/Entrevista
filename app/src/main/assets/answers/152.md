## Concepto
Singleton garantiza una única instancia de una clase en todo el proceso. En Kotlin suele implementarse con object; en Android, dependencias globales se registran en Hilt/Koin con scope @Singleton o equivalente.

Evita singletons con estado mutable compartido sin disciplina: complica tests y concurrencia.

Ejemplo:
object Analytics {
    fun track(event: String) { /* ... */ }
}

@Singleton
class AppGraph @Inject constructor() { /* ... */ }


ExplicaciónKotlin:
`object Analytics { ... }` declara singleton de Kotlin: una única instancia lazy thread-safe sin constructor público. `fun track(event: String)` es método de instancia del object invocable como `Analytics.track(...)`. `@Singleton class AppGraph @Inject constructor() { ... }` combina scope de Dagger/Hilt con constructor inyectable vacío — el cuerpo `{ /* ... */ }` puede contener dependencias grafadas. `object` y `@Singleton` resuelven instancia única pero por mecanismos distintos: el primero es idioma Kotlin, el segundo es contenedor DI. Preferir `@Inject constructor` mejora testabilidad frente a `object` con estado mutable. Ambos patrones conviven en codebases Android modernas.

Consejo:
Prefiere DI con scope de aplicación antes de object con estado. Los singletons ocultan dependencias: el constructor inyectado es más claro.
