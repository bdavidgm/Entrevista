## Concepto
Application es el objeto Singleton que el sistema crea al lanzar el proceso de la app y que sobrevive a todas sus Activities. Es el lugar habitual para inicializar dependencias de proceso ([Hilt](pregunta:74), [Timber](pregunta:265), [WorkManager](pregunta:72), crash reporting) y para mantener estado global.

Se declara en el [AndroidManifest](pregunta:35) con android:name=".MyApp" y se inicializa en onCreate. Con [Hilt](pregunta:74) se anota con @HiltAndroidApp.

Ejemplo:
@HiltAndroidApp
class MyApp : Application() {
    override fun onCreate() {
        super.onCreate()
        Timber.plant(Timber.DebugTree())
        Stetho.initializeWithDefaults(this)
    }
}


ExplicaciónKotlin:
`@HiltAndroidApp class MyApp : Application()` — anotación [Hilt](pregunta:74) genera componente Application y clases DI base; debe existir una sola en app. `: Application()` hereda clase Android del proceso. `override fun onCreate()` callback de arranque del proceso, antes de cualquier [Activity](pregunta:31). `super.onCreate()` delegación obligatoria al framework. `Timber.plant(Timber.DebugTree())` configura logging; `Stetho.initializeWithDefaults(this)` pasa `this` (Application [Context](pregunta:127)) a Facebook Stetho. `this` en Application es Application [Context](pregunta:127) válido globalmente. Inicializaciones aquí bloquean cold start; deben ser ligeras o diferidas.

Consejo:
"onCreate del Application es global y bloqueante en el arranque": evita trabajo pesado, considera [App Startup](pregunta:77) library para inicialización ordenada y diferida.
