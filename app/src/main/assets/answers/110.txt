Concepto:
buildTypes definen variantes de compilación (debug, release) con flags distintos (minify, debuggable, applicationIdSuffix). productFlavors definen dimensiones de producto (free/paid, dev/prod) que se combinan con buildTypes formando variant matrix (freeDebug, prodRelease, etc.).

Explicación detallada:
buildTypes { release { isMinifyEnabled = true; signingConfig } debug { applicationIdSuffix = ".debug" } }. flavors en flavorDimensions: dimension "tier" { create("free") { applicationIdSuffix = ".free" }; create("paid") }.

manifestPlaceholders y resValue por flavor para API keys distintas. matchingFallbacks si flavors no alinean en todos módulos.

Variant filter en CI si no quieres compilar todas. sourceSets: src/free/java extras. BuildConfig generado por flavor (BuildConfig.API_URL).

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto de tu experiencia o proyecto demo, y trade-offs frente a alternativas. Mencionar versiones recientes de API o biblioteca (Material3, Paging 3, Hilt 2.x) demuestra que sigues el ecosistema, sin necesidad de memorizar números de versión exactos.

Para profundizar tras la entrevista, revisa documentación oficial, codelabs y el código fuente de tu propio proyecto Entrevista aplicando el concepto en un flujo mínimo reproducible. La práctica deliberada con tests unitarios refuerza la confianza al explicar el tema en viva voz.

Cuándo usarlo:
Múltiples marcas white-label, entornos staging/prod, versiones free/pro. buildTypes casi siempre debug/release mínimo.

Ejemplo:
android {
    flavorDimensions += "env"
    productFlavors {
        create("dev") {
            dimension = "env"
            buildConfigField("String", "BASE_URL", "\"https://dev.api.example.com\"")
        }
        create("prod") {
            dimension = "env"
            buildConfigField("String", "BASE_URL", "\"https://api.example.com\"")
        }
    }
    buildTypes {
        getByName("release") { isMinifyEnabled = true }
    }
}


ExplicaciónKotlin:
`flavorDimensions += "env"` añade dimensión de product flavors con operador `+=` sobre colección mutable del DSL. `productFlavors { create("dev") { ... } }` usa función `create` con lambda de configuración; `dimension = "env"` asigna dimensión. `buildConfigField("String", "BASE_URL", ""https://dev.api.example.com"")` genera campo estático en `BuildConfig` Java/Kotlin interop: el valor es string escapado para código generado. Flavor `prod` repite patrón con URL distinta. `buildTypes { getByName("release") { isMinifyEnabled = true } }` obtiene build type existente por nombre y muta propiedad. En runtime Kotlin se accede `BuildConfig.BASE_URL` como `String` según flavor compilado.

Consejo para entrevista:
buildType = cómo compila; flavor = qué producto. Variants = combinación. Menciona BuildConfig y signing por variant.
