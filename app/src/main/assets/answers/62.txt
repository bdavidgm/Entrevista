Concepto:
MVVM (Model-View-ViewModel) separa la app en tres capas: Model (datos y lógica de negocio), View (UI pasiva que observa estado y emite eventos) y ViewModel (mantiene el estado de la pantalla y expone APIs para que la View actúe). En Android, ViewModel sobrevive a cambios de configuración y suele exponer StateFlow/LiveData.

Beneficios: testabilidad (ViewModel puro y sin Android), separación clara de responsabilidades y un flujo unidireccional natural (state down, events up).

Ejemplo:
class HomeViewModel(repo: Repository) : ViewModel() {
    val state: StateFlow<HomeState> = repo.users()
        .map(HomeState::Success)
        .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), HomeState.Loading)
}

@Composable
fun HomeScreen(vm: HomeViewModel = hiltViewModel()) {
    val state by vm.state.collectAsStateWithLifecycle()
    HomeContent(state)
}

Consejo:
MVVM en Android suele combinarse con Repository, UDF y, opcionalmente, Use Cases. Comparte conceptos con MVI: la diferencia clave es cómo se modela el "intent".
