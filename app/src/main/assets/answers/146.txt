Concepto:
Un test de integración verifica que varios componentes reales o casi reales trabajan juntos correctamente (varias capas, BD, red simulada, DI), a diferencia del test unitario que aísla una unidad con mocks.

Explicación detallada:
Pirámide de testing en Android:
- Unit tests (JVM): ViewModel, UseCase, mappers — rápidos, MockK/fakes.
- Integration tests: Repository + Room en memoria, DAO + migraciones, módulos Hilt de test.
- Instrumented/UI tests (androidTest): Espresso, Compose Test, dispositivo/emulador.

Ejemplos de integración:
- Room con base in-memory y verificar que Repository persiste y lee.
- Retrofit con MockWebServer + Repository real.
- WorkManager TestListenableWorkerBuilder.

Características:
- Más lentos y frágiles que unitarios.
- Menos mocks; más fidelidad al comportamiento de producción.
- Requieren androidTest si usan Context real o Room instrumentado (Room también permite JVM con Robolectric en algunos casos).

Cuándo usarlo:
Validar contratos entre capas, migraciones SQL, serialización JSON real, configuración de módulos DI de integración.

Ejemplo:
@RunWith(AndroidJUnit4::class)
class UserDaoIntegrationTest {
    private lateinit var db: AppDatabase
    private lateinit var dao: UserDao

    @Before
    fun setup() {
        val context = ApplicationProvider.getApplicationContext<Context>()
        db = Room.inMemoryDatabaseBuilder(context, AppDatabase::class.java)
            .allowMainThreadQueries()
            .build()
        dao = db.userDao()
    }

    @Test
    fun insertAndRead() = runBlocking {
        dao.insert(User("1", "Ana"))
        val users = dao.getAll()
        assertEquals(1, users.size)
        assertEquals("Ana", users[0].name)
    }
}

Consejo para entrevista:
Define integración vs unitario vs UI. Menciona Room in-memory, MockWebServer y que no todo debe ser integración por velocidad CI.
