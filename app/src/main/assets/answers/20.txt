Concepto:
El modificador inline en Kotlin indica al compilador que sustituya el cuerpo de la función en los sitios de llamada en lugar de generar una invocación normal, cuando es posible. Las funciones inline de orden superior (que reciben lambdas) evitan la allocación de objetos Function en JVM y reducen coste en bucles y APIs de colecciones.

Explicación detallada:
Sin inline, pasar un lambda en Kotlin/JVM crea una instancia anónima (salvo optimizaciones del compilador Java). inline copia el bytecode del lambda en el call site, similar a macros C con type safety. Por eso forEach, map y las scope functions estándar son inline.

Restricciones: funciones inline no pueden llamarse desde Java de forma natural en algunos casos; no puedes almacenar fácilmente referencias a funciones inline privadas; el cuerpo no debería ser enorme o inflar el DEX. crossinline y noinline marcan parámetros lambda: crossinline para lambdas que no pueden hacer return no local; noinline para lambdas que se pasan a otras funciones no inline.

En Android, inline es crítico en capas de utilidades de alto tráfico y en DSLs (Compose compiler tiene su propio modelo, pero Kotlin inline sigue en stdlib). reified type parameters solo funcionan en funciones inline porque el tipo T se conserva en runtime en el call site.

Cuándo usarlo:
Funciones que aceptan lambdas y se llaman frecuentemente; APIs de biblioteca (stdlib). No marques inline funciones grandes o que solo llaman una vez: aumenta tamaño de APK sin beneficio. Para funciones públicas de módulo, documenta si esperas uso desde Java.

Ejemplo:
inline fun <T> measureMillis(block: () -> T): Pair<T, Long> {
    val start = System.currentTimeMillis()
    val result = block()
    return result to (System.currentTimeMillis() - start)
}

inline fun View.onClick(crossinline action: (View) -> Unit) {
    setOnClickListener { action(it) }
}

// Uso
val (data, ms) = measureMillis { repository.fetchHeavy() }
Log.d(TAG, "fetch en ${ms}ms")
