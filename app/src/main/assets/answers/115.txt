Concepto:
Inmutabilidad significa que un objeto o valor no puede modificarse después de crearse; cualquier cambio produce una nueva instancia en lugar de mutar la existente.

Explicación detallada:
En Kotlin: val fija la referencia; var permite reasignar. Las colecciones List son de solo lectura a nivel de API; copy() en data class crea variantes. Inmutabilidad profunda requiere que los campos internos tampoco sean mutables compartidos.

Por qué importa en Android:
- Concurrencia: con coroutines y varios hilos, objetos inmutables evitan condiciones de carrera sin synchronized.
- UI predecible: RecyclerView DiffUtil, Compose y StateFlow asumen que un cambio de referencia o de estado dispara actualización; mutar una lista in-place sin notificar deja la UI inconsistente.
- Depuración: cada snapshot de UiState es reproducible en tests.
- Arquitectura: MVI, Redux-like y UiState data class fomentan transiciones de estado explícitas.

Trade-offs: más objetos temporales y presión en GC; en listas muy grandes combinar inmutabilidad con estructuras eficientes (Paging, actualizaciones parciales, keys estables en LazyColumn).

No confundir val con inmutabilidad total: val list: List<Int> puede envolver un MutableList si alguien expuso la referencia mutable.

Cuándo usarlo:
Estado de pantalla expuesto por ViewModel, modelos de dominio entre capas, eventos en colas concurrentes, argumentos de navegación. La mutabilidad queda encapsulada en implementaciones internas (buffer, builder antes de publicar).

Ejemplo:
data class CartUiState(
    val items: List<CartItem> = emptyList(),
    val total: Money = Money.ZERO,
    val isLoading: Boolean = false
)

fun CartViewModel.addItem(item: CartItem) {
    _uiState.update { state ->
        state.copy(items = state.items + item)
    }
}

// En Compose
@Composable
fun CartScreen(viewModel: CartViewModel) {
    val state by viewModel.uiState.collectAsState()
    LazyColumn {
        items(state.items, key = { it.id }) { item ->
            CartRow(item)
        }
    }
}

Consejo para entrevista:
Relaciona inmutabilidad con thread-safety, Compose (estabilidad de parámetros) y testing. Aclara que variables locales mutables en un algoritmo están bien; el estado compartido de la app debe ser inmutable o muy bien encapsulado.
