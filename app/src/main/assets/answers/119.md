## Concepto
Un deep link (enlace profundo) es un URI o App Link que abre directamente una pantalla o flujo específico dentro de la app, en lugar de solo lanzar la [Activity](pregunta:31) principal.

## Explicación detallada
Tipos:
- Deep link clásico: [intent](pregunta:33)-filter con scheme custom (myapp://product/42). Puede mostrar selector de apps si varias coinciden.
- App Links (Android 6+): https con autoVerify="true" y Digital Asset Links en el dominio. Abren tu app sin diálogo si la verificación es correcta.
- Dynamic Links (Firebase, legacy) y enlaces de marketing que redirigen al destino in-app.

Configuración en [AndroidManifest](pregunta:35):
- [intent](pregunta:33)-filter con action VIEW, categorías DEFAULT y BROWSABLE.
- data: scheme, host, pathPrefix o pathPattern.

[Navigation Component](pregunta:75):
- <deepLink app:uri="https://example.com/item/{id}" /> en el destino.
- navController.handleDeepLink([intent](pregunta:33)) en onCreate/onNewIntent.

Flujo típico: usuario toca enlace en email o web -> sistema resuelve [Intent](pregunta:33) -> [Activity](pregunta:31) host del [NavHost](pregunta:60) -> grafo navega al destino con argumentos.

Consideraciones:
- Estado cold start: la app puede no tener sesión; redirigir a login y guardar destino pendiente.
- onNewIntent si la [Activity](pregunta:31) existe con [launchMode](pregunta:199) [singleTop](pregunta:199).
- Pruebas con [adb](pregunta:226) shell am start -a android.[intent](pregunta:33).action.VIEW -d "https://..."

## Cuándo usarlo
Compartir contenido, campañas, recuperación de carrito, notificaciones que abren pantalla concreta, integración web-to-app.

Ejemplo:
// AndroidManifest.xml dentro de activity
<intent-filter android:autoVerify="true">
    <action android:name="android.intent.action.VIEW" />
    <category android:name="android.intent.category.DEFAULT" />
    <category android:name="android.intent.category.BROWSABLE" />
    <data android:scheme="https"
          android:host="example.com"
          android:pathPrefix="/product" />
</intent-filter>

class MainActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContent { App(navController) }
        navController.handleDeepLink(intent)
    }
}


ExplicaciónKotlin:
El XML del manifest no es Kotlin; define [intent](pregunta:33)-filter para App Links. `class MainActivity : AppCompatActivity()` en Kotlin hereda [Activity](pregunta:31). `override fun onCreate(savedInstanceState: Bundle?)` — `Bundle?` nullable. `setContent { App(navController) }` usa Compose [Activity](pregunta:31) API: lambda `@Composable` configura UI; `navController` parámetro de grafo. `navController.handleDeepLink(intent)` procesa URI del [Intent](pregunta:33) entrante (cold start o `onNewIntent`). `intent` es propiedad de [Activity](pregunta:31). El flujo combina interoperabilidad XML manifest + Kotlin [Activity](pregunta:31) + [Navigation Compose](pregunta:60); `handleDeepLink` es extensión o método del NavController que parsea URI y navega al destino con argumentos.

Consejo para entrevista:
Diferencia deep link custom vs App Links verificados. Menciona handleDeepLink, cold start y prueba con [adb](pregunta:226). Si hablas de notificaciones, enlaza con [PendingIntent](pregunta:164) que lleva el URI.
