Concepto:
State hoisting (elevación de estado) es el patrón de mover el estado y la lógica de mutación hacia un ancestro común, dejando composables hijos sin estado (stateless) que reciben el valor y un callback para cambios. Mejora reutilización, testabilidad y fuente única de verdad.

Explicación detallada:
Un composable stateless recibe value: String y onValueChange: (String) -> Unit. El padre hace remember { mutableStateOf("") } o lee del ViewModel. Facilita previews y tests sin emulador completo.

Variantes: stateness parcial (hoist solo lo compartido). En formularios, el ViewModel puede ser el hoist ultimate para sobrevivir rotación. Regla: hoist al nivel más bajo que necesita leer/escribir ese estado compartido.

En entrevistas piden refactorizar TextField con estado interno a hoisted. Relacionan con unidirectional data flow (UDF): Event up, State down.

Cuándo usarlo:
Componentes de diseño reutilizables (botones, campos, cards). Pantallas donde varios hijos leen el mismo estado. Siempre que un mismo estado controle múltiples partes del layout.

Ejemplo:
@Composable
fun SearchScreen(viewModel: SearchViewModel = hiltViewModel()) {
    val query by viewModel.query.collectAsStateWithLifecycle()
    SearchField(
        value = query,
        onQueryChange = viewModel::onQueryChange
    )
}

@Composable
fun SearchField(value: String, onQueryChange: (String) -> Unit) {
    TextField(
        value = value,
        onValueChange = onQueryChange,
        label = { Text("Buscar") }
    )
}

Consejo para entrevista:
Dibuja árbol padre/hijo. Diferencia stateful wrapper vs stateless core. ViewModel como punto de hoist para datos de negocio.
