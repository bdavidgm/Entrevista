Concepto:
State hoisting es el patrón de "subir" el estado a un ancestro común, dejando los hijos sin estado propio (stateless) que reciben el valor y un callback para cambiarlo. Mejora la testabilidad, la reutilización y mantiene una única fuente de verdad.

Forma canónica de un componente hoisted: value: T y onValueChange: (T) -> Unit. El padre puede tener el estado en remember o, mejor, en un ViewModel.

Ejemplo:
@Composable
fun SearchScreen(viewModel: SearchViewModel = hiltViewModel()) {
    val query by viewModel.query.collectAsStateWithLifecycle()
    SearchField(value = query, onChange = viewModel::onQueryChange)
}

@Composable
fun SearchField(value: String, onChange: (String) -> Unit) {
    TextField(value = value, onValueChange = onChange)
}

ExplicaciónKotlin:
SearchScreen muestra state hoisting hacia un ViewModel. El parámetro viewModel: SearchViewModel = hiltViewModel() usa valor por defecto y hiltViewModel() como función composable de Hilt que resuelve el ViewModel del NavBackStackEntry o componente actual. val query by viewModel.query.collectAsStateWithLifecycle() combina delegación by con collectAsStateWithLifecycle, que convierte el Flow query en State observable respetando el Lifecycle. SearchField(value = query, onChange = viewModel::onQueryChange) pasa el estado hacia abajo y usa method reference viewModel::onQueryChange para el callback, equivalente a { viewModel.onQueryChange(it) } si la firma lo permite. SearchField es stateless: recibe value: String y onChange: (String) -> Unit, tipando explícitamente la función lambda que recibe String y devuelve Unit. TextField reenvía onValueChange = onChange, conectando la API de Material con el callback del padre sin lógica intermedia.

Consejo:
"State down, events up": flujo unidireccional. Hoist al nivel más bajo que necesite compartirlo. El ViewModel suele ser el hoist final.
