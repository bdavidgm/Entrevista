## Concepto
Mockito (Java) y MockK (Kotlin) son librerías de mocking que crean objetos dobles para aislar la unidad bajo test: simulan dependencias (repositorios, APIs) y verifican interacciones (verify) sin implementaciones reales.

## Explicación detallada
MockK: every { repo.get() } returns listOf(); coEvery para [suspend](pregunta:79); verify { repo.get() } wasCalled 1 times. relaxed = true mocks devuelven defaults. mockkStatic para extensiones.

Mockito: when().thenReturn(); menos idiomático en Kotlin por nullability. spy() envuelve objeto real.

Preferir fakes (FakeRepository con mapa en memoria) cuando la interfaz es pequeña: tests más legibles y menos frágiles a refactor. Mocks para interacciones puntuales o SDKs externos.

@Mock [lateinit](pregunta:16) var con MockitoJUnit rule; MockK usa mockk() inline.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, [ANR](pregunta:95), Vitals) cuando aplique. ## Cuándo usarlo
Unit tests de [ViewModel](pregunta:68)/[use case](pregunta:66) con dependencias inyectadas. Evita mock de clases finales sin configuración o [data classes](pregunta:3) complejas si fake es simple.

Ejemplo:
@Test
fun `loads products`() = runTest {
    val repo = mockk<ProductRepository>()
    coEvery { repo.getProducts() } returns listOf(Product("1", "A"))
    val vm = ProductListViewModel(repo)
    vm.load()
    assertEquals(1, vm.uiState.value.items.size)
}


ExplicaciónKotlin:
`@Test` y `fun \`loads products\`() = runTest { }` combinan [JUnit](pregunta:89) con el scope de prueba de [corrutinas](pregunta:78): el cuerpo puede llamar funciones `suspend` y controlar el scheduler. `mockk<ProductRepository>()` crea un mock con la librería MockK, idiomática en Kotlin por soporte de funciones de orden superior y objetos finales. `coEvery { repo.getProducts() } returns listOf(...)` stubea la [función suspend](pregunta:79): `coEvery` es el equivalente de `every` para [corrutinas](pregunta:78). `listOf(Product("1", "A"))` construye una `List` inmutable con un `Product`, presumiblemente `data class` con constructor posicional. `ProductListViewModel(repo)` instancia el [ViewModel](pregunta:68) inyectando el mock manualmente (sin [Hilt](pregunta:74) en unit test). `vm.load()` lanza la lógica bajo prueba; `vm.uiState.value.items.size` accede al valor actual del `StateFlow`/`MutableStateFlow` sin colectar, y `assertEquals(1, ...)` verifica el tamaño de la lista.
