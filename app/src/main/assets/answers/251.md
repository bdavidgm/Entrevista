## Concepto
Herencia expresa una relación «es un» y acopla la subclase al comportamiento del padre. Composición expresa «tiene un»: una clase recibe colaboradores y delega en ellos. Suele ser más flexible porque puedes sustituir piezas sin alterar una jerarquía.

Ejemplo:
interface Engine {
    fun start()
}

class ElectricEngine : Engine {
    override fun start() = println("Motor eléctrico")
}

class Car(private val engine: Engine) {
    fun drive() {
        engine.start()
        println("En marcha")
    }
}

ExplicaciónKotlin:
`interface Engine { fun start() }` define capacidad mínima sin estado. `class ElectricEngine : Engine { override fun start() = println("Motor eléctrico") }` es implementación concreta intercambiable. `class Car(private val engine: Engine)` usa composición: `Car` no hereda de `Engine`, lo tiene como dependencia inyectada por constructor (`private val` almacena referencia). `fun drive() { engine.start(); println("En marcha") }` delega arranque al colaborador antes de lógica propia. Podrías sustituir `ElectricEngine` por otra implementación sin cambiar `Car`, cumpliendo principio de inversión de dependencias. Kotlin no necesita `@Inject` aquí, pero [Hilt](pregunta:74) proveería `Engine` en runtime. Herencia `Car : Engine` sería "es un" incorrecto semánticamente; composición modela "tiene un". Constructor primario conciso evita boilerplate Java.

Consejo:
Prefiere composición para reutilizar comportamiento. Usa herencia cuando exista una relación estable y el subtipo cumpla completamente el contrato del padre.
