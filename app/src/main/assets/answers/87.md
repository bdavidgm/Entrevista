## Concepto
Dispatchers.Main es el dispatcher de coroutines asociado al hilo principal de Android (UI thread), donde deben ejecutarse mutaciones de Views y Composables. Dispatchers.IO y Default se usan para IO y CPU respectivamente.

## Explicación detallada
viewModelScope por defecto usa Dispatchers.Main.immediate en Android (configuración del artifact kotlinx-coroutines-android). Actualizar MutableStateFlow desde viewModelScope.launch {} suele ser seguro en Main.

withContext(Dispatchers.IO) para red, disco, BD. Dispatchers.Default para cálculos CPU intensivos (sort grande, parsing JSON pesado). No uses IO para CPU puro (pool compartido con red).

Main.immediate ejecuta en el mismo hilo si ya estás en Main sin dispatch extra. Testing: MainDispatcherRule o setMain(StandardTestDispatcher).

Violación: bloquear Main con Thread.sleep o compute pesado causa ANR y jank. Compose recomposición ocurre en Main.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

## Cuándo usarlo
UI updates en Main; red/BD en IO; transformaciones pesadas en Default. Inyecta dispatchers en tests con @IoDispatcher qualifier.

Ejemplo:
@HiltViewModel
class ReportViewModel @Inject constructor(
    @IoDispatcher private val io: CoroutineDispatcher
) : ViewModel() {
    fun generate() = viewModelScope.launch {
        val report = withContext(io) { heavyRepository.buildReport() }
        _uiState.update { it.copy(report = report) }
    }
}


ExplicaciónKotlin:
La anotación `@HiltViewModel` marca la clase para que el compilador de Hilt genere el `ViewModel` factory y resuelva dependencias del constructor primario. `@Inject constructor(...)` declara inyección explícita del parámetro `io` tipado como `CoroutineDispatcher`, calificado con `@IoDispatcher` (qualifier personalizado de Hilt). La herencia `: ViewModel()` integra el ciclo de vida del ViewModel con `viewModelScope`, un `CoroutineScope` cancelado automáticamente en `onCleared()`. La función `generate()` devuelve el `Job` de `viewModelScope.launch { }`, una corrutina suspendida que arranca en el dispatcher Main por defecto en Android. Dentro, `withContext(io)` es una función suspendida que cambia temporalmente de contexto al dispatcher IO para ejecutar `heavyRepository.buildReport()` sin bloquear Main; al terminar, el control vuelve al dispatcher original. `_uiState.update { it.copy(report = report) }` usa la extensión de `MutableStateFlow` que recibe una lambda `(T) -> T`: `it` es el estado actual y `copy()` es el método generado por la `data class` del estado UI, produciendo una nueva instancia inmutable con el campo `report` actualizado.

Consejo para entrevista:
Nombra los tres dispatchers y qué pasa si haces red en Main. Menciona MainDispatcherRule en tests.
