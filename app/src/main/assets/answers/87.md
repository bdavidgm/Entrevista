## Concepto
[Dispatchers](pregunta:30).Main es el dispatcher de [coroutines](pregunta:78) asociado al hilo principal de Android (UI thread), donde deben ejecutarse mutaciones de Views y Composables. [Dispatchers.IO](pregunta:30) y Default se usan para IO y CPU respectivamente.

## Explicación detallada
[viewModelScope](pregunta:131) por defecto usa [Dispatchers](pregunta:30).Main.immediate en Android (configuración del artifact kotlinx-[coroutines](pregunta:78)-android). Actualizar [MutableStateFlow](pregunta:86) desde [viewModelScope](pregunta:131).launch {} suele ser seguro en Main.

[withContext](pregunta:82)([Dispatchers.IO](pregunta:30)) para red, disco, BD. [Dispatchers.Default](pregunta:30) para cálculos CPU intensivos (sort grande, parsing JSON pesado). No uses IO para CPU puro (pool compartido con red).

Main.immediate ejecuta en el mismo hilo si ya estás en Main sin dispatch extra. Testing: [MainDispatcherRule](pregunta:148) o setMain(StandardTestDispatcher).

Violación: bloquear Main con Thread.sleep o compute pesado causa [ANR](pregunta:95) y jank. Compose [recomposición](pregunta:47) ocurre en Main.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, [ANR](pregunta:95), Vitals) cuando aplique. ## Cuándo usarlo
UI updates en Main; red/BD en IO; transformaciones pesadas en Default. Inyecta [dispatchers](pregunta:30) en tests con @IoDispatcher qualifier.

Ejemplo:
@HiltViewModel
class ReportViewModel @Inject constructor(
    @IoDispatcher private val io: CoroutineDispatcher
) : ViewModel() {
    fun generate() = viewModelScope.launch {
        val report = withContext(io) { heavyRepository.buildReport() }
        _uiState.update { it.copy(report = report) }
    }
}


ExplicaciónKotlin:
La anotación `@HiltViewModel` marca la clase para que el compilador de [Hilt](pregunta:74) genere el `ViewModel` factory y resuelva dependencias del constructor primario. `@Inject constructor(...)` declara inyección explícita del parámetro `io` tipado como `CoroutineDispatcher`, calificado con `@IoDispatcher` (qualifier personalizado de [Hilt](pregunta:74)). La herencia `: ViewModel()` integra el ciclo de vida del [ViewModel](pregunta:68) con `viewModelScope`, un `CoroutineScope` cancelado automáticamente en `onCleared()`. La función `generate()` devuelve el `Job` de `viewModelScope.launch { }`, una [corrutina](pregunta:78) suspendida que arranca en el dispatcher Main por defecto en Android. Dentro, `withContext(io)` es una [función suspend](pregunta:79)ida que cambia temporalmente de contexto al dispatcher IO para ejecutar `heavyRepository.buildReport()` sin bloquear Main; al terminar, el control vuelve al dispatcher original. `_uiState.update { it.copy(report = report) }` usa la extensión de `MutableStateFlow` que recibe una lambda `(T) -> T`: `it` es el estado actual y `copy()` es el método generado por la `data class` del estado UI, produciendo una nueva instancia inmutable con el campo `report` actualizado.

Nombra los tres [dispatchers](pregunta:30) y qué pasa si haces red en Main. Menciona [MainDispatcherRule](pregunta:148) en tests.
