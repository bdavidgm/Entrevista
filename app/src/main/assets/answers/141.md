## Concepto
@HiltViewModel marca un [ViewModel](pregunta:68) para que [Hilt](pregunta:74) genere el ViewModelProvider.Factory e inyecte el constructor (repos, [use case](pregunta:66)s, [SavedStateHandle](pregunta:140), etc.). En pantallas usas hiltViewModel() (Compose) o by viewModels() ([Fragment](pregunta:32)/[Activity](pregunta:31)) con @AndroidEntryPoint.

Evita escribir Factory manual ni pasar dependencias a mano entre capas.

Ejemplo:
@HiltViewModel
class ProfileViewModel @Inject constructor(
    private val getUser: GetUserUseCase,
    savedStateHandle: SavedStateHandle
) : ViewModel() {
    private val userId: String = checkNotNull(savedStateHandle["userId"])
}

@Composable
fun ProfileRoute() {
    val vm: ProfileViewModel = hiltViewModel()
}


ExplicaciónKotlin:
`@HiltViewModel` y `@Inject constructor` registran dependencias `GetUserUseCase` y `SavedStateHandle` en el grafo de [Hilt](pregunta:74). `private val userId: String = checkNotNull(savedStateHandle["userId"])` usa `checkNotNull` de la stdlib: si el valor es null lanza `IllegalStateException` con mensaje; el compilador smart-castea a `String` no nullable. El operador `get` de `SavedStateHandle` devuelve nullable porque la clave puede faltar. `@Composable fun ProfileRoute()` sin parámetros obtiene [ViewModel](pregunta:68) con `hiltViewModel()` y anotación de tipo explícita `val vm: ProfileViewModel` — la inferencia también sería válida. `hiltViewModel()` es [composable](pregunta:46) y debe llamarse desde contexto [@Composable](pregunta:46); [Hilt](pregunta:74) resuelve el scope de navegación/[back stack](pregunta:118).

Consejo:
No inyectes [Activity](pregunta:31) ni [Context](pregunta:127) de UI; si hace falta [Context](pregunta:127), @ApplicationContext. Los bindings interfaz→impl van en módulos @InstallIn(ViewModelComponent::class) con @ViewModelScoped cuando aplique.
