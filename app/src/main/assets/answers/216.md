## Concepto
Publicar en Play Console implica: cuenta de desarrollador, crear la app, subir AAB firmado, completar ficha de tienda (textos, capturas, clasificación de contenido), declarar permisos sensibles/políticas de privacidad, configurar países y precios, pruebas internas/cerradas/abiertas y revisión de Google.

Tras aprobación, releases por fases o completos.

Ejemplo:
bundleRelease → firma con keystore → Play Console → Producción → revisión (horas/días).

ExplicaciónKotlin:
El ejemplo no muestra código Kotlin, sino el flujo de publicación que normalmente se dispara desde Gradle con tareas como `bundleRelease` definidas en el script Kotlin DSL (`build.gradle.kts`). En ese archivo, el bloque `android { buildTypes { release { ... } } }` configura variantes de compilación usando funciones y propiedades tipadas del Android Gradle Plugin. La firma se enlaza mediante `signingConfigs.create("release") { ... }`, donde lambdas y asignaciones con `=` son sintaxis idiomática de Kotlin. Las credenciales suelen leerse con `System.getenv("STORE_PW")`, evitando secretos en el repositorio. El resultado es un `.aab` generado en `build/outputs/bundle/release/`, listo para subir manualmente o con plugins de CI escritos también en Kotlin. Desde el lado del proyecto, conviene tratar la publicación como otro paso del pipeline: el mismo módulo `:app` que compilas en debug es el que produce el artefacto firmado. En entrevista, relacionar Gradle Kotlin DSL con Play Console demuestra que entiendes tanto el build como el ciclo de release.

Consejo:
Data safety form, política de permisos de ubicación en background y mensajes de promoción in-app son causas frecuentes de rechazo. Planifica rollout por porcentajes.
