## Concepto
@Insert es la anotación de [Room](pregunta:70) para definir métodos que insertan filas. Admite el parámetro onConflict (OnConflictStrategy.REPLACE / IGNORE / ABORT) para decidir qué hacer si ya existe una fila con la misma clave primaria. Acepta una entidad, varargs o List, y devuelve void/Long/List<Long> según rowIds.

Para "insert-or-update" sin escribir tu propio SQL, hoy es habitual usar @Upsert ([Room](pregunta:70) 2.5+).

Ejemplo:
@Dao
interface UserDao {
    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun upsertAll(users: List<User>)

    @Upsert
    suspend fun upsert(user: User)
}


ExplicaciónKotlin:
`interface UserDao` declara contrato sin implementación; [Room](pregunta:70) genera la implementación en tiempo de compilación. `@Insert(onConflict = OnConflictStrategy.REPLACE)` es anotación con parámetro nombrado que referencia enum `OnConflictStrategy.REPLACE`. `suspend fun upsertAll(users: List<User>)` marca [función suspend](pregunta:79)ida que solo puede llamarse desde [corrutina](pregunta:78) u otra [suspend](pregunta:79); `List<User>` es tipo de solo lectura inmutable en la firma. `@Upsert` es anotación más reciente que [Room](pregunta:70) traduce a insert-or-update sin SQL manual. Ambas funciones devuelven `Unit` implícitamente. Las interfaces en Kotlin pueden contener funciones [suspend](pregunta:79) y anotaciones JVM/Android; no necesitan cuerpo. El compilador de [Room](pregunta:70) valida tipos de entidad y claves en compile-time.

Consejo:
"REPLACE puede borrar y reinsertar (cuidado con triggers/foreign keys cascada)". Para insertar y actualizar mejor @Upsert, más explícito.
