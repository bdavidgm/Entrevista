Concepto:
SideEffect y DisposableEffect son APIs de efectos secundarios en Compose. SideEffect publica lógica que debe ejecutarse en cada recomposición exitosa después del layout, en el hilo UI. DisposableEffect ejecuta onDispose al salir del árbol o cuando cambian keys, ideal para registrar y liberar recursos.

Explicación detallada:
SideEffect { analytics.setScreen(name) } — sincrónico, no suspend. Útil sincronizar estado de no-Compose (Analytics, listeners simples). Cuidado: no hacer trabajo pesado.

DisposableEffect(lifecycleOwner) { val observer = ...; lifecycle.addObserver(observer); onDispose { lifecycle.removeObserver(observer) } } — patrón clásico para LifecycleObserver, BroadcastReceiver dinámico, registrar callback.

Comparación LaunchedEffect: suspend y corrutina; DisposableEffect: setup/teardown pareado. rememberUpdatedState mantiene referencia actualizada en efectos con callbacks stale.

En entrevistas: cuándo DisposableEffect vs LaunchedEffect; fugas si olvidas onDispose. AndroidView factory/update también usa DisposableEffect internamente.

Cuándo usarlo:
DisposableEffect: suscripciones, listeners, animaciones con cleanup. SideEffect: sincronizar valor Compose con API imperativa ligera cada frame lógico.

Ejemplo:
@Composable
fun MapLocationTracker(onLocation: (Location) -> Unit) {
    val currentOnLocation by rememberUpdatedState(onLocation)
    DisposableEffect(Unit) {
        val client = LocationClient { loc -> currentOnLocation(loc) }
        client.start()
        onDispose { client.stop() }
    }
}

@Composable
fun TrackScreenView(screenName: String) {
    SideEffect {
        Analytics.logScreen(screenName)
    }
}

Consejo para entrevista:
DisposableEffect onDispose siempre. SideEffect no para IO. rememberUpdatedState evita capturar callback viejo.
