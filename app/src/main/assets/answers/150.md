## Concepto
Un evento de UI one-shot es una acción que debe ejecutarse una sola vez (navegar, mostrar Snackbar, cerrar pantalla) y no formar parte del estado persistente. Se modela con un canal aparte ([Channel](pregunta:144) + receiveAsFlow o [SharedFlow](pregunta:86) con replay=0), no con [StateFlow](pregunta:86), para evitar reemisión al rotar.

El consumidor (Compose, [Fragment](pregunta:32)) los recolecta solo cuando está STARTED y los ejecuta como side effects.

Ejemplo:
class LoginViewModel : ViewModel() {
    private val _events = Channel<LoginEvent>(Channel.BUFFERED)
    val events = _events.receiveAsFlow()

    fun onLogin() = viewModelScope.launch {
        _events.send(LoginEvent.NavigateHome)
    }
}

// Compose
LaunchedEffect(Unit) {
    vm.events.collect { event -> when (event) { ... } }
}


ExplicaciónKotlin:
`private val _events = Channel<LoginEvent>(Channel.BUFFERED)` encapsula canal mutable; `val events = _events.receiveAsFlow()` expone `Flow` de solo consumo transformando el canal. `fun onLogin() = viewModelScope.launch { _events.send(LoginEvent.NavigateHome) }` combina scope del [ViewModel](pregunta:68) con envío suspendido de evento sellado (`LoginEvent` probablemente [sealed class](pregunta:19)). `send` suspende si el buffer está lleno. En Compose, `LaunchedEffect(Unit) { vm.events.collect { event -> when (event) { ... } } }` colecta flujo como side effect; `when` exhaustivo si `LoginEvent` es sealed. El comentario `// Compose` indica contexto de uso. Separar `_events` privado y `events` público es patrón de [encapsulación](pregunta:244) idiomático en Kotlin.

Consejo:
"State para datos, [Channel](pregunta:144)/[SharedFlow](pregunta:86)(0) para eventos one-shot". Evita [LiveData](pregunta:69)<Event<T>> con wrappers, ya no hace falta con Flows bien diseñados.
