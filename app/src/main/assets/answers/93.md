## Concepto
Turbine es una librería de testing para [Flow](pregunta:84) que facilita assertar emisiones de forma declarativa con awaitItem(), awaitComplete() y cancel, reduciendo race conditions frente a collect manual en tests de [corrutinas](pregunta:78).

## Explicación detallada
[flow](pregunta:84).test { } abre un scope donde cada awaitItem() suspende hasta la siguiente emisión o falla por timeout. assertEquals sobre el valor recibido. awaitComplete() verifica cierre limpio del [Flow](pregunta:84). Para [StateFlow](pregunta:86) hot, el primer awaitItem suele ser el valor actual al suscribirse; planifica el orden de asserts.

Operadores útiles: skipItems(n) salta emisiones iniciales; expectNoEvents() dentro de un bloque verifica que no lleguen más valores en un intervalo; cancelAndIgnoreRemainingEvents() termina el test sin esperar infinito en flows que no completan.

Combina con kotlinx-[coroutines](pregunta:78)-test runTest y [MainDispatcherRule](pregunta:148). Para debounce, advanceTimeBy en el TestScope antes de awaitItem. Para flatMapLatest, emite búsquedas rápidas y verifica que solo la última llega al collector.

Turbine evita el patrón frágil de launch + delay + assert en un MutableList compartido, que causa flakes en CI. Es estándar en proyectos Kotlin que testean [ViewModel](pregunta:68) [StateFlow](pregunta:86), [repository](pregunta:63) [Room](pregunta:70) [Flow](pregunta:84) y operadores personalizados.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto y trade-offs frente a collect manual o toList(). Mencionar Turbine demuestra experiencia con [corrutinas](pregunta:78) modernas en tests.

Errores comunes: no cancelar el test y dejar collectors colgados; olvidar advanceUntilIdle antes de awaitItem cuando el [Flow](pregunta:84) emite desde otra [coroutine](pregunta:78); comparar listas completas sin tener en cuenta emisiones intermedias de loading. La dependencia es cashapp.turbine:turbine en testImplementation.

Para [SharedFlow](pregunta:86) de eventos UI, usa test { action(); awaitItem() es el evento esperado } y verifica que no hay replay no deseado. Integración con [Hilt](pregunta:74) y [ViewModel](pregunta:68): instancia [ViewModel](pregunta:68) con fakes, llama métodos públicos, Turbine en uiState o effects.

## Cuándo usarlo
Cualquier test que colecte [Flow](pregunta:84) en proyecto Kotlin moderno. Alternativa manual es frágil y propenso a flakes. No uses toList() en flows infinitos o sin complete.

Ejemplo:
@Test
fun searchEmitsResults() = runTest {
    repository.results.test {
        repository.search("kotlin")
        assertEquals(listOf("item"), awaitItem())
        cancelAndIgnoreRemainingEvents()
    }
}

@Test
fun debouncedSearch() = runTest {
    viewModel.queryFlow.test {
        viewModel.onQuery("a")
        advanceTimeBy(300)
        val result = awaitItem()
        assertTrue(result.isNotEmpty())
        cancelAndIgnoreRemainingEvents()
    }
}


ExplicaciónKotlin:
Ambos tests usan `runTest` como scope de [corrutinas](pregunta:78) controlado. En el primero, `repository.results.test { }` es la API de Turbine: abre un collector sobre el `Flow` y permite `awaitItem()` suspendido hasta la siguiente emisión. `repository.search("kotlin")` dispara la lógica; `assertEquals(listOf("item"), awaitItem())` compara la lista recibida con la esperada. `cancelAndIgnoreRemainingEvents()` cierra el test sin esperar completion del [Flow](pregunta:84) (útil en flows calientes o infinitos). El segundo test usa `advanceTimeBy(300)` del `TestScope` para simular paso de 300 ms en un debounce. `viewModel.onQuery("a")` emite entrada; tras avanzar el tiempo, `awaitItem()` captura la emisión debounced. `assertTrue(result.isNotEmpty())` usa función de kotlin.test/[JUnit](pregunta:89) sobre la colección recibida.

Consejo para entrevista:
Nombre Turbine y awaitItem. Contrasta con toList() que puede colgar si [flow](pregunta:84) infinito. Menciona timeout y primer valor de [StateFlow](pregunta:86).
