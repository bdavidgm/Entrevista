## Concepto
Clean Architecture organiza el código en capas concéntricas con una regla de dependencia: las capas externas conocen las internas, nunca al revés. Típicamente: UI / Presentation → Domain (entidades y [use case](pregunta:66)s) → Data (repositorios e implementaciones). El dominio no depende de Android.

Aporta testabilidad y desacoplamiento; el coste es más boilerplate y archivos, por lo que conviene aplicarla con criterio según el tamaño del proyecto.

Ejemplo:
// Domain
interface UserRepository { suspend fun get(id: String): User }
class GetUserUseCase(private val repo: UserRepository) {
    suspend operator fun invoke(id: String): User = repo.get(id)
}

// Data
class UserRepositoryImpl(private val api: UserApi) : UserRepository {
    override suspend fun get(id: String): User = api.fetch(id).toDomain()
}

// Presentation (ViewModel) usa GetUserUseCase

Consejo:
"Las flechas apuntan hacia dentro". En apps pequeñas suele bastar con [ViewModel](pregunta:68) + [Repository](pregunta:63); reserva Clean completo para apps grandes/modulares.
