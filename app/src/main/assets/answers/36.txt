Concepto:
Un Service es un componente sin UI que realiza operaciones de larga duración en segundo plano o expone funcionalidad a otros componentes. Existen tres tipos: started (se inicia con startService), bound (se enlaza con bindService) y foreground (visible al usuario mediante notificación).

A partir de Android 8 hay fuertes restricciones para servicios en background; para trabajo diferible suele preferirse WorkManager, y para tareas visibles (música, navegación, llamada) un Foreground Service.

Ejemplo:
class MusicService : Service() {
    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        startForeground(NOTIF_ID, buildNotification())
        playMusic()
        return START_STICKY
    }
    override fun onBind(intent: Intent?): IBinder? = null
}

ExplicaciónKotlin:
`class MusicService : Service()` hereda del componente Android `Service`. `override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int` sobrescribe el callback con parámetros anulables (`Intent?`) y retorno `Int` (constant de reinicio). `startForeground(NOTIF_ID, buildNotification())` es llamada a API Android dentro del servicio. `return START_STICKY` devuelve una constante Int que indica al sistema cómo reiniciar el servicio. `override fun onBind(intent: Intent?): IBinder? = null` usa expresión de cuerpo único: devuelve null indicando que no es un servicio bound. Kotlin permite mezclar bloques con llaves y funciones de una sola expresión. Las clases de componentes Android en Kotlin siguen el mismo ciclo de vida que en Java, con ventajas de nullability explícita en firmas como `Intent?`.

Consejo:
"Hoy usa WorkManager para trabajo diferible y Foreground Service solo si hay que ser visible". Conoce los tipos y los límites de background a partir de Android 8.
