## Concepto
Un test de integración verifica que varias piezas reales o casi reales cooperan: por ejemplo [Room](pregunta:70) + [Repository](pregunta:63), navegación + [ViewModel](pregunta:68), o un módulo contra un servidor de pruebas. No sustituye tests unitarios rápidos ni tests de UI end-to-end.

Suelen vivir en androidTest, ser más lentos y usar [Hilt](pregunta:74) test modules o bases de datos en memoria.

Ejemplo:
@Test
fun `dao y repo devuelven mismos datos`() = runTest {
    val db = Room.inMemoryDatabaseBuilder(context, AppDb::class.java).build()
    val repo = UserRepositoryImpl(db.dao(), FakeApi())
    repo.refresh()
    assertEquals(1, repo.observeUsers().first().size)
}


ExplicaciónKotlin:
La función de test con nombre entre backticks (dao y repo devuelven mismos datos) usa `runTest` de kotlinx-[coroutines](pregunta:78)-test, scheduler de tiempo de prueba en lugar de bloquear. `Room.inMemoryDatabaseBuilder(context, AppDb::class.java).build()` encadena builder con `.build()` final; la DB vive en RAM. `UserRepositoryImpl(db.dao(), FakeApi())` construye implementación con constructor posicional; `FakeApi()` es clase de prueba. `repo.refresh()` debe ser `suspend` invocado dentro de `runTest`. `repo.observeUsers().first()` es extensión de `Flow` que suspende hasta la primera emisión — import de `kotlinx.coroutines.flow.first`. `assertEquals(1, repo.observeUsers().first().size)` compara tamaño de lista; aquí se llama `first()` dos veces (patrón mejorable con variable local). El test valida integración real [Room](pregunta:70) + [repositorio](pregunta:63).

Consejo:
Distingue unit (lógica aislada), integration (varias capas) y UI ([Espresso](pregunta:91)/Compose). Los de integración detectan fallos de wiring y migraciones.
