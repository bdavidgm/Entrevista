Concepto:
Tipos de permisos en Android: normales (solo manifest, bajo riesgo: INTERNET), peligrosos/dangerous (runtime: cámara, micrófono, SMS), especiales (WRITE_SETTINGS, MANAGE_EXTERNAL_STORAGE, PACKAGE_USAGE_STATS, SCHEDULE_EXACT_ALARM…) y por instalación (notification en 13+, nearby en 12+).

Cada categoría tiene flujo y políticas de Play Console distintos.

Ejemplo:
<uses-permission android:name="android.permission.CAMERA"/>
ActivityCompat.requestPermissions(this, arrayOf(Manifest.permission.CAMERA), REQ)

ExplicaciónKotlin:
La línea XML declara permiso en manifest; en Kotlin moderno preferirías `android.Manifest.permission.CAMERA` con `<uses-permission>` equivalente. `ActivityCompat.requestPermissions(this, arrayOf(Manifest.permission.CAMERA), REQ)` usa `arrayOf()` para crear un `Array<String>` varargs-compatible con la API Java de AndroidX. `this` debe ser `Activity` o `Fragment` que implemente el callback de resultado (hoy sustituido por `registerForActivityResult`). `REQ` es una constante `Int` definida típicamente en `companion object`. `Manifest.permission.CAMERA` es constante generada en compile time desde el manifest merge. En Kotlin idiomático actual, `ActivityResultContracts.RequestPermission()` evita códigos mágicos y callbacks deprecados. Los permisos normales como `INTERNET` no requieren `requestPermissions`; los dangerous sí en runtime desde API 23+. La mezcla XML + Kotlin refleja transición de proyectos Java a Kotlin sin migrar aún el contrato de permisos moderno.

Consejo:
Declara el permiso mínimo (fine vs coarse location). Para APIs especiales guía al usuario a la pantalla de Ajustes con intents de settings.
