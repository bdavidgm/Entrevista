## Concepto
El hilo principal (UI thread) en Android es responsable de dibujar frames, despachar eventos táctiles y ejecutar callbacks de Activity. Realizar trabajo largo o bloqueante en Main provoca jank, frames perdidos y ANR (Application Not Responding).

## Explicación detallada
Regla histórica: NetworkOnMainThreadException en API estricta para red en main. Room y muchas APIs modernas exigen background. Compose recomposición y medición ocurren en Main; cálculos pesados en composición bloquean 16ms por frame.

Solución: coroutines withContext(IO), Executor, WorkManager para deferible. Actualizar UI solo en Main tras resultado. StrictMode detecta disk/network en main en debug.

Perfis: Systrace, Android Studio Profiler CPU. ANR trace muestra main bloqueado.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto de tu experiencia o proyecto demo, y trade-offs frente a alternativas. Mencionar versiones recientes de API o biblioteca (Material3, Paging 3, Hilt 2.x) demuestra que sigues el ecosistema, sin necesidad de memorizar números de versión exactos.

## Cuándo usarlo
Siempre offload IO y CPU pesado. Lectura rápida DataStore ya es async. Imagen: Coil en background automático.

Ejemplo:
// Mal
fun onClick() {
    val json = URL(url).readText() // ANR risk
    textView.text = json
}

// Bien
fun onClick() = lifecycleScope.launch {
    val json = withContext(Dispatchers.IO) { api.fetch() }
    textView.text = json
}


ExplicaciónKotlin:
El ejemplo "Mal" muestra función `onClick()` regular (no suspend) que llama `URL(url).readText()`: operación bloqueante de red en Main. `textView.text = json` muta la vista desde el mismo hilo. El ejemplo "Bien" usa `fun onClick() = lifecycleScope.launch { }`: expresión de cuerpo único donde `launch` devuelve `Job` y la función devuelve ese Job implícitamente. `lifecycleScope` es extensión de `LifecycleOwner` cancelada al destruir el lifecycle. `withContext(Dispatchers.IO) { api.fetch() }` cambia a pool IO para trabajo bloqueante; `api.fetch()` presumiblemente es `suspend` o bloqueante envuelto. Tras `withContext`, la corrutina vuelve a Main (dispatcher de lanzamiento en Android), permitiendo `textView.text = json` de forma segura. `Dispatchers.IO` y `Dispatchers.Main` son singletons del objeto `Dispatchers`.

Consejo para entrevista:
Menciona 16ms frame budget y ANR. Herramientas: StrictMode, Profiler. Compose: no parses JSON en @Composable body.
