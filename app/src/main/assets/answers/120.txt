Concepto:
Android App Bundle (AAB) es el formato de publicación en Google Play: un paquete que Play Store usa para generar APKs optimizados por dispositivo (ABI, densidad, idioma), en lugar de subir un APK monolítico.

Explicación detallada:
Al subir un .aab, Google Play aplica Dynamic Delivery:
- Split APKs por arquitectura (arm64-v8a, armeabi-v7a, x86_64): el usuario descarga solo su ABI.
- Configuration APKs por densidad de pantalla e idioma: reduce tamaño vs APK universal con todos los recursos.
- Asset packs y feature modules para entrega bajo demanda (Play Feature Delivery).

El desarrollador compila con:
./gradlew bundleRelease
El artefacto está en app/build/outputs/bundle/release/

Play App Signing: Google puede firmar el APK final; subes el AAB firmado con upload key. Los usuarios instalan APKs firmados por la clave de distribución de Play.

AAB no se instala directamente en dispositivos de desarrollo habitualmente; para pruebas locales usas bundletool build-apks o instala vía Play Internal Testing.

Ventajas:
- Menor tamaño de descarga e instalación.
- Requisito para publicaciones nuevas en Play Console.
- Soporte de módulos dinámicos.

Limitaciones:
- Solo distribución vía Google Play (otras tiendas pueden requerir APK).
- Builds de debug siguen siendo APK para rapidez en IDE.

Cuándo usarlo:
Siempre para producción en Play Store desde 2021 (nuevas apps deben usar AAB). Para sideload o CI interno puedes generar APK desde bundle.

Ejemplo:
// build.gradle.kts
android {
    defaultConfig { ... }
    buildTypes {
        release {
            isMinifyEnabled = true
            proguardFiles(
                getDefaultProguardFile("proguard-android-optimize.txt"),
                "proguard-rules.pro"
            )
        }
    }
}
// Terminal: ./gradlew :app:bundleRelease

Consejo para entrevista:
Explica que AAB no es el APK que instala el usuario; Play lo divide. Menciona tamaño, Play Signing y diferencia con APK universal. No confundas con Android Instant (deprecated) salvo que pregunten histórico.
