Concepto:
minSdk (API mínima) define la versión Android más antigua en la que la app puede instalarse; targetSdk (API objetivo) indica para qué versión del sistema optimizaste y aceptas comportamientos de compatibilidad y políticas nuevas.

Explicación detallada:
minSdk:
- Dispositivos por debajo no ven la app en Play Store (o no instalan el APK).
- Permite usar APIs sin comprobaciones solo si compileSdk y minSdk lo permiten; para APIs superiores a minSdk usas Build.VERSION.SDK_INT o androidx checks.
- Subir minSdk reduce fragmentación pero excluye usuarios en versiones viejas.

targetSdk:
- Señal a Google Play y al sistema de que probaste contra esa versión.
- Google exige targetSdk reciente para publicar (políticas anuales).
- Activan cambios de comportamiento (scoped storage, permisos en runtime, restricciones de background, notificaciones con canal obligatorio desde API 26, etc.) aunque el dispositivo tenga Android más nuevo.

compileSdk:
- Versión del SDK con la que compilas; debe ser >= targetSdk. No afecta qué dispositivos instalan, solo qué APIs ves en compilación.

Ejemplo numérico: minSdk 26 (Android 8), targetSdk 35 (Android 15), compileSdk 35.

defaultConfig en Gradle:
minSdk = 26
targetSdk = 35

Cuándo usarlo:
Definir minSdk según analytics de usuarios y APIs necesarias (Biometric, WorkManager moderno). Actualizar targetSdk cada año antes de deadlines de Play. Probar en emuladores de targetSdk tras cada bump.

Ejemplo:
android {
    compileSdk = 35
    defaultConfig {
        minSdk = 26
        targetSdk = 35
        applicationId = "com.example.app"
    }
}

// Uso condicional de API
if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU) {
    requestPermissions(arrayOf(Manifest.permission.POST_NOTIFICATIONS), REQ_CODE)
}

Consejo para entrevista:
Diferencia las tres: compile (build), min (instalación), target (comportamiento/políticas). Menciona que subir targetSdk puede romper tests si no adaptas permisos y background limits.
