## Concepto
Git es control de versiones distribuido: ramas, merges, pull requests, bisect y etiquetas de release. En Android permite colaborar en paralelo, revisar cambios en [Gradle](pregunta:109)/manifestes y revertir regresiones con precisión.

Se integra con CI (GitHub Actions, Bitrise) para lint, tests y builds por commit.

Ejemplo:
git checkout -b feature/login
git commit -am "Add login screen"
git push -u origin feature/login

ExplicaciónKotlin:
El ejemplo son comandos Git en shell, no código Kotlin; contextualizan el flujo de trabajo donde conviven archivos `.kt`, [Gradle](pregunta:109) y recursos Android en el mismo [repositorio](pregunta:63). En proyectos Kotlin/Android, `.gitignore` suele excluir `build/`, `.gradle/` y archivos locales de IDE para no versionar artefactos compilados. Los commits que contienen cambios en `.kt` se revisan en pull requests igual que XML o [Gradle](pregunta:109). Desde Kotlin no se invoca Git directamente en runtime; herramientas CI ejecutan `./gradlew test` sobre el código Kotlin en cada push. La rama `feature/login` típicamente incluiría Activities/Fragments/Composables y tests en Kotlin. El flag `-u` de Git establece upstream para futuros `git push`/`pull`, independiente del lenguaje del proyecto.

Consejo:
Buenas prácticas: commits pequeños, .gitignore de build/, keystores fuera del repo o cifrados, y conventional commits si el equipo los usa.
