Concepto:
El modificador out marca un parámetro de tipo como covariante: el tipo solo aparece como salida (producer), nunca como entrada. Permite que Producer<Hijo> sea subtipo de Producer<Padre>, igual que List<String> puede tratarse como List<Any>.

Es la versión de "? extends T" de Java aplicada en la declaración, no en cada uso. Aporta seguridad de tipos al asignar genéricos relacionados.

Ejemplo:
interface Producer<out T> {
    fun produce(): T
}

class StringProducer : Producer<String> { override fun produce() = "hola" }

val producer: Producer<Any> = StringProducer() // OK gracias a out


ExplicaciónKotlin:
`interface Producer<out T> { fun produce(): T }` — type parameter marcado `out` (covariante): `T` solo aparece en posición de salida (return type), nunca en parámetros de entrada. Permite subtipado: `Producer<String>` es subtipo de `Producer<Any>`. `class StringProducer : Producer<String> { override fun produce() = "hola" }` — expresión única `= "hola"` devuelve String sin llaves. `val producer: Producer<Any> = StringProducer()` asignación legal por covarianza de `out T`. Sin `out`, la asignación no compilaría (invarianza). `out` en declaración es contrato de solo-producer, análogo PECS "extends" en Java.

Consejo:
"out = covariante = producer". Si necesitas leer y escribir, no puedes usar out; usa invariante o star projection según el caso.
