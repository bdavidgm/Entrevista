Concepto:
val declara una referencia de solo lectura (no se puede reasignar tras inicializarla), mientras que var declara una referencia mutable que sí admite reasignación. val no implica inmutabilidad profunda: el objeto referenciado puede seguir cambiando su estado interno.

Por defecto se prefiere val en Kotlin para favorecer la inmutabilidad y reducir errores de concurrencia y de razonamiento.

Ejemplo:
val list = mutableListOf(1, 2)
list.add(3)        // permitido: el objeto cambia
// list = mutableListOf() // error: no se puede reasignar

var counter = 0
counter += 1       // permitido

ExplicaciónKotlin:
`val list = mutableListOf(1, 2)` declara una referencia inmutable (`val`) a un objeto mutable: no puedes reasignar `list`, pero sí invocar métodos que alteren su contenido, como `add(3)`. `mutableListOf` es una función de fábrica del estándar que infiere `MutableList<Int>`. La línea comentada `list = mutableListOf()` fallaría en compilación porque `val` prohíbe reasignar la referencia. En cambio, `var counter = 0` declara una variable reasignable; Kotlin infiere `Int` y `counter += 1` es azúcar sintáctico equivalente a `counter = counter + 1`. Este ejemplo ilustra la distinción kotliniana entre inmutabilidad de la referencia (`val`/`var`) e inmutabilidad del objeto referenciado.

Consejo:
"val == final reference, no deep immutable". Para inmutabilidad real usa val + tipos inmutables (List, data class con val).
