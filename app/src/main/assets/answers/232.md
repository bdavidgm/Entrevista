## Concepto
Recovery es un mini-entorno de arranque para aplicar OTA, borrar datos (wipe) o montar almacenamiento. El recovery de stock es limitado; recoveries custom (TWRP) ofrecen backups nandroid, sideload ZIP y gestión de archivos con touch.

Se accede con combinación de teclas o [adb](pregunta:226) reboot recovery.

Ejemplo:
adb reboot recovery
// En TWRP: Install → seleccionar ROM zip → swipe

ExplicaciónKotlin:
Recovery es un entorno separado del runtime donde corre tu app Kotlin; `adb reboot recovery` reinicia fuera del sistema normal para aplicar OTAs o zips. En Kotlin de producción, las actualizaciones oficiales usan `PackageInstaller` o Play Core In-App Updates, no TWRP manual. Un zip de ROM puede contener APKs del sistema, pero tu módulo `:app` sigue compilándose con [Gradle](pregunta:109) [Kotlin DSL](pregunta:112) independientemente. TWRP sideload se invoca con `adb sideload rom.zip` desde recovery custom. Si cifrado de datos está activo, versiones incompatibles de recovery fallan antes de montar `/data`, donde [Room](pregunta:70) o [DataStore](pregunta:73) guardarían datos Kotlin. Los comentarios `// En TWRP: ...` son sintaxis Kotlin de comentario de línea aunque el ejemplo sea mayormente shell. Para desarrolladores, recovery importa al probar builds release firmados o wipes, no al escribir funciones `@Composable`. Distinguir recovery de `WorkManager` o `ForegroundService` aclara capas del stack Android.

Consejo:
Cifrado de datos + TWRP requiere versión compatible con tu Android. Los OTA oficiales pueden fallar con recovery modificado.
