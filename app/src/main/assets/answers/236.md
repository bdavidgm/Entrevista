## Concepto
Ubicación en Android se divide en aproximada (red) vs precisa (GPS/multilateración). Desde Android 10 necesitas ACCESS_BACKGROUND_LOCATION aparte de fine/coarse si rastreas con la app en segundo plano (y Foreground Service tipo location).

Minimiza frecuencia y precisión; muestra justificación en UI y desactiva tracking cuando basta coarse.

Ejemplo:
requestPermissions(arrayOf(ACCESS_FINE_LOCATION), 0)
// Si necesitas background:
requestPermissions(arrayOf(ACCESS_BACKGROUND_LOCATION), 1)

ExplicaciónKotlin:
`requestPermissions(arrayOf(ACCESS_FINE_LOCATION), 0)` asume extensión o import de `Manifest.permission.ACCESS_FINE_LOCATION`; `arrayOf()` crea el array requerido por la API legacy. El segundo argumento `0` es `requestCode`, patrón antiguo antes de Activity Result API. La segunda llamada pide `ACCESS_BACKGROUND_LOCATION`, permiso separado desde Android 10 que exige haber concedido fine/coarse en primer plano antes. En Kotlin moderno usarías `registerForActivityResult(RequestMultiplePermissions()) { map -> ... }` con lambdas que reciben `Map<String, Boolean>`. Para background real, además necesitas `ForegroundService` con `foregroundServiceType="location"` declarado en manifest. `arrayOf(ACCESS_BACKGROUND_LOCATION)` con código `1` distingue callbacks si mantienes API legacy. Kotlin nullable safety ayuda al manejar `grantResults` en `onRequestPermissionsResult`. Minimizar permisos se expresa en tipos: solo declarar constantes que realmente usas en código Kotlin.

Consejo:
Data access auditing y permisos temporales (approximate) reducen superficie. Play Console exige declaración detallada de uso de ubicación.
