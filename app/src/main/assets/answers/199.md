## Concepto
launchMode en el manifest controla cómo se instancia una [Activity](pregunta:31) ante nuevos [intents](pregunta:33): standard (siempre nueva), singleTop (reusa si está arriba del stack), singleTask (una instancia por task, limpia encima) y singleInstance (task exclusivo).

singleTask/singleInstance requieren entender taskAffinity y cuidado con datos en memoria.

Ejemplo:
<activity android:name=".MainActivity" android:launchMode="singleTask"/>

ExplicaciónKotlin:
El ejemplo es declaración XML en el manifest, no Kotlin; `android:launchMode="singleTask"` es atributo que el sistema lee al arrancar componentes. En Kotlin, el equivalente programático no existe para launchMode — debe declararse en XML o vía manifest merger en build [Gradle](pregunta:109). La clase `.MainActivity` se resuelve a `com.example.MainActivity` y en Kotlin se declararía `class MainActivity : AppCompatActivity()`. Cuando un [intent](pregunta:33) llega a una [Activity](pregunta:31) `singleTask` existente, Kotlin recibe el [intent](pregunta:33) en `onNewIntent(intent: Intent?)`, método [override](pregunta:247) con parámetro anulable. Comprender launchMode ayuda al diseñar apps Kotlin single-[Activity](pregunta:31) donde solo la host [Activity](pregunta:31) usa `singleTask` y la navegación interna es `NavController`. [Gradle](pregunta:109) y el manifest merger integran el XML con el código Kotlin en el APK final.

Consejo:
Sobreactual: la mayoría de apps Compose usan single-[activity](pregunta:31) + [NavHost](pregunta:60). launchMode sirve para flows especiales (pago, captura) o integración con notificaciones.
