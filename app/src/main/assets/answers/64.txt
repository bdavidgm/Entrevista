Concepto:
Clean Architecture organiza el código en capas concéntricas con una regla de dependencia: las capas externas conocen las internas, nunca al revés. Típicamente: UI / Presentation → Domain (entidades y use cases) → Data (repositorios e implementaciones). El dominio no depende de Android.

Aporta testabilidad y desacoplamiento; el coste es más boilerplate y archivos, por lo que conviene aplicarla con criterio según el tamaño del proyecto.

Ejemplo:
// Domain
interface UserRepository { suspend fun get(id: String): User }
class GetUserUseCase(private val repo: UserRepository) {
    suspend operator fun invoke(id: String): User = repo.get(id)
}

// Data
class UserRepositoryImpl(private val api: UserApi) : UserRepository {
    override suspend fun get(id: String): User = api.fetch(id).toDomain()
}

// Presentation (ViewModel) usa GetUserUseCase

ExplicaciónKotlin:
// Domain comentario separa capas. interface UserRepository { suspend fun get(id: String): User } define contrato con función suspend y parámetro id: String no nullable. class GetUserUseCase(private val repo: UserRepository) encapsula dependencia; suspend operator fun invoke(id: String): User = repo.get(id) usa operator fun invoke para invocar la clase como función (getUserUseCase(id)). UserRepositoryImpl(private val api: UserApi) : UserRepository implementa la interfaz con override suspend fun get(id: String): User = api.fetch(id).toDomain(): override indica implementación, encadenamiento fetch y extensión toDomain() convierte DTO a dominio. Las capas usan clases, interfaces, herencia (: UserRepository), funciones suspend y expresiones de una línea con =.

Consejo:
"Las flechas apuntan hacia dentro". En apps pequeñas suele bastar con ViewModel + Repository; reserva Clean completo para apps grandes/modulares.
