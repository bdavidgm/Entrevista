Concepto:
MVI (Model-View-Intent) es un patrón unidireccional donde la View envía Intents (acciones del usuario o del sistema), un componente reduce el estado anterior más la intent en un nuevo Model (estado inmutable único), y la View renderiza solo ese Model. Garantiza trazabilidad y reproducibilidad del estado de pantalla.

Explicación detallada:
A diferencia de MVVM con varios flujos sueltos, MVI centraliza todo en un State: data class ScreenState(val list, val error, val dialog). Cada cambio pasa por reduce(oldState, intent): ScreenState. Los efectos secundarios (red, navegación) se modelan como intents async o en capa intermedia (Actor, middleware).

En Kotlin/Android suele implementarse con sealed class Intent, data class State, y ViewModel que expone StateFlow y función dispatch(Intent). Compose encaja bien: un solo collectAsState y when sobre state. Ventaja: debugging con logs de intents, time-travel en teoría, tests deterministas del reducer.

Desventajas: boilerplate para pantallas simples, riesgo de estados imposibles si el reducer no es exhaustivo, y eventos one-shot (toast) requieren patrón adicional (effect channel). Orbit MVI, MVIKotlin o implementación manual son opciones.

Comparación MVVM: MVVM permite múltiples LiveData; MVI fuerza un solo estado. MVI se parece a Redux; en entrevistas menciona predictibilidad y que es popular en equipos con flujos complejos (fintech, wizards).

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto de tu experiencia o proyecto demo, y trade-offs frente a alternativas. Mencionar versiones recientes de API o biblioteca (Material3, Paging 3, Hilt 2.x) demuestra que sigues el ecosistema, sin necesidad de memorizar números de versión exactos.

Para profundizar tras la entrevista, revisa documentación oficial, codelabs y el código fuente de tu propio proyecto Entrevista aplicando el concepto en un flujo mínimo reproducible. La práctica deliberada con tests unitarios refuerza la confianza al explicar el tema en viva voz.

Preguntas de seguimiento frecuentes en entrevistas para este tema incluyen comparación con alternativas del ecosistema, implicaciones en testing y en el ciclo de vida de Android, y cómo lo aplicarías en una app modular con varios flavors. Si el entrevistador pide profundidad, menciona herramientas de diagnóstico (Profiler, Lint, StrictMode, Layout Inspector) y cómo validarías en CI que no hay regresiones.

En proyectos reales, documenta decisiones en ADR breves o en README del módulo para que el equipo no revierta el patrón por desconocimiento. La mejor respuesta de entrevista combina definición precisa, ejemplo Kotlin o Compose concreto y honestidad sobre límites: ninguna solución es gratis en complejidad o tamaño de APK.

Relaciona este concepto con los demás bloques del manifiesto Entrevista (arquitectura, corrutinas, seguridad, build) para demostrar visión de sistema completo, no conocimiento aislado de una API.

Cuándo usarlo:
Pantallas con muchos estados combinados (loading + error + partial data + modal), equipos que valoran auditoría de cambios, o cuando ya usan librería MVI. Para CRUD simple, MVVM basta.

Ejemplo:
sealed interface CartIntent {
    data object Load : CartIntent
    data class Remove(val id: String) : CartIntent
}

data class CartState(val items: List<Item> = emptyList(), val loading: Boolean = false)

class CartViewModel : ViewModel() {
    private val _state = MutableStateFlow(CartState())
    val state = _state.asStateFlow()

    fun dispatch(intent: CartIntent) = viewModelScope.launch {
        _state.update { reduce(it, intent) }
    }
}

private fun reduce(state: CartState, intent: CartIntent): CartState = when (intent) {
    CartIntent.Load -> state.copy(loading = true)
    is CartIntent.Remove -> state.copy(items = state.items.filter { it.id != intent.id })
}

Consejo para entrevista:
Explica ciclo Intent -> Reducer -> State -> View. Menciona eventos únicos como problema y solución (Channel/Effect). No digas que MVI reemplaza siempre a MVVM; es trade-off de complejidad.
