Concepto:
Un ContentProvider es un componente que expone datos estructurados de la app a otros procesos mediante un modelo de URIs y operaciones CRUD (query, insert, update, delete) mediadas por ContentResolver. También permite compartir archivos con FileProvider y es la base de acceso a datos del sistema (contactos, media).

Explicación detallada:
Los clientes no acceden a tu base de datos directamente; usan content://authority/path con cursor. Room puede exponer provider en casos raros; suele usarse FileProvider para compartir URI seguras de caché (cámara, share sheet).

android:exported y permisos read/write protegen datos sensibles. Contract class documenta columnas y URIs. LoaderManager antiguo; hoy preferir Repository + Room dentro de tu app sin provider salvo interoperabilidad.

Cuándo usarlo:
Compartir archivos con otras apps de forma segura (FileProvider casi obligatorio), SDK que expone datos a terceros, integración con buscadores. No crees provider para cada tabla interna si no hay consumidor externo.

Ejemplo:
// AndroidManifest
// <provider android:name="androidx.core.content.FileProvider"
//     android:authorities="${applicationId}.fileprovider"
//     android:exported="false"
//     android:grantUriPermissions="true">

fun shareImage(context: Context, file: File) {
    val uri = FileProvider.getUriForFile(
        context,
        "${context.packageName}.fileprovider",
        file
    )
    val intent = Intent(Intent.ACTION_SEND).apply {
        type = "image/*"
        putExtra(Intent.EXTRA_STREAM, uri)
        addFlags(Intent.FLAG_GRANT_READ_URI_PERMISSION)
    }
    context.startActivity(Intent.createChooser(intent, null))
}

Consejo para entrevista:
FileProvider es pregunta muy frecuente aunque no sea provider SQL. URI permissions. Contrasta con almacenamiento interno privado.
