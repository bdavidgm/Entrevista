## Concepto
runBlocking bloquea el hilo actual hasta que termina el bloque suspendido y crea ahí un event loop de corrutinas. Útil en tests JVM o en main() de herramientas, pero no en la UI de Android: en el hilo principal puede provocar ANR.

En producción prefiere viewModelScope, lifecycleScope o withContext desde código suspendido.

Ejemplo:
@Test
fun `repo devuelve usuario`() = runBlocking {
    val user = repository.getUser("1")
    assertEquals("Ana", user.name)
}


ExplicaciónKotlin:
La anotación `@Test` marca una función de test cuyo nombre usa backticks en el código fuente (`repo devuelve usuario`) para permitir espacios y acentos. `runBlocking` es función de la librería kotlinx.coroutines que bloquea el hilo llamante hasta completar el lambda trailing suspendido. Dentro, `repository.getUser("1")` requiere contexto suspend o bloqueante; aquí es válido por estar en `runBlocking`. `assertEquals("Ana", user.name)` compara valor esperado con propiedad `name` de la data class u objeto devuelto — típico de JUnit en tests JVM. La función de test devuelve implícitamente `Unit`; el bloque es expresión única tras `=`. Este patrón es idiomático en tests unitarios Kotlin, no en código Android de producción.

Consejo:
"Tests y CLI sí; Activities/Composables no". Si ves runBlocking en onCreate o en un click, es señal de refactor a launch.
