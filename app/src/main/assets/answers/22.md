## Concepto
apply y also devuelven ambos el receptor (útiles en cadenas builder), pero se diferencian en cómo lo exponen: apply usa this implícito (accedes a propiedades y métodos directamente), mientras que also lo expone como it (queda explícito que estás trabajando con el objeto).

Regla práctica: apply para configurar el objeto desde dentro (mutar campos, llamar setters); also para side effects laterales (loggear, validar, registrar) sin contaminar this.

Ejemplo:
val user = User().apply {
    name = "Ana"
    age = 30
}.also {
    Log.d(TAG, "Usuario creado: ${it.id}")
    analytics.track(it)
}

ExplicaciónKotlin:
`User()` invoca el constructor sin argumentos (o el default de la clase). `.apply { name = "Ana"; age = 30 }` configura propiedades mutables del receptor con `this` implícito; `apply` devuelve el mismo `User` configurado, permitiendo encadenar. `.also { ... }` recibe el objeto como `it` (parámetro explícito del lambda) y ejecuta efectos secundarios: logging con plantilla `${it.id}` e `analytics.track(it)`; `also` también devuelve el receptor original, no el resultado del bloque. La diferencia clave: en `apply` accedes a miembros sin prefijo; en `also` el objeto queda visible como `it`, útil cuando quieres distinguirlo de otros identificadores en scope. El encadenamiento `apply { }.also { }` es idiomático para "construir y registrar".

Consejo:
"apply = configure, also = side effect". Si dentro del bloque vas a referirte mucho al objeto, también ayuda a evitar conflictos de nombre.
