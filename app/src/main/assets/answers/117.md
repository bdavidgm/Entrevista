## Concepto
Un memory leak (fuga de memoria) en Android ocurre cuando objetos que ya no deberían usarse siguen referenciados y no pueden ser recolectados por el GC, provocando crecimiento de memoria, lentitud y eventual OutOfMemoryError.

## Explicación detallada
El garbage collector libera objetos sin referencias fuertes desde GC roots. Una fuga mantiene viva una cadena de referencias hacia [Activity](pregunta:31), [Fragment](pregunta:32), View o [Context](pregunta:127) destruidos.

Causas típicas en Android:
- Listener o callback registrado en singleton o objeto de larga vida que captura [Activity](pregunta:31) (this implícito en clase interna no estática).
- [Handler](pregunta:159) con mensajes pendientes que retienen [Activity](pregunta:31) (usar [Handler](pregunta:159)([Looper](pregunta:159).getMainLooper()) y removeCallbacks en onDestroy, o WeakReference).
- [Coroutine](pregunta:78) o Job sin cancelar ligada al lifecycle (usar [lifecycleScope](pregunta:130), [viewModelScope](pregunta:131)).
- [LiveData](pregunta:69).observe sin [LifecycleOwner](pregunta:44) o con referencia a View destruida.
- Static fields que guardan [Context](pregunta:127) (usar [applicationContext](pregunta:127) solo si imprescindible, mejor no guardar [Context](pregunta:127)).
- Bitmap grande retenido tras rotación si no se libera.
- [Hilt](pregunta:74)/[Dagger](pregunta:269) mal usado con scope incorrecto reteniendo UI.

Detección:
- [Android Studio](pregunta:194) Profiler (Memory): heap dump, ver instancias huérfanas de [Activity](pregunta:31).
- [LeakCanary](pregunta:264) en debug: notifica con cadena de referencias.
- [StrictMode](pregunta:97) y logs de GC en dispositivos de prueba.

Prevención:
- Observar con [lifecycleScope](pregunta:130) y repeatOnLifecycle.
- Usar [viewModelScope](pregunta:131) para trabajo de pantalla.
- Evitar pasar [Activity](pregunta:31) a capas de datos; inyectar [Application](pregunta:128) o interfaces.
- Limpiar binding en onDestroyView en Fragments.

## Cuándo usarlo
El concepto aplica siempre al diseñar listeners, async y DI. En entrevistas suelen pedir ejemplos concretos y cómo los evitarías.

Ejemplo:
// Mal: fuga
class BadManager {
    companion object {
        var activityRef: Activity? = null
    }
}

// Mejor
class GoodRepository @Inject constructor(
    @ApplicationContext private val context: Context
) { /* context de aplicación, no Activity */ }

class MyActivity : AppCompatActivity() {
  private val job = SupervisorJob()
  private val scope = CoroutineScope(Dispatchers.Main + job)

  override fun onDestroy() {
    job.cancel()
    super.onDestroy()
  }
}


ExplicaciónKotlin:
`class BadManager { companion object { var activityRef: Activity? = null } }` — `companion object` es singleton en la clase; `var` nullable retiene [Activity](pregunta:31) estática, fuga clásica. `class GoodRepository @Inject constructor(@ApplicationContext private val context: Context)` — qualifier [Hilt](pregunta:74) `@ApplicationContext` garantiza [Context](pregunta:127) de aplicación, no [Activity](pregunta:31). `class MyActivity : AppCompatActivity()` define `private val job = SupervisorJob()`: Job que no cancela hijos por fallo de uno. `private val scope = CoroutineScope(Dispatchers.Main + job)` combina [dispatchers](pregunta:30) con `+` sobre `CoroutineContext`. `override fun onDestroy() { job.cancel(); super.onDestroy() }` cancela [corrutinas](pregunta:78) antes de destruir [Activity](pregunta:31). `SupervisorJob` y cancelación explícita evitan retener [Activity](pregunta:31) tras destroy.

Consejo para entrevista:
Cita [Activity](pregunta:31) retenida por static/singleton/[Handler](pregunta:159) y [corrutinas](pregunta:78) sin cancelar. Menciona [LeakCanary](pregunta:264) y lifecycle-aware components. Demuestra que entiendes la diferencia entre [Context](pregunta:127) de [Activity](pregunta:31) y [Application](pregunta:128).
