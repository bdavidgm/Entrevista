## Concepto
Los sensores hardware se acceden con SensorManager.getDefaultSensor(TYPE_ACCELEROMETER) etc. y se registran con registerListener, recibiendo onSensorChanged en el hilo del listener ([Looper](pregunta:159) opcional). Recuerda unregisterListener al pausar para ahorrar batería.

Los valores vienen en coordenadas del dispositivo; el giroscopio mide velocidad angular.

Ejemplo:
val sm = getSystemService(SensorManager::class.java)
val accel = sm.getDefaultSensor(Sensor.TYPE_ACCELEROMETER)
sm.registerListener({ e, _ ->
    val (x, y, z) = e.values
}, accel, SensorManager.SENSOR_DELAY_GAME)

ExplicaciónKotlin:
`getSystemService(SensorManager::class.java)` usa la referencia de clase de Kotlin (`::class.java`) para obtener el servicio tipado sin cast manual, aprovechando la interop con la API Java del SDK. Las propiedades `val` declaran referencias inmutables a `sm` y `accel`. El tercer argumento de `registerListener` es una lambda `{ e, _ -> ... }`: el primer parámetro `e` es el evento del sensor y `_` descarta el segundo parámetro (`Sensor`) con la convención de guion bajo. Dentro del cuerpo, `val (x, y, z) = e.values` aplica [destructuring](pregunta:158) sobre el array `FloatArray` devuelto por `values`, extrayendo las tres componentes en variables locales de solo lectura. La lambda se pasa como objeto funcional compatible con `SensorEventListener`; no hace falta declarar una clase anónima explícita. Las constantes `Sensor.TYPE_ACCELEROMETER` y `SensorManager.SENSOR_DELAY_GAME` son miembros estáticos Java accesibles directamente desde Kotlin.

Consejo:
Usa SENSOR_DELAY_UI o GAME según necesidad; muestreos altos gastan batería. Para fusionar sensores y orientación, SensorManager.getRotationMatrix/getOrientation.
