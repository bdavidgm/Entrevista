Concepto:
El patrón Factory proporciona una interfaz para crear objetos sin especificar la clase concreta, centralizando la lógica de construcción y permitiendo variar el tipo creado según parámetros o configuración.

Explicación detallada:
Variantes:
- Simple factory method: fun create(type: Type): Product
- Abstract Factory: familias de objetos relacionados.
- Factory registrada en DI: @Provides fun provideParser(format: Format): Parser

En Android:
- ViewModelFactory (antes manual; Hilt lo genera).
- FragmentFactory para inyección en fragments.
- Room: @Database abstract class proporciona DAOs (factory interna generada).

Ventajas:
- Oculta detalles de construcción (dependencias, API keys).
- Facilita tests sustituyendo factory por fake.
- Cumple Open/Closed: nuevos tipos sin cambiar clientes si extiendes factory.

Kotlin:
- Funciones de extensión o top-level como factories ligeras.
- sealed class + when para tipos cerrados.

Cuándo usarlo:
Cuando la creación requiere lógica (permisos, config, múltiples impl) o no quieres que el cliente conozca implementaciones concretas.

Ejemplo:
interface PaymentProcessor
class StripeProcessor : PaymentProcessor
class PayPalProcessor : PaymentProcessor

object PaymentProcessorFactory {
    fun create(provider: Provider): PaymentProcessor = when (provider) {
        Provider.STRIPE -> StripeProcessor()
        Provider.PAYPAL -> PayPalProcessor()
    }
}

// Con Hilt @Binds múltiples @IntoMap o qualifiers es el enfoque escalable

Consejo para entrevista:
Relaciona con Hilt @Provides/@Binds como factory centralizada. Contrasta con new directo en Activity. Menciona ViewModelFactory histórico.
