## Concepto
ADB (Android Debug Bridge) es el puente entre PC y dispositivo/emulador: instala APK, lee logs, abre shell, reenvía puertos, captura pantallas y reinicia el daemon de USB.

Es la herramienta base de desarrollo y soporte.

Ejemplo:
adb devices
adb install -r app-debug.apk
adb logcat | grep MyTag
adb shell screencap -p /sdcard/screen.png && adb pull /sdcard/screen.png

ExplicaciónKotlin:
El ejemplo son comandos de terminal, no código Kotlin, pero en desarrollo Android los invocas desde scripts Gradle, tareas de CI o tests instrumentados escritos en Kotlin. Por ejemplo, un `TestWatcher` o regla JUnit puede ejecutar `Runtime.getRuntime().exec("adb logcat ...")`, aunque lo habitual es usar APIs de AndroidX Test o el plugin `com.android.tools.build:gradle` con `connectedDebugAndroidTest`. En Kotlin, `ProcessBuilder("adb", "devices").start()` construye listas de argumentos type-safe frente a concatenar strings. Para logcat filtrado, `adb logcat --pid=${Process.myPid()}` se combina con interpolación de cadenas Kotlin en scripts de depuración. Los tests instrumentados en Kotlin usan `@RunWith(AndroidJUnit4::class)` y acceden al dispositivo vía `InstrumentationRegistry.getInstrumentation().targetContext`. Entender ADB complementa el código Kotlin de la app: instalas el APK generado por `./gradlew installDebug` y lees logs mientras depuras corrutinas o Compose.

Consejo:
adb shell pm list packages y am start ayudan a reproducir intents. En CI, adb para tests instrumentados en emulador headless.
