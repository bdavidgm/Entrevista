## Concepto
lifecycleScope es un [CoroutineScope](pregunta:81) vinculado al ciclo de vida de un [LifecycleOwner](pregunta:44) ([Activity](pregunta:31), [Fragment](pregunta:32), [ProcessLifecycleOwner](pregunta:42)). Las [coroutines](pregunta:78) lanzadas en él se cancelan automáticamente cuando el ciclo llega a DESTROYED, evitando trabajo huérfano y [memory leak](pregunta:117)s.

En Fragments suele usarse viewLifecycleOwner.lifecycleScope para que se cancele al destruir la vista (onDestroyView), no al destruir el [Fragment](pregunta:32).

Ejemplo:
class MainActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        lifecycleScope.launch {
            repeatOnLifecycle(Lifecycle.State.STARTED) {
                viewModel.events.collect(::handleEvent)
            }
        }
    }
}


ExplicaciónKotlin:
`MainActivity` hereda de `AppCompatActivity` y sobrescribe `onCreate`, recibiendo un `Bundle?` nullable que Kotlin trata con seguridad de tipos. La llamada a `super.onCreate` delega en la clase base antes de registrar trabajo asíncrono. `lifecycleScope` es una propiedad de extensión de `LifecycleOwner` que expone un `CoroutineScope` ligado al ciclo de vida; `launch` inicia una [corrutina](pregunta:78) hija sin bloquear el hilo principal. Dentro, `repeatOnLifecycle(Lifecycle.State.STARTED)` es una [función suspend](pregunta:79)ida que reinicia el bloque lambda cada vez que el estado alcanza STARTED y lo cancela al bajar, evitando collectors activos en background. `viewModel.events.collect(::handleEvent)` usa una referencia a función (`::handleEvent`) como lambda de un solo parámetro, equivalente a `{ event -> handleEvent(event) }`. El operador `::` enlaza el método de instancia y encaja con la firma `(LoginEvent) -> Unit` que espera `collect`.

Consejo:
"En Fragments, viewLifecycleOwner.lifecycleScope". Mezclar con repeatOnLifecycle es la receta estándar para colectar Flows en UI.
