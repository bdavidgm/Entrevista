## Concepto
Inyectar dependencias por constructor (constructor injection) hace explícitas las colaboraciones de una clase y las convierte en parte de su API. Frente a singletons globales o [service](pregunta:36) locators, mejora la testabilidad (puedes pasar fakes en tests) y evita acoplamientos ocultos.

Frameworks como [Hilt](pregunta:74) o [Koin](pregunta:214) generan o resuelven esos constructores, pero el patrón funciona incluso sin DI: si tu clase compila sin tocar nada estático, ya es inyectable.

Ejemplo:
class GetUser @Inject constructor(
    private val repo: UserRepository,
    private val analytics: Analytics
) {
    suspend operator fun invoke(id: String): User {
        analytics.track("get_user")
        return repo.get(id)
    }
}

ExplicaciónKotlin:
class GetUser @Inject constructor(private val repo: UserRepository, private val analytics: Analytics) usa @Inject de [Hilt](pregunta:74)/[Dagger](pregunta:269) sobre el constructor primario para inyección automática. [suspend](pregunta:79) operator fun invoke(id: String): User define operación única invocable como getUser(id). analytics.track("get_user") ejecuta efecto secundario antes del return. return repo.get(id) devuelve User desde el [repositorio](pregunta:63); en Kotlin return es explícito aunque podría ser expresión = repo.get(id). Las clases de un solo propósito con invoke son idiomáticas en Kotlin para [use case](pregunta:66)s. private val en parámetros de constructor crea propiedades de instancia inmutables accesibles en invoke.

Consejo:
"Constructor sobre setter, setter sobre [Service](pregunta:36) Locator". Si tienes muchos parámetros, repiensa la responsabilidad de la clase ([SRP](pregunta:116)).
