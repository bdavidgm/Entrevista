Concepto:
Los callbacks del ciclo de vida de una Activity son métodos que el framework invoca cuando la activity transita entre estados: creación, visibilidad parcial o total, primer plano interactivo, y destrucción. Permiten inicializar y liberar recursos en el momento correcto.

Explicación detallada:
Orden típico al abrir: onCreate (una vez por creación, setContentView/setContent), onStart (visible pero quizá no enfocada), onResume (interactiva, animaciones, sensores). Al perder foco: onPause (pausar animaciones, guardar datos ligeros), onStop (ya no visible), onDestroy (liberar). onRestart si vuelve desde stopped sin recrear.

onCreate savedInstanceState restaura estado tras muerte de proceso. onSaveInstanceState guarda Bundle temporal (no para datos grandes ni persistentes). Configuration changes (rotación) pueden recrear activity si no usas configChanges o ViewModel.

En entrevistas dibujan diagrama y preguntan dónde poner observadores (lifecycleScope, repeatOnLifecycle). Comparan con Fragment que añade onCreateView/onDestroyView.

Cuándo usarlo:
onCreate: setup UI y ViewModel. onResume: reanudar cámara o location si aplica. onPause/onStop: pausar. onDestroy: cleanup no cubierto por lifecycle-aware. Evita lógica de negocio pesada en cada callback; usa ViewModel.

Ejemplo:
class PlayerActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContent { PlayerScreen() }
    }
    override fun onStart() {
        super.onStart()
        // registrar observers ligados a visible
    }
    override fun onResume() {
        super.onResume()
        player.play()
    }
    override fun onPause() {
        player.pause()
        super.onPause()
    }
}

Consejo para entrevista:
Secuencia completa A→B y B→A al volver atrás. Dónde savedInstanceState. Process death vs rotación. repeatOnLifecycle(STARTED).
