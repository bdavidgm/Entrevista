## Concepto
La accesibilidad (a11y) en Android garantiza que personas con discapacidad visual, motriz, auditiva o cognitiva puedan usar la app mediante TalkBack, ampliación, contrastes, navegación por teclado/switch y ajustes del sistema.

## Explicación detallada
contentDescription en Icon/Image para TalkBack; null si decorativo. Semantics en Compose: mergeDescendants, heading, stateDescription. Touch target mínimo 48dp. Contraste de color suficiente en light/dark.

Escalable texto: sp no px; respeta fontScale del usuario. No deshabilitar zoom salvo UX justificada. Focus order lógico en formularios.

Pruebas: TalkBack manual, Accessibility Scanner, checks Lint. Estados anunciados (loading, error). Live regions para cambios dinámicos.

En el contexto de proyectos Android reales, combina esta idea con las guías oficiales de Android Developers y con métricas de calidad (crashes, ANR, Vitals) cuando aplique. En code review, pregunta siempre por ciclo de vida, testabilidad y si la solución escala cuando el equipo crece.

Durante la entrevista, estructura la respuesta en definición, ejemplo concreto de tu experiencia o proyecto demo, y trade-offs frente a alternativas. Mencionar versiones recientes de API o biblioteca (Material3, Paging 3, Hilt 2.x) demuestra que sigues el ecosistema, sin necesidad de memorizar números de versión exactos.

Para profundizar tras la entrevista, revisa documentación oficial, codelabs y el código fuente de tu propio proyecto Entrevista aplicando el concepto en un flujo mínimo reproducible. La práctica deliberada con tests unitarios refuerza la confianza al explicar el tema en viva voz.

## Cuándo usarlo
Siempre; requisitos legales en muchos mercados y Play policies. Prioriza flujos core login/checkout.

Ejemplo:
Icon(
    imageVector = Icons.Default.Delete,
    contentDescription = stringResource(R.string.delete_item)
)

Modifier.semantics {
    contentDescription = "Carrito, ${cart.count} artículos"
    stateDescription = if (cart.isEmpty) "vacío" else "con artículos"
}


ExplicaciónKotlin:
`Icon(imageVector = Icons.Default.Delete, contentDescription = stringResource(R.string.delete_item))` usa composables Material; `Icons.Default.Delete` es `ImageVector` del catálogo Material. `stringResource(R.string.delete_item)` carga string de recursos en Compose (suspende recomposición si cambia locale). El segundo fragmento usa `Modifier.semantics { }`: lambda con receiver `SemanticsPropertyReceiver` donde se asignan `contentDescription` y `stateDescription` con template strings `"Carrito, ${cart.count} artículos"`. Interpolación `$cart.count` llama getter de propiedad. `if (cart.isEmpty) "vacío" else "con artículos"` es expresión condicional para `stateDescription`. `Modifier` es interfaz encadenable; `.semantics` es función de extensión de Modifier.

Consejo para entrevista:
contentDescription, 48dp, sp, TalkBack, Semantics Compose. Menciona inclusión como calidad, no extra opcional.
